Langfuse 是什么

LLM 应用最让人头疼的地方是非确定性：同样的输入，今天和明天可能给出不一样的答案； 一次多步的 agent 调用里，到底是哪一步出了错、用了多少 token、花了 多少钱、慢在哪一环，光看几行日志几乎无从判断。 Langfuse 是一个开源的 LLM 工程平台 （langfuse/langfuse，MIT 许可）。一句话：它把你 LLM 应用里那些看不见的每一步， 变成可观测、可评估、可复现的数据——围绕开发 → 监控 → 评估 → 调试这一个闭环转。 说白了，它要回答的是工程师每天都在问、却很难答的三个问题：这次到底发生了什么？好不好？怎么变得更好？

🔌 生活类比
把一个 LLM 应用想成一架飞机：平时能飞，可一旦出状况，你两眼一抹黑。Langfuse 给它装上两样东西—— 黑匣子（飞行记录仪）和实验记录本。黑匣子把每一次模型调用、每一步检索、用到的 prompt、 消耗的 token、花掉的钱、各环节延迟，统统录下来；实验记录本则让你把“换个 prompt / 换个模型 / 改个参数”的每一次尝试 并排摆开对比。出了事能回放，要改进有依据——这就是 Langfuse 想给 LLM 应用的东西。说到底，黑匣子负责「看见」，实验记录本负责「比较」，两者合起来才让「凭感觉调」变成「拿数据改」。
一个闭环：你的应用经 SDK 把运行事件送进 Langfuse；平台沉淀成 可观测 / 评估 / Prompt / 数据集 四类能力；工程师据此发现问题、改进应用，再把改进送回线上。Langfuse 始终在应用旁边观测，不侵入业务逻辑。

它到底是什么

一句话定义：一个把 LLM 应用的运行过程「数据化」的平台。它不替你训练模型，也不是又一个调用大模型的框架； 它做的事发生在你的应用之外——把每一次运行原原本本地记录、度量，并支撑你对它评估与改进。 官方把能力归为四块（在仓库里就是 web/src/features/ 下一个个目录）：

能力解决什么问题仓库里的实现
🔭 可观测 Tracing看清每一次调用：调用树、输入输出、token、成本、延迟web/src/features/traces · tracing-tables
✅ 评估 Evaluation给输出打分：LLM-as-judge、人工标注、自定义代码评估web/src/features/evals · scores · annotation-queues
📝 Prompt 管理把 prompt 版本化，线上动态拉取、回滚、对比web/src/features/prompts
🧪 数据集与实验攒测试集、跑实验、并排对比不同版本web/src/features/datasets · experiments

要特别说清楚它不是什么：它不是一个像 LangChain 那样帮你「写」LLM 应用的框架——你用什么框架写应用都行， Langfuse 只在运行时把发生的事记下来；它也不是一个模型服务或网关，不替你「调」模型（虽然 Playground 能帮你试）。 它的位置是旁路观测者：你的应用照常跑，只是顺手把每一步「报告」给 Langfuse。

🗺️ 宏观理解
Langfuse 既是 SaaS 云服务（cloud.langfuse.com），也能完全自托管（self-host）——因为它开源、且只依赖 Postgres、ClickHouse、Redis、S3 这几样常见基础设施。本教程讲的就是自托管那一套代码：你在 langfuse/langfuse 仓库里看到的，正是云上跑的同一套。理解了这套代码，你既能把它用好，也能改它、给它提 PR。

三大数据支柱：trace / observation / score

Langfuse 里所有「可观测」数据，归根结底是三种东西。它们的领域模型定义在 packages/shared/src/domain/ 下的 traces.ts / observations.ts / scores.ts，分析存储则落在 ClickHouse 同名的三张表里 （packages/shared/clickhouse/migrations/unclustered/0001_traces、0002_observations、 0003_scores）：

包含关系：一个 trace 把一次完整交互的所有步骤装在一起；每一步是一个 observation（可嵌套）；score 则是对 trace 或某个 observation 的评分。注意架构取向——observation 才是主分析单元，trace 更像把它们串起来的「关联句柄」。

🧵 trace · 关联句柄

代表一次完整的请求/会话。它本身信息不多（名字、用户、会话、标签、时间），主要作用是把同一次交互里的所有 observation 关联起来，方便你顺藤摸瓜。定义见 domain/traces.ts。

⭐ observation · 主角

一步具体操作。最核心是三型：GENERATION（一次 LLM 调用，带 model/token/cost）、 SPAN（一段有耗时的逻辑，如检索）、EVENT（一个瞬时事件）；新版领域模型在此之上还细分出 AGENT / TOOL / CHAIN / RETRIEVER 等更具体的类型。它带着 input/output、时间、用量、成本等丰富属性，是查询与聚合的主体。

那第三支柱 score 呢？它是评分，可挂在 trace 或某个 observation 上。取值主要有三类—— NUMERIC 数值（如 0.9）、CATEGORICAL 分类（如「正确 / 错误」）、BOOLEAN 布尔（领域模型里还有 CORRECTION、TEXT）； 来源有三种——EVAL（自动评审）、ANNOTATION（人工标注）、API（SDK 直接提交），见 packages/shared/src/domain/scores.ts。 正是 score 把「可观测」升级成「可评估」：有了可比较的分数，你才能量化「这次改动到底有没有让应用变好」，而不是凭感觉。score 的配置（数值范围、分类项） 存在 Postgres 的 ScoreConfig 里，分数本身则和 trace/observation 一样进 ClickHouse。

🔬 源码细节
observation 的类型在领域层是一个枚举：最初的 LegacyPrismaObservationType 只有 SPAN / EVENT / GENERATION 三种（见 packages/shared/prisma/schema.prisma）， 而当前的领域模型 packages/shared/src/domain/observations.ts 的 ObservationType 已扩展到 10 种 （新增 AGENT / TOOL / CHAIN / RETRIEVER / EMBEDDING / EVALUATOR / GUARDRAIL）。ClickHouse 的 observations 表把 model、 provided_usage_details、cost_details、prompt_id 等都内联成一行的列与 Map—— 这正是「宽事件」的样子（下一节展开）。

看得见，才改得动：一个排错场景

抽象的定义不如一个具体场景。假设你做了个客服问答机器人，用户反馈「它偶尔会答非所问」。 没有 Langfuse 时，你只有应用日志里几行 prompt=... response=...，既不知道是哪一步错了， 也复现不出来。接上 Langfuse 后，排查会变成这样：

① 在 trace 列表里按「用户点了踩」或「低分」过滤，几秒就翻出那几条出问题的 trace； ② 点开一条，observation 树一目了然——原来是 retrieve-docs 这一步召回了不相关的文档， 而不是模型本身的问题；③ 看这条 generation 的 input，确认被污染的上下文正是那几段错误文档； ④ 你把这条 trace 加进一个数据集，改完检索逻辑后跑实验对比新旧版本，用评分确认真的修好了。 整个过程，靠的就是「每一步都被原样记下来」。

⚠️ 没有可观测时的常见坑
只盯着「模型输出不好」很容易错怪模型。真实世界里大量「LLM 答得差」其实是检索召回错了、 prompt 拼错了、上游某步超时返回了空。这些只有把整条链路摊开才看得见—— 这就是为什么 Langfuse 把重点放在 observation 这一「步」，而不只是最终答案。

它由什么组成

把镜头拉到系统层面，Langfuse 是一个 pnpm + Turbo 的 monorepo，运行时主要分两个容器，外加四类存储 （见根目录 docker-compose.yml 与 AGENTS.md 的项目结构说明）：

数据怎么流：SDK 把事件发给 web 的公共 API，web 快速入队 Redis；worker 消费队列、做合并与成本计算，把结果写进 ClickHouse（宽事件）与 S3（原始事件/媒体）；UI 再由 web 从 Postgres（配置）和 ClickHouse（分析）读出来渲染。摄取走异步，所以 web 端永远轻快。
packages/shared共享内核
领域模型、Prisma/ClickHouse 访问、队列契约、仓储层。被 web 与 worker 共用；它不反向依赖任何一方。
webNext.js 应用
UI（Pages Router）+ 给 UI 用的 tRPC + 给 SDK 用的公共 REST API。依赖 @langfuse/shared 与 @langfuse/ee。
worker后台处理
BullMQ 队列消费者：摄取、评估、实验、导出、删除、集成……几十个队列。只依赖 @langfuse/shared。
ee企业版
企业功能与 license 校验，被 web 引用；其余皆 MIT。依赖方向严格单向：web/worker/ee → shared。

为什么把核心抽进 packages/shared、让 web 与 worker 都只单向依赖它？因为 web 与 worker 是两种完全不同的负载： web 面向用户、追求低延迟、要能随时重启；worker 啃重活、要可重试、按队列伸缩。把共享的领域模型、存储访问、队列契约 收进一个「谁都不反向依赖」的窄腰，两边就能各自独立演进、独立部署、独立扩容，同时又共用同一套契约、不会各写一套导致漂移。 这条单向依赖纪律（web/worker/ee → shared，而 shared 谁都不依赖），是整个 monorepo 能长期维护的关键。

为什么这么设计：宽事件 / 可观测性 2.0

Langfuse 的存储与查询路径，都被一条设计原则贯穿——把 observation 当作主分析单元，用「宽事件」记录一切 （明确写在 .agents/ARCHITECTURE_PRINCIPLES.md 里）。所谓宽事件，就是不把信息拆成零散的 指标（metrics）、日志（logs）、链路（traces）再事后拼接，而是把一步操作的所有高基数上下文 （模型、用量、成本、prompt、用户、标签、环境……）原样塞进同一行。

对比一下传统的「可观测性三件套」：指标（metrics）是预先聚合好的数字（如 QPS、平均延迟），省空间但丢了细节， 事后想按「某个用户 + 某个模型 + 某段时间」下钻就抓瞎；日志（logs）是零散的文本行，细节有了却难以聚合与关联； 链路（traces）能看调用结构，却往往不带业务属性。宽事件的思路是把这三者合一：每条 observation 既是一条 带结构的「链路节点」，又自带可聚合的数值（token/cost/latency），还携带任意业务字段（user/tag/metadata）。于是「下钻」和「聚合」 不再是两套系统，而是对同一张宽表的不同查询。这也是 Langfuse 选择 ClickHouse 的根本原因——列存天生擅长 「在一张宽表上做时间窗口扫描 + 任意维度聚合」。

🎯 设计取舍
为什么宁可让每行很「宽」，也不拆成规整的多张表？ 因为可观测的本质是事后回答你当时没想到的问题 （unknown unknowns）。如果上线前就把字段拆进固定指标，那你只能查预设好的维度；而把高基数上下文整行留下， 你就能在事后任意切片、分组、过滤。代价是单行更大、存储更多、写入要更聪明——Langfuse 用列存 + ZSTD 压缩 + 仅在详情页才读大字段来平衡（后续课展开）。用一点存储，换无限的提问自由。

🧪 自测 · 想一想为什么这么设计

1. Langfuse 把 observation（而不是 trace）当作“主分析单元”。这个定位最主要带来什么不同？
  1. 它取消了 trace 概念，只保留扁平日志
  2. 它把所有数据压缩成一个 trace 大对象，查询时整体读出
  3. 它要求每个 trace 只能有一个 observation
  4. 每个 LLM 调用/检索步骤本身就是一条带丰富属性的宽事件，可独立切片聚合；trace 只是把它们关联起来的句柄
看答案与解析 点击展开
答案：D。这是“可观测性 2.0 / 宽事件”的核心：把高基数上下文留在每条 observation 上，用户就能在不预设问题的前提下任意切片、分组、过滤。trace 仍在，但它是关联句柄，不是唯一入口（见 .agents/ARCHITECTURE_PRINCIPLES.md）。
2. Langfuse 为什么要同时用 Postgres 和 ClickHouse 两套数据库，而不是只用一套？
  1. 因为 Postgres 不能存字符串
  2. 纯粹为了用更多技术显得复杂
  3. ClickHouse 用来做用户登录，Postgres 用来跑 ML
  4. 事务性配置/元数据（项目、用户、prompt、配置）适合 Postgres；海量、可追加的宽事件（trace/observation/score）适合列存的 ClickHouse 做时间窗口扫描与聚合
看答案与解析 点击展开
答案：D。两类负载的访问模式根本不同：元数据是小数据量、强一致、频繁更新——交给 Postgres；遥测是大数据量、写多于改、按时间和项目聚合——交给 ClickHouse（宽事件、列存、ReplacingMergeTree）。用对存储，热路径才不会被拖垮。
3. Langfuse 的摄取为什么走“SDK → 公共 API → 队列 → worker → ClickHouse”这条异步链路，而不是 API 直接写 ClickHouse？
  1. 队列是用来加密数据的
  2. 为了让写入更慢以节省成本
  3. 因为 ClickHouse 只接受来自 worker 的连接
  4. 解耦与削峰：API 只需快速收下事件并入队，重活（合并、token/成本计算、批量写）交给 worker 异步处理，既抗流量尖峰又能批量高效写入
看答案与解析 点击展开
答案：D。把“接收”和“处理”分开，是高吞吐遥测系统的经典做法。API 端轻量、可水平扩展、对 SDK 低延迟；worker 端可重试、可批量、可分片（主/次队列隔离高吞吐项目）。代价是“最终一致”——数据有秒级处理延迟，这正是 §设计取舍要权衡的。

可观测性 2.0 与宽事件

上一课说 Langfuse 的「设计灵魂」是宽事件 / 可观测性 2.0。这一课把这句口号讲透——它不是营销词， 而是写在 .agents/ARCHITECTURE_PRINCIPLES.md 里、约束着 Langfuse 每一处存储与查询设计的第一性原则。 理解了它，后面「为什么用 ClickHouse」「为什么事件不可变」「为什么列表只读窄字段」这些问题，答案都会自己浮现。 可以说，这一课是整份指南的地基课：它不讲某个具体模块，而讲一种看问题的方式，后面五十多课都建在它之上。

🔌 生活类比
传统「可观测性 1.0」像医院把你的体检拆成三个互不相通的科室：一个只记数字（血压、心率——指标 metrics）， 一个只写文字（医生手记——日志 logs），一个只画就诊路线（你今天先挂号再抽血——链路 traces）。 想搞清「为什么这次不舒服」，你得自己跨三个科室对账。「可观测性 2.0」反过来：每看一次病，就写一份 完整病历——数字、描述、过程全在一张纸上。这张「完整病历」，就是宽事件（wide event）。 好处显而易见：换个医生、换个时间回看，都能从这一张纸上还原全部情况，而不必再去三个科室翻档案对时间线。
1.0 vs 2.0：传统三件套把信息拆散，回答跨维度问题前先得人肉拼接；宽事件把一步操作的数字、结构、业务字段放进同一行，「下钻」和「聚合」只是对同一张宽表的不同查询。这正是 ARCHITECTURE_PRINCIPLES.md 说的「偏好宽而属性丰富的事件，而非需要事后重建的零散指标/日志/链路」。

可观测性 1.0 的困境

「指标 + 日志 + 链路」这套经典组合（业内常称可观测性 1.0）每一件单看都没错，问题出在三者分离：

  • 指标（metrics）是预先聚合的数字，比如「平均延迟」「每分钟请求数」。省空间，但你上线前就得决定聚合哪些维度—— 事后想问「某个用户用某个模型在某段时间的延迟」，没预聚合就抓瞎。
  • 日志（logs）是一行行文本，细节都在，但难以聚合与关联：你很难对一堆自由文本做「按模型分组求 token 中位数」。
  • 链路（traces）能看清调用结构（谁调了谁、各花多久），却往往不带业务属性——它知道「调了 LLM」，但不知道「这次的 user 是谁、prompt 是哪版」。

于是真要排查一个跨维度的问题，你得在三套系统之间来回对账。Langfuse 的架构原则第一条就是冲着这个来的—— 把 observation 当作主分析单元，trace 只是关联句柄（原文："Model observations as the primary analytical unit. A trace is a correlation handle that links related observations, not the only useful entry point."）。

举个具体的对比。线上有人投诉「晚高峰回答变慢」，你想知道「是不是某个模型在某些用户上特别慢」。 在 1.0 里：指标只有「全局平均延迟」这一条曲线，看不出是哪个模型、哪批用户；日志里能翻到单条记录，但几万行文本没法快速按模型分组算分位数； 链路能看单次调用的耗时，却没记 user 和 model 这些业务字段。三套都「沾边」，但没有一套能直接回答，你只能导出数据自己写脚本拼。 在 2.0 里：每条 observation 都同时带着 model、userId、latency，于是这个问题退化成一句普通查询—— 「按 model + userId 分组、看 latency 的 p95」。同一个问题，1.0 要跨三系统拼半天，2.0 改个查询就答了。

宽事件：把一切放进一行

「可观测性 2.0」的解法很直接：不要事后拼接，写的时候就别拆开。把一步操作的所有高基数上下文 （哪个模型、多少 token、多少钱、多久、哪个用户、哪些标签、什么环境……）原样塞进同一条记录——这条记录就是 Langfuse 里的 observation。它既是「链路节点」（有 parentObservationId 能拼出调用树）， 又自带可聚合的数字（usageDetails / costDetails / latency），还携带任意业务字段 （metadata / tags / userId）。三件套想分开做的事，一条宽事件全包了。

你可能会问：那为什么不把这些字段拆成多张规整的表、用外键关联，岂不是更「干净」？答案是——查询时的 JOIN 是热路径上的隐藏成本。 可观测数据量极大（一个活跃项目一天可能上亿条 observation），如果每次「按模型看成本」都要把 observation 表和一张「模型表」「用量表」JOIN 起来， 扫描与连接的代价会随数据量爆炸。Langfuse 的原则是谨慎反范式化（denormalize carefully）：把常用来过滤、聚合的字段直接内联到 observation 行上，让「按模型 / 按用户 / 按时间」这些高频查询变成直接的列谓词，而不是 JOIN。这就是「宽」的另一层含义—— 不只是字段多，更是有意把该放在一起的放在一起，用一点存储冗余换查询时不必连表。

😕 1.0：先拆再拼

指标、日志、链路各存一处；问一个跨维度问题，要在三套系统里查三次再人肉对账。能问什么，上线前就被字段设计锁死了。

😀 2.0：一行装下

一条 observation 同时有数字、结构、业务字段。切片、分组、过滤都是对同一张宽表的查询，事后想问什么都行。代价：行更宽、存得更多。

为什么值得：unknown unknowns

宽事件最大的价值，是回答你当初没想到要问的问题。预先聚合的指标只能回答已知问题（你建了哪张仪表盘就只能看哪个维度）； 而把高基数上下文整行留下，新问题来了，换个查询就能答，不必改埋点、不必等下次上线。

这一点对一个平台尤其关键。Langfuse 要给成千上万个项目用，它不可能预先知道每个团队会关心什么维度—— 有人想看「按 prompt 版本对比成本」，有人想看「按 session 看用户多轮对话的总延迟」，有人想按自定义 metadata 标签切分。 如果走预聚合的老路，平台就得为每种需求预先建一张指标——这既做不完、也僵化。宽事件让 Langfuse 把「定义新视图」这件事， 从「改埋点 + 重新上线」降级成「写一条新查询」：仪表盘、自定义指标、过滤器（第 23、40 课）本质上都是在同一张宽表上换查询。 这就是为什么「保留高基数上下文」不是锦上添花，而是整个平台可扩展性的地基。

已知 vs 未知：预聚合把「能问什么」在上线时就固定了；宽事件把高基数上下文整行留存，于是面对「unknown unknowns」——那些你当初根本没想到要问的问题——只需换一个查询。这就是 Langfuse 宁可让行更宽、也要保留上下文的根本理由。

九到十条原则乍看零散，其实围着四个主题转——它们层层递进，正好对应这份指南后面四大块内容。把它们分组记，比逐条背更有用：

数据模型observation 为主 · 高基数

存储宽 · 不可变 · 反范式 · 列式
再把「怎么存」定对：宽事件、追加不可变（ReplacingMergeTree）、谨慎反范式化、围绕列式访问设计排序键与分区。
查询紧凑列表 · 详情才读大字段
然后把「怎么读」定对：列表/仪表盘用紧凑的查询优化表示，大块 input/output 只在单条详情页才拉取。
契约与成本规模感知 API · 运维从简
最后把「对外/对运维」定对：API 强制时间窗口与 token 分页，避免扫全表；把额外的库、队列、物化视图的运维负担当成真实成本来权衡。

这四组也正是本指南的主线：数据模型是第一部分（本部分），存储是第二部分，写入与查询两条链路是第三、四部分， 契约与成本则散落在公共 API（第 27 课）与运维（第 53 课）。换句话说，你现在读的这份「设计哲学」，就是后面五十多课的提纲。

还有一条容易被忽略但很关键的原则：保留近实时的调试体验（原文："Preserve real-time or near-real-time debugging workflows. Batch processing can help, but it should not make fresh production behavior invisible."）。这解释了为什么 Langfuse 的摄取虽是异步的，却要把处理延迟压到秒级——可观测性如果「看不到刚刚发生的事」，就失去了一半意义。这条会在第 14–17 课的摄取链路里反复出现。

三大支柱深入：trace/observation/score

第 1 课认识了 trace / observation / score 三个名字，这一课把它们拆开看里子：每个到底有哪些字段、为什么这样设计、 又怎么落到 ClickHouse 的三张表上。它们的「真身」是 packages/shared/src/domain/ 下的三个 Zod schema—— 读懂这三个 schema，等于拿到了理解 Langfuse 后面所有功能的钥匙，因为评估、数据集、仪表盘……最终都在操作这三种对象。 打个比方，如果说第 1 课给了你一张「Langfuse 全景照」，这一课就是把照片里的三个主角拉到跟前，挨个看清他们的五官—— 记住他们长什么样，后面无论他们出现在哪一课的剧情里，你都能一眼认出。

🔌 生活类比
把一次完整的用户交互想成医院里一个病历夹（trace）：夹子封面只写几行身份信息（谁、什么时候、贴了哪些标签）。 夹子里一页页具体记录才是重点（observation）：这页是「拍了 CT」、那页是「开了药」、再一页是「医生下了诊断」—— 每页都详细写明用了什么、花了多久、多少钱。最后，页边的红笔批注（score）给某一页或整个夹子打分：「这个诊断准确吗？9 分」。 封面薄、内页厚、批注点睛——这正是三大支柱的分工。把这套「夹子—内页—批注」的画面记住，下面看字段时就不会迷路：凡是「整次交互级」的信息往封面（trace）放，凡是「某一步级」的信息往内页（observation）放，凡是「评判级」的信息就是批注（score）。
树是「指」出来的：observation 表里并没有「children」字段；每个 observation 只记一个 parentObservationId 指向父节点，UI 再据此反向拼出整棵调用树（第 25 课）。根节点的 parent 为 null。这种「只存父指针」的设计，让写入时各 observation 互相独立、可乱序到达——非常契合异步摄取。

trace：薄薄的关联句柄

先看最简单的 trace。它的领域模型 TraceDomain 字段很少，几乎都是身份与标记，没有什么「重」数据：

packages/shared/src/domain/traces.tsTraceDomain
export const TraceDomain = z.object({
  id: z.string(),            // trace 的唯一 id
  name: z.string().nullable(),
  timestamp: z.date(),
  environment: z.string(),    // 多租户隔离维度（第 10 课）
  tags: z.array(z.string()),
  bookmarked: z.boolean(),
  public: z.boolean(),
  sessionId: z.string().nullable(), // 串成会话（第 26 课）
  userId: z.string().nullable(),
  input: jsonSchema.nullable(),   // 整次交互的输入/输出
  output: jsonSchema.nullable(),
  metadata: MetadataDomain,
  projectId: z.string(),
});

注意 trace 上几乎没有「成本」「token」「模型」这类字段——那些都在 observation 上。trace 的职责就是把一次交互的所有 observation 关联起来， 外加几个方便检索的标记：userId（哪个用户）、sessionId（哪一轮会话）、tags（自定义标签）、bookmarked（人工收藏）、 public（是否可公开分享）。这印证了第 1 课那句话：trace 是关联句柄，不是主分析单元。

别小看这几个「标记」字段，它们各自对应一个真实的产品功能：bookmarked 让你在排查时把可疑 trace 收藏起来稍后细看； public 控制这条 trace 能否生成一个公开分享链接（第 27 课），方便你把一个坏 case 甩给同事而不用对方登录； environment 把 prod / staging / dev 的数据隔离开（第 10 课），避免测试流量污染线上看板；release 和 version 记录「这条 trace 来自应用的哪个版本」，于是上线后回归时，你能直接按版本对比质量与成本。换句话说，trace 虽薄， 但每个字段都是为「事后好查、好分享、好归类」而精心挑选的。

observation 才是「厚」的那一个。先看类型——很多人以为只有 3 种，其实当前领域模型 ObservationType 有 10 种：

在看字段之前，先理解上面那张树图背后的一个关键设计：observation 之间的父子关系，只靠子节点上的一个 parentObservationId 指针表达， 父节点完全不知道自己有哪些孩子。这看似简单，却是为异步摄取量身定做的——因为每个 observation 都自带「我爸是谁」，它们就能 各自独立、乱序到达服务端：子节点比父节点先到也没关系，反正等查询时 UI 再按指针把树拼出来（第 25 课）。如果反过来让父节点维护一个 children 列表， 那就得「等所有孩子到齐才能写父节点」，在高并发异步管线里这是灾难。一个小小的指针方向选择，背后是整条摄取链路的可行性。

  toolDefinitions, toolCalls, toolCallNames,  // 工具调用信息

看到没——一条 observation 就把「这步用了哪个模型、吃了多少 token、花了多少钱、多久、关联哪版 prompt、调了哪些工具」全装下了。 这正是第 2 课说的「宽事件」：高基数上下文整行内联，于是「按模型看成本」「按 prompt 版本比延迟」都成了对这张宽表的普通查询。

这里有个容易混淆、但很能体现设计用心的细节：用量和成本各有两套字段——providedUsageDetails / providedCostDetails 是 SDK 上报时「自带」的（你的应用如果知道 token 数和价格，可以直接报上来），而 usageDetails / costDetails 是 Langfuse 自己「算出来」的（摄取时按模型定价表计算，第 16 课）。两套并存的好处是：你报的我尊重，你没报的我兜底—— 既允许应用精确控制，又保证即使应用什么都不报，Langfuse 也能根据 model 名匹配定价、把成本补齐。这种「provided 优先、computed 兜底」的模式， 在后面成本计算一课会反复出现。

还有两个不起眼但排错时极有用的字段：level（DEBUG / DEFAULT / WARNING / ERROR）和 statusMessage。前者给每个 observation 标上 严重程度，后者记一句状态说明（比如报错信息）。有了它们，你就能在成千上万条 observation 里一键过滤出 level = ERROR 的那些——直接定位「哪一步挂了、报了什么」，而不必从头翻整棵树。这又一次体现了「把高基数上下文留在每一步上」的价值： 连「这步是不是出错了」都是可查询、可聚合的一列。

score：评判

第三支柱 score 把「可观测」升级成「可评估」。它的两个核心维度是来源（source）和数据类型（dataType），都定义在 packages/shared/src/domain/scores.ts：

score 的两端：左边是「谁打的分」——EVAL（自动评审，第 29–31 课）、ANNOTATION（人工，第 32 课）、API（SDK 提交）；右边是「打给谁」——可挂在 trace（整次交互）或某个 observation（具体一步）上。中间的 score 本身带一个值和一个 dataType。

score 的数据类型有 5 种（ScoreDataTypeArray）：NUMERIC 数值、CATEGORICAL 分类、 BOOLEAN 布尔、CORRECTION 订正、TEXT 文本。来源（ScoreSourceArray）有 3 种： API / EVAL / ANNOTATION。一条规则值得记住：ANNOTATION 来源的 score 必须带一个匹配的 configId（唯一例外是 CORRECTION 类型），这样它才能在标注队列 UI 里正确渲染——见 scores.ts 的 isAnnotationScoreMissingConfigId。

dataType取值长什么样典型用途
NUMERIC一个数，如 0.9相关性、帮助度等连续评分
CATEGORICAL一个标签，如「正确」把输出分到若干类别
BOOLEANtrue / false是否通过、是否安全
CORRECTION一段订正后的文本人工把错误输出改对（不需要 configId）
TEXT一段自由文本（≤500 字）评语、理由

三种来源也不是随便分的，它们对应着不同的「打分主体」，权限也不同：EVAL 是 Langfuse 内部评估器的产物 （LLM-as-judge、代码评估，第 29–31 课），保留给系统内部使用；ANNOTATION 是人在 UI 里手动标注；API 则是你的应用 通过 SDK 直接提交（比如把用户点的「👍/👎」上报成分数）。源码里有个细节能印证这点：公共建分接口允许的来源只有 API 和 ANNOTATION 两种（PublicApiCreateScoreSourceDomain），EVAL 被刻意排除在外——因为它是平台自己产生的，不该由外部冒充。这种「按来源划权限」的小设计， 保证了「谁打的分」这件事可信。

为什么三支柱要落到三张分开的 ClickHouse 表，而不是塞进一张大表？因为它们的访问模式不同：observation 量最大、查询最频繁、字段最宽； score 数量少、常按 trace/observation 关联查；trace 居中、是列表页的主入口。分表能让每张表用最适合自己的排序键与分区（第 8 课）， 互不拖累。这又是一次「按访问模式选存储」的体现——和「为什么 Postgres + ClickHouse 并存」（第 7 课）是同一种思路，只是粒度更细。

还有一个常被忽略的「第四类」聚合：session（会话）。它不是一张宽事件表，而是靠 trace 上的 sessionId 把多次 trace 串成一段连续对话（比如用户和你的 bot 来回聊了十轮）。session 的元数据在 Postgres 的 TraceSession 模型里， 聚合视图则在查询时算出来（第 26 课）。所以严格说，Langfuse 的核心数据是「三支柱 + 会话聚合」。

举个会话的例子：用户在客服 bot 上连问了三轮——「我的订单到哪了？」→「那能改地址吗？」→「帮我改成公司地址」。这三轮各自是一个独立的 trace （每轮一次完整问答），但它们带着同一个 sessionId。于是在「会话」视图里，你能把这三个 trace 当成一段连续对话来看：整段聊了多久、 一共花了多少钱、用户在第几轮开始不耐烦。这种「单次看 trace、连续看 session」的双视角，正是因为 sessionId 这一个字段把关联关系提前埋在了写入里， 查询时不必再做复杂的会话切分。

把三支柱合起来看，你会发现一个漂亮的分工：trace 负责「关联」、observation 负责「记录」、score 负责「评判」、session 负责「串联」。 四者各司其职、字段不重复堆砌，又都能在一次查询里被关联起来。后面五十多课无论讲摄取、查询、评估还是仪表盘，操作的都是这四种对象—— 现在把它们的形状刻进脑子里，后面会轻松很多。

🎯 设计取舍
为什么把「重」字段都堆在 observation，而让 trace 这么薄？ 因为查询绝大多数发生在「步」这个粒度：你想按模型、按 prompt 版本、 按工具调用去聚合和过滤，这些维度天然属于某一步，而不是整次交互。把它们放在 observation 上，常用查询就成了对 observations 宽表的直接列谓词； trace 只需保留关联和少数检索标记。代价是同一信息（如 userId）可能要在 trace 和 observation 间冗余一份——这正是第 2 课「谨慎反范式化」的体现。 反过来理解这个取舍：Langfuse 主动付出一点重复，换来用户每天高频运行的查询无需连表、足够快。在可观测的数据规模下，这笔账几乎总是划算的。
🎯 本课要点
  • trace 很薄：几乎只有身份与标记（userId/sessionId/tags/bookmarked/public），职责是关联。
  • observation 很厚：ObservationType 有 10 种（核心三型 SPAN/EVENT/GENERATION），靠 parentObservationId 拼树，内联 model/usage/cost/prompt/tool 等富属性。
  • score 两维度：来源（API/EVAL/ANNOTATION）× 数据类型（NUMERIC/CATEGORICAL/BOOLEAN/CORRECTION/TEXT），可挂 trace 或 observation。
  • 三者 1:1 落到 三张 ClickHouse 宽事件表；session 靠 sessionId 把多个 trace 串成会话。
  • 读懂这三个 domain/*.ts schema，就拿到了理解后面所有功能的钥匙，因为评估、数据集、仪表盘最终都在操作它们。

项目全景地图（monorepo 与窄腰）

前三课讲的是「数据长什么样」，这一课换个维度：代码住在哪里。Langfuse 是一个 monorepo（单仓库多工程）， 所有东西都在 langfuse/langfuse 一个仓库里，但内部分成几个各司其职的「工作区」。把这张地图记在心里，后面任何一课说「打开 web/src/...」或「这段逻辑在 worker 里」，你都能立刻知道它在哪、为什么在那。

🔌 生活类比
把这个仓库想成一座大型购物中心：整座楼是一个 monorepo，里面有几家独立店铺—— web 是面向顾客的门店（人来人往、要快、随时营业），worker 是后场的加工车间（埋头干重活、可以排队慢慢做）。 而 packages/shared 是中央仓库：所有店铺的货（数据模型、数据库访问、队列契约）都从这里取，但中央仓库从不反过来依赖某家店铺。 大家在同一栋楼里，共用水电（构建工具），却又边界清晰、各自营业。逛这座楼之前先拿到这张「楼层导览图」，后面无论去哪家店，你都不会迷路。
顶层就这几块：web（门店）和 worker（车间）是两个运行时容器，都向上依赖 packages/shared（中央仓库 / 窄腰）；ee 是企业功能、被 web 引用；fern 定义公共 API 契约并生成 generated 客户端。记住这张图，后面所有「文件在哪」的问题都有了坐标系。

稍微展开一下两个运行时各自在忙什么，你会更有体感。web 是那个你浏览器里打开的 Next.js 应用，它干三件事：渲染界面、 给界面提供类型安全的内部 API（tRPC，第 21 课）、以及给外部 SDK 提供公共 REST API（web/src/pages/api/public，第 27 课）—— 摄取事件的入口就在这里。worker 则是个没有界面的后台进程，它订阅几十个 BullMQ 队列，把所有重活、慢活、可重试的活都揽下来： 把摄取的事件合并写进 ClickHouse、跑 LLM 评估、导出数据、删除项目……（第三部分起逐个讲）。一句话：web 负责「快进快出」，worker 负责「埋头苦干」， 两者通过中间的 Redis 队列解耦——这也正是下一课「双存储」和第三部分「摄取链路」要展开的主线。

一个仓库，四个工作区

Langfuse 用 pnpm workspaces 把一个仓库切成几个独立可构建的包。根目录的 pnpm-workspace.yaml 一行行列出了它们：

pnpm-workspace.yamlworkspaces
packages:
  - "web"          # Next.js 应用（UI + tRPC + 公共 REST API）
  - "worker"       # 队列消费者，后台处理
  - "packages/**"  # 共享包，主要是 packages/shared
  - "ee"           # 企业版功能

packages/** 下其实不止 shared 一个，还有 config-eslint、config-typescript、 eslint-plugin 这些「工具配置包」；但真正承载业务的共享内核是 packages/shared——它就是上图里的中央仓库。 为什么要拆成工作区而不是一个大文件夹？因为这样每个包能独立声明依赖、独立构建、独立测试，pnpm 还能在它们之间做符号链接， 让 web 直接 import ... from "@langfuse/shared" 用上最新代码，不必发版。 这点对开发体验影响很大：你在 packages/shared 里改一个类型，web 和 worker 的编辑器会立刻飘红提示哪里要跟着改—— 等于把「契约变更」的影响在编码阶段就暴露出来，而不是等到运行时才报错。

依赖方向：为什么是「窄腰」

四个工作区之间不是随便互相引用的，而是有一条严格单向的依赖规则（写在 .agents/AGENTS.md 里）：

.agents/AGENTS.mdDependency direction
- web    -> @langfuse/shared, @langfuse/ee
- worker -> @langfuse/shared
- ee     -> @langfuse/shared
- @langfuse/shared -> no imports from web, worker, or ee
「窄腰」（narrow waist）：把所有人都要用的东西收进一个谁都不反向依赖的薄层。web/worker/ee 像沙漏的上半，shared 是中间最细的「腰」，再往下是数据库。好处是：两个容器各自独立演进、独立部署，却共用同一套领域模型与队列契约，契约不会漂移。

这条纪律为什么重要？因为 web 和 worker 是两种完全不同的负载：web 要面向用户、低延迟、可随时重启；worker 要啃重活、可重试、按队列伸缩。 如果让它们互相直接引用，改一个就可能牵连另一个，部署也会绑死。把共享的东西收进 shared、并规定只能向下依赖， 就等于在两个容器之间立了一道清晰的契约墙：队列里传什么、数据库表长什么样，都由 shared 这一处定义（比如队列负载就归 packages/shared/src/server/queues.ts 管）。两边照着同一份契约各干各的，互不踩脚。

举个这条规则挡住的真实 bug：假设没有窄腰，worker 里图省事直接 import 了 web 的某个工具函数。某天 web 为了改界面重构了那个函数、 顺手改了它的参数——结果 worker 在毫不知情的情况下被改坏了，可能要到线上摄取出错才发现。而在窄腰规则下，worker 根本不允许依赖 web； 两边唯一的「共同语言」是 shared 里那份明确的契约。想共享逻辑？那就把它下沉到 shared，让它成为受契约约束的一等公民，而不是从隔壁悄悄借用。 「依赖只能向下」这一条，本质是在用编译器帮你挡住跨容器的隐式耦合。

多个工作区的构建、测试、lint 由 Turbo 统一编排（根目录 turbo.json）。它最有用的两件事是任务依赖图和缓存：

看 turbo.json 里 build 任务的定义："dependsOn": ["db:generate", "^build"]——意思是「构建任何一个包之前， 先生成数据库客户端、并先把它依赖的包（^ 表示上游依赖）构建好」。于是你只要 pnpm build，Turbo 就会自动算出正确顺序、 把能并行的并行、能缓存的跳过（"cache": true）。这就是为什么在一个有几十个包的大仓库里，改一行代码也不必全量重建。

把窄腰和 Turbo 合起来，看一个具体的连锁反应：假设你要给 trace 加一个字段。你只需在 packages/shared/prisma/schema.prisma 改一处、再在 domain/traces.ts 改一处——一次 db:generate 重新生成 Prisma 客户端，web 和 worker 立刻都看到新字段，因为它们都从同一个 @langfuse/shared 取类型。整件事在一个 PR 里完成， CI 一起跑通。要是分成三个仓库，这同一个改动得发三次版、改三处依赖、还要祈祷三边版本对得上——这就是「原子地一起改」的实际意义。

🧩 polyrepo（分仓）

web/worker/shared 各一个仓。改一个共享结构 → 跨仓发版、对版本、易漂移。好处是各仓更小、更独立。

📦 monorepo（Langfuse 的选择）

一个仓装下全部。共享改动一个 PR 原子完成、CI 一起验。代价：仓更大、要靠 Turbo 管构建缓存。

缓存这件事对持续集成（CI）尤其值钱。一个几十包的大仓，如果每次 PR 都把所有包从头 build + lint + test 一遍，CI 会慢到没法用。 Turbo 会给每个任务的输入算一个指纹（源码 + 依赖 + 配置），只要指纹没变，就直接复用上次的输出、整段跳过。于是改了 worker 的一个文件， web 那些没被波及的构建/测试就命中缓存、秒过。配上远端缓存，团队成员之间还能共享构建结果。这就是大仓也能保持 CI 快的秘诀—— 也是「用 monorepo」这个选择能成立的工程前提：没有好的缓存，monorepo 的构建成本会劝退所有人。

🎯 设计取舍
为什么用 monorepo + 窄腰，而不是把 web/worker/shared 拆成三个独立仓库（polyrepo）？ 因为 web 和 worker 高度共享领域模型与队列契约， 如果分仓，每次改一个数据结构就要跨仓发版、对版本，极易出现「web 以为字段叫 A、worker 还在用 B」的漂移。monorepo 让三者原子地一起改—— 一个 PR 同时改 shared 的契约和两边的用法，CI 一起验。代价是仓库更大、构建工具（Turbo）更复杂， 但对一个契约高度耦合的系统，这笔账划算。换个角度说：分仓省的是「单仓的复杂度」，monorepo 省的是「跨仓协调的复杂度」——而后者往往才是真正拖慢团队的那个。
🎯 本课要点
  • Langfuse 是 pnpm + Turbo 的 monorepo：web（门店）、worker（后场车间）两个运行时 + packages/shared（中央仓库）+ ee。
  • 窄腰：依赖严格单向 web/worker/ee → shared，而 shared 谁都不依赖；契约（领域模型、队列）集中在 shared。
  • Turbo 用任务依赖图（build 依赖 db:generate + ^build）+ 缓存，让大仓也能快速、增量构建。
  • 记住高频目录：web/src/{pages,server,features}、worker/src/{queues,services}、packages/shared/src/{domain,server}——这是后面查代码的速查表。
  • 选 monorepo 是为了让契约高度耦合的 web/worker/shared 能原子地一起改，避免跨仓版本漂移与协调成本。

第一部分快收尾了。你已经知道数据长什么样（trace/observation/score）、代码住在哪（monorepo 与窄腰）、 背后的设计哲学（宽事件）。这一课把它们串成一条线：跟着一条具体的 trace，从你应用里 langfuse.trace(...) 那一刻起， 一路走到它出现在 UI 列表里、被你点开、甚至被自动评分。这是一支「预告片」——每一段路都标好了由后面哪一课详细展开，看完你就有了整张地图的行车路线。 换句话说，前四课是把零件一个个拆给你看，这一课是把它们装回去、发动一次，让你看见数据真正流动起来的样子——有了这个动态全景，再去逐课深挖每个零件，就不会只见树木不见森林。

🔌 生活类比
把一条 trace 的一生想成寄一件快递：你在家下单封箱（SDK 里创建 trace/observation）；快递员上门收件，扫码入袋 （公共 API 收下事件、快速入队）；包裹进了分拣中心排队，由后台分拣处理（worker 从队列取出、合并、算运费=成本）； 然后上架入库（写进 ClickHouse）；最后你打开手机查物流（在 UI 里看到这条 trace）。关键是——下单那一刻并不等于已入库， 中间隔着收件、分拣、上架几步，所以会有短暂的处理延迟。这正是 Langfuse 摄取「异步」的本质。
一条 trace 的完整旅程：上半是写入链路（SDK→API→Redis→worker→ClickHouse，旁挂 S3），每段标了对应课号（第 12–19 课）；下半是读取链路（ClickHouse→repository→tRPC→UI，第 20–27 课）。两条链路在 ClickHouse 这张宽表上「碰头」：写入把数据放进去，读取把它取出来。

上行 · 写：从 SDK 到 ClickHouse

先看包裹「寄出到入库」的全过程。它被刻意设计成异步的——API 只负责飞快地收下并入队，真正的重活交给 worker 慢慢做：

1

SDK 创建事件（第 6 课）

你的应用调用 SDK 或发 OTel span，生成 trace-create / observation-create / score-create 等事件。

2

公共 API 收件入队（第 12·14 课）

web/src/pages/api/public/ingestion.ts 鉴权、用 Zod 校验一批事件，然后快速塞进 Redis 队列就返回——不等处理完。

3

worker 合并聚合（第 15 课）

IngestionService 从队列取出事件，从 S3 取这条记录的历史事件合并（同一个 observation 的多次 update 会合成一条），再决定最终长什么样。

4

算 token 与成本（第 16 课）

按模型名匹配定价表，把没上报的 usage/cost 兜底算出来（还记得第 3 课的 provided vs computed 吗）。

5

ClickhouseWriter 批量写（第 17 课）

把处理好的记录攒成批，一次性写进 ClickHouse 的 traces/observations/scores 表（ReplacingMergeTree 负责「同 id 后写覆盖先写」）。

这条链路里，S3 扮演了「事件原始底稿」的角色：每个进来的事件先落 S3，worker 合并时再回 S3 取历史。这样即使处理出错也能重放， 而且让「同一条 observation 的多次更新」能被正确合并——这些细节在第 15–17 课会逐一拆开。现在你只需记住这条主干：收件 → 入队 → 合并 → 算钱 → 入库。

这里要特别点出第 3 步「合并」为什么不可省。原因是：SDK 上报一个 observation 往往不是一次性的。比如一次 LLM 调用，SDK 可能先发一个 「observation-create」说「我开始了，模型是 gpt-4o」，等调用返回后再发一个「observation-update」补上「输出是 X、用了 1240 token」。这两条事件分别到达， worker 必须把它们合并成同一条最终记录。这就是为什么要回 S3 取「这个 id 的历史事件」——把先后到达的碎片拼成完整的一行。理解了这一点，你就明白第 3 课说的 「observation 靠 parentObservationId 乱序到达也没关系」是怎么在摄取里真正实现的：事件可以分多次、乱序到，worker 负责把它们收敛成一致的最终态。

下行 · 读：从 UI 回到 ClickHouse

数据落库后，你在浏览器里看到它，走的是另一条更实时的链路——你点一下，它就去查：

1

UI 发起查询（第 20 课）

你打开 trace 列表页，前端通过 tRPC 的 hook（如 api.traces.all.useQuery）发起一次类型安全的请求。

2

tRPC 鉴权分派（第 21 课）

请求进 web/src/server/api，中间件先校验「你属于这个项目吗」（RBAC），再分派到对应的路由处理函数。

3

仓储层拼查询（第 22·23 课）

路由调用 packages/shared/src/server/repositories 里的函数，把你的过滤条件翻译成 ClickHouse SQL（带时间窗口、只选需要的列）。

4

ClickHouse 返回，UI 渲染（第 24·25 课）

列表页只取紧凑的窄字段（快）；你点开某条 trace 时，才去拉完整的 input/output 大字段并拼出观测树。

注意写入和读取在 ClickHouse 这张表上会师：写入链路负责把宽事件放进去，读取链路负责用各种姿势把它查出来。第 2 课说的「列表只读窄字段、详情才读大字段」， 正是在这条读取链路上落地的。

为什么要把「收件」和「处理」分开

这条链路最妙的一步，是 API「收下就返回」。它把整件事掰成了「快路」和「慢路」两半：

一刀切开：应用和 API 之间是快路（毫秒级返回，几乎不可能慢）；API 之后是慢路（入队 → worker 后台批量处理）。队列像蓄水池，洪峰来了先排队，worker 扛不住就横向加机器。

想想看：一个高流量的 LLM 应用，每秒可能产生成千上万条 observation。如果 API 必须同步地合并、算成本、写 ClickHouse 才返回，那么两件坏事会发生： 其一，你的应用每次调用都要多等这几百毫秒，平白被监控工具拖慢；其二，一旦 ClickHouse 抖动或洪峰打满，API 就会超时、甚至把错误反弹给你的业务。 把「收件」和「处理」一刀切开，API 就退化成一个极轻的「只管校验 + 入队」端点——它几乎不可能慢，也几乎不可能因为下游问题而连累你的应用。

队列则像一个蓄水池：洪峰来了先在队列里排队，worker 按自己的节奏批量消费，扛不住就加 worker（横向扩容）。 Langfuse 甚至把摄取队列分成主/次两条（第 14 课），把高吞吐的大项目和普通项目隔开，避免一个项目的洪峰拖垮所有人。 代价就是后面要说的那点延迟——但只要把 Δ 压到秒级，用户几乎无感。这也是为什么几乎所有高吞吐遥测系统（不只是 Langfuse）都长成这个样子： 轻量接收 + 异步处理 + 批量落库。看懂了这条主干，你就能预判后面第三部分每一课在解决什么问题。

异步的代价：最终一致

把上下行拼起来，有一个你必须心里有数的事实：从 SDK 发出，到 UI 能查到，中间隔着队列处理，有秒级延迟。这叫最终一致。

「最终一致」在实践中意味着什么？意味着你刚在应用里跑完一次调用、马上切到 Langfuse 界面刷新，可能还看不到它——再等一两秒、刷新一下，它就出现了。 这对绝大多数可观测场景完全够用：你是事后来排查、来看板、来评估的，差几秒无所谓。但它也提醒你两件事：第一，别拿 Langfuse 当强一致的业务数据库—— 它是遥测系统，不保证「写完立刻可读」；第二，如果你写自动化测试去验证「埋点对不对」，要记得给摄取留出处理时间，否则会误判成「数据没上来」。 理解这条边界，你就不会对那点延迟感到意外，反而会欣赏它背后「永不拖累业务」的取舍。

为什么宁可有这点延迟，也要异步？ 因为摄取要扛极高的写入吞吐，而你的应用等不起。如果 API 同步地做完合并、算成本、写库才返回， 你的每次 LLM 调用都会被 Langfuse 拖慢、还可能因为下游抖动而失败。异步把「收件」和「处理」解耦：API 永远毫秒级返回、几乎不可能拖累你的应用； worker 则能批量、可重试、按队列扩容地消化洪峰。代价就是那点处理延迟——而第 2 课提到的「保留近实时调试」原则，要求把这个 Δ 压到秒级， 让可观测性依然「几乎实时」。用一点延迟，换一个永不拖累业务、又扛得住洪峰的摄取层。 这套「轻接收 + 异步处理」的取舍，会在第三部分被你反复看到——它不是某个模块的小聪明，而是贯穿整条摄取链路的指导思想。

给 LLM 应用埋点

第 5 课的全链路里，第一段是「SDK 创建事件」。这一课就把镜头对准这第一段：在数据还没离开你的应用之前，它是怎么被「埋点」生成的。 你会看到 Langfuse 接收数据的两个入口——一种是用 Langfuse 自家的 SDK 发「原生事件」，另一种是走业界标准的 OpenTelemetry（OTLP）。理解了「应用这头发出的是什么」，后面第三部分「服务端怎么收、怎么处理」才有的放矢。

🔌 生活类比
埋点就像给一栋大楼装传感器：你不改变大楼本身的功能，只是在关键位置（大门、电梯、机房）装上记录器，把「谁什么时候经过、停留多久」记下来。 Langfuse 的 SDK 是它自家定制的传感器，装起来最贴合、信息最全；而 OpenTelemetry 像一套通用接口的传感器标准—— 你楼里要是已经按这套标准装了线，Langfuse 直接接上现成的线就能收数据，不用重装。两条路，殊途同归：都是把应用运行的「踪迹」送出来。 这一课的目标，就是让你看清「传感器这头到底发出了什么」——只有先看懂应用发出的原料，才能理解后面服务端是怎么把它加工成 trace 树和宽事件表的。
两个入口：用 Langfuse SDK 发原生事件（web/src/pages/api/public/ingestion.ts），或用 OpenTelemetry OTLP 发标准 span（web/src/pages/api/public/otel/v1/traces，span 会被映射成 observation）。两条路最后汇入同一条服务端摄取链路（第三部分）。

事件信封：你的应用发出的是什么

无论走哪条入口，应用发出的本质都是一批事件（event）。Langfuse 定义了一组事件类型（packages/shared/src/server/ingestion/types.ts 的 eventTypes）， 每个事件描述「对哪个对象做了什么」：

事件类型含义
TRACE_CREATE创建/更新一个 trace（整次交互的外壳）
SPAN_CREATE / SPAN_UPDATE开始 / 结束一个 span 型 observation（有耗时的一段）
GENERATION_CREATE记录一次 LLM 调用（generation 型 observation）
OBSERVATION_CREATE / OBSERVATION_UPDATE通用的 observation 创建 / 更新
EVENT_CREATE记录一个瞬时事件
SCORE_CREATE提交一个 score（评分）

注意这里有 CREATE 也有 UPDATE——这正是第 5 课「合并」的来源：一个 span 通常先 SPAN_CREATE（开始时），结束时再 SPAN_UPDATE（补上耗时、输出）。 应用把这些事件攒成一批，一次 HTTP 请求发给摄取 API，服务端再去合并。所以你在 SDK 里写的 langfuse.trace(...)、span.end()， 本质上就是在生成这些事件。

概念上，你在应用里写的埋点代码大致长这样（不同语言 SDK 形态略有差异，这里只示意「调用 → 事件」的对应关系）：

# 创建一个 trace（发出 TRACE_CREATE）
trace = langfuse.trace(name="chat", user_id="u_42")

# 在 trace 下记录一次 LLM 调用（发出 GENERATION_CREATE）
gen = trace.generation(name="answer", model="gpt-4o", input=prompt)
result = call_llm(prompt)
# 调用结束，补上输出与用量（发出 OBSERVATION_UPDATE）
gen.end(output=result, usage={"input": 800, "output": 440})

# 给这次回答打个分（发出 SCORE_CREATE）
trace.score(name="helpfulness", value=0.9)

关键观察：你只是在原有业务代码旁边「顺手记一笔」，并没有改变 call_llm 本身的逻辑。好的埋点应当尽量不侵入—— 它不该决定你的应用怎么跑，只是旁观并记录。这也呼应了第 1 课的定位：Langfuse 站在你应用旁边，而不是里面。 正因为埋点是「旁路」的，它出问题（比如网络抖动发不出去）也不应拖垮你的主流程——SDK 通常会异步、批量、失败可丢地发送，把对应用的影响压到最低。把交互拆成「一串带类型的小事件」而不是「一个大对象」，正是为了让它们能分批、增量、乱序地发送。

两种入口：原生 SDK vs OpenTelemetry

为什么要同时支持两条入口？因为它们服务两类用户：

🎯 Langfuse SDK（原生）

Langfuse 官方的 Python/JS SDK，直接发上面那套原生事件。信息最全、最贴合（prompt 关联、用量、成本字段都能精确上报），是大多数人的首选。入口：ingestion.ts。

🔌 OpenTelemetry（OTLP）

如果你的系统已经用 OTel 做可观测，不想再引一套 SDK，就可以把 OTel 的 span 直接发到 Langfuse 的 OTLP 端点（otel/v1/traces）。Langfuse 把标准 span 映射成 observation。代价是有些 Langfuse 专有语义要靠约定俗成的属性名传递。

这是一个典型的「拥抱标准 vs 发挥专长」的权衡。OTel 是云原生可观测的事实标准，支持它意味着 Langfuse 能无缝接入已有的 OTel 生态—— 你不必为了用 Langfuse 就把埋点全部重写。但 OTel 的通用模型并不天然懂「prompt 版本」「token 用量」这些 LLM 专有概念，所以原生 SDK 在表达力上仍更强。 Langfuse 的选择是两者都要：用 OTLP 降低接入门槛，用原生 SDK 提供最佳体验。OTel 这条路的服务端处理，单独放在第 18 课讲。

create 然后 update：事件是「增量」的

再强调一遍这个对理解整条链路至关重要的点：一个 observation 的信息往往不是一次发齐的。看一次 LLM 调用的典型时间线：

t0 开始GENERATION_CREATEmodel=gpt-4o, input=…（这时还没有输出）
t1 结束OBSERVATION_UPDATEoutput=…, usage=1240 tok, latency=1.8s
服务端worker 把 create + update 合并成一条最终 observation（第 15 课）
两个事件，一条记录：先后到达的 CREATE 与 UPDATE 被 worker 合并成同一条最终 observation。这就是为什么 SDK 那头可以「拆开发」、而你在 UI 里看到的却是完整的一条。

为什么要这样「增量」上报，而不是等全部结束再发一条完整的？因为这样应用能更早地把「开始了」这件事记下来——即使后面调用崩了、超时了， 你至少知道「它确实开始过、卡在了这一步」。如果非要等结束才发，那一旦中途挂掉，这一步就彻底消失，反而最该被观测的失败 case 被漏掉了。 增量上报 + 服务端合并，是「边发生边记录」的可观测理念在协议层的体现。

🎯 设计取舍
把「事件增量上报 + 服务端合并」这套协议，复杂度放在哪一端？ 放在服务端。SDK 那头被刻意做得很「笨」——只管把发生的事拆成小事件发出去， 不负责拼装最终状态；真正麻烦的「把先后到达、可能乱序、可能重复的事件收敛成一致的一行」交给 worker（第 15 课）。这么分工的好处是：SDK 轻、对应用影响小、各语言都好实现； 而最复杂、最需要小心的合并逻辑只在一处（服务端）维护，不必每个语言的 SDK 各写一遍、各错一遍。把复杂度收敛到一个能严格测试的地方。这条「客户端薄、服务端厚」的取舍，在很多大型系统里都能见到，Langfuse 只是把它用在了遥测协议上。
🎯 本课要点
  • 埋点发生在你的应用里：把运行踪迹拆成一批事件发给 Langfuse，不改变业务逻辑。
  • 两个入口：Langfuse 原生 SDK（ingestion.ts，表达力最强）或 OpenTelemetry OTLP（otel/v1/traces，无缝接入已有 OTel）。
  • 事件类型（eventTypes）：TRACE_CREATE / SPAN_CREATE / GENERATION_CREATE / OBSERVATION_UPDATE / SCORE_CREATE …，描述「对哪个对象做了什么」。
  • 事件是增量的（create 开始、update 补全），所以「边发生边记录」，失败 case 也不丢——服务端负责合并（第 15 课）。
  • 协议把复杂度放在服务端：SDK 轻、好移植；合并逻辑只在一处维护。

双存储架构

上一课，数据以「事件」的形式离开了你的应用。这一课讲它们最终落在哪。很多人以为一个数据库就够了，但 Langfuse 用了四种存储—— Postgres、ClickHouse、Redis、S3，各管一摊。这不是炫技，而是因为不同数据的访问模式天差地别：用错存储，热路径会被活活拖垮。 看懂这四者的分工，你就理解了 Langfuse 性能与可扩展性的地基。

🔌 生活类比
把这四种存储想成一家公司的四种「存放方式」：Postgres 是文件柜——放重要档案（谁是员工、项目配置），要求准、可改、查得快，但量不大； ClickHouse 是巨型货仓——堆放海量货物（每天上亿条事件），要的是按品类批量盘点的能力；Redis 是传送带 + 暂存台—— 货物先上传送带排队（队列），常用的小东西放手边随手取（缓存）；S3 是外部仓储——存又大又不常翻的原件（事件底稿、上传的图片、导出的大文件）。 四者各有所长，硬塞进一个，必然顾此失彼。看懂这四种存储分别擅长什么、为什么这么分，是理解后面摄取与查询两条链路的前提。
四种存储的分工：Postgres=配置/元数据（事务、强一致、量小）；ClickHouse=宽事件（列存、海量、聚合快）；Redis=队列+缓存（内存、极快、临时）；S3=事件底稿/媒体/导出（对象存储、大、便宜）。web 偏读、worker 偏写，但都按数据性质选对应的库。

为什么不能只用一个

核心原因一句话：事务型负载（OLTP）和分析型负载（OLAP）是两种相反的优化目标，没有哪个库能同时把两头都做到极致。

  • 配置/元数据（项目、用户、API key、prompt 定义）是典型 OLTP：数据量小、要频繁精确读写单条、要强一致（创建了项目立刻就能用）、还有外键关系。这正是 Postgres 的主场。
  • 遥测数据（trace/observation/score）是典型 OLAP：数据量巨大（一天上亿）、写多于改、查询多是按时间窗口 + 某些维度做聚合（如「过去 7 天按模型看成本」）。这正是列存的 ClickHouse 的主场。

🗄️ Postgres 擅长（OLTP）

精确读写单条、强一致、外键关系、事务。适合小而重要、要立刻可用的配置类数据。不适合：海量行的聚合扫描。

📊 ClickHouse 擅长（OLAP）

对海量行的某几列做时间窗口扫描 + 聚合，列存 + 压缩。适合巨量、追加写、按维度统计的遥测数据。不适合：频繁改单条 + 强一致。

如果硬把遥测塞进 Postgres，海量行 + 频繁聚合会让它不堪重负；反过来把配置塞进 ClickHouse，它的弱事务、最终一致又满足不了「创建即可用」。 所以 Langfuse 的选择不是「贪多」，而是让每种数据待在最适合它的库里——这正是第 2 课「按列式访问设计」「谨慎反范式化」等原则在存储层的总落点。

两种相反的访问模式：OLTP 靠索引精确命中一行（读配置、改设置）；OLAP 要扫一大片行的某几列再聚合（按时间/模型统计成本）。行存数据库擅长前者，列存数据库擅长后者——这就是为什么配置归 Postgres、遥测归 ClickHouse。

🎯 设计取舍
四个存储 = 四份运维负担，为什么还要这么拆？ 第 2 课有一条原则：「把成本与运维简单性当作架构约束」——多一个数据库就要多一份备份、监控、升级、容灾。 Langfuse 之所以仍坚持四存储，是因为每一个都在解决一个单一存储解决不了的硬问题：没有 ClickHouse，海量聚合查询会慢到不可用；没有 Redis 队列，摄取就无法异步抗洪峰； 没有 S3，大字段和媒体会把数据库撑爆、也无法重放。换句话说，这四个不是「想加就加」，而是各自挣到了自己的存在。这也是判断「该不该再加一个存储」的标准—— 它解决的问题，是否值得它带来的长期运维成本。
🎯 本课要点
  • 四种存储各管一摊：Postgres（配置/元数据·OLTP）、ClickHouse（宽事件·OLAP）、Redis（队列+缓存）、S3（事件底稿+媒体+导出）。
  • 核心理由：OLTP 与 OLAP 是相反的优化目标，配置要精确单条+强一致，遥测要海量扫描+聚合，必须分库。
  • S3 是「真相之源」：事件先落 S3，ClickHouse 的宽事件是其合并后的派生表，可重放。
  • 控制面（Postgres）vs 数据面（ClickHouse）的划分，是第 3 课「领域形状 vs 物理存储」的宏观版。
  • 四存储不是堆技术，而是各自挣到了存在——这也是「该不该再加一个存储」的判断标准。

ClickHouse 与宽事件表

上一课说遥测数据都进 ClickHouse。这一课打开引擎盖，看这三张宽事件表（traces / observations / scores）到底是怎么建的、为什么这么建。 这是整个存储层最该看懂的一课——一旦理解了表引擎、排序键、分区这几样，后面「为什么列表快、为什么按项目按时间查特别高效、为什么数据能边写边查」 这些问题都会迎刃而解。我们以 observations 表为例（建表 SQL 在 packages/shared/clickhouse/migrations/unclustered/0002_observations.up.sql）。

🔌 生活类比
把 ClickHouse 的表想成一座超大图书馆。分区（PARTITION）像按出版年月分大书架——找「2024 年 6 月的书」直接去那个架，别的架碰都不用碰； 排序键（ORDER BY）像每个架内的摆放顺序（先按学科、再按日期），于是「某学科某段时间的书」只要在架上扫连续一小段； ReplacingMergeTree 引擎则像一条上架规则：同一个书号来了新版本，用新版盖掉旧版（按版本号判断谁新）。 三者配合，海量藏书也能秒级定位与盘点。这一课就把这三条「上架与摆放规则」逐一拆开，让你看清 Langfuse 是怎么在上亿行里做到「边写边查、查得还快」的。
一行宽事件的五类列：标识/关联列（拼树、定位）、内联的富属性宽列（model、cost、prompt…）、半结构化的 Map 列（metadata、usage/cost 明细）、用 ZSTD 压缩的大字段（input/output）、以及版本控制列（event_ts、is_deleted）。一行就装下了第 3 课说的全部富属性。

ReplacingMergeTree：同 id 后写覆盖先写

三张表用的引擎都是 ReplacingMergeTree。它的建表声明长这样：

clickhouse/migrations/unclustered/0002_observations.up.sqlENGINE
ENGINE = ReplacingMergeTree(event_ts, is_deleted)
PARTITION BY toYYYYMM(start_time)
ORDER BY (project_id, type, toDate(start_time), id)

关键就在 ReplacingMergeTree(event_ts, is_deleted) 这一句。它的语义是：排序键完全相同的多行，会被「替换」成最新的一行—— 谁的 event_ts（版本时间戳）更大，就保留谁；is_deleted=1 则表示这行被软删除。这正好完美契合第 5、6 课的合并模型： 一个 observation 先 create、后 update，会产生两次写入（相同 id、不同 event_ts），ClickHouse 在后台合并时自动用 update 那条盖掉 create 那条。 你不需要先查出旧行、改完再写回——只管追加，覆盖交给引擎。这就是第 2 课「不可变 / 追加式事件」能成立的存储基础。

后写覆盖先写：相同排序键的两次写入，引擎按 event_ts 保留更新的那条，is_deleted 实现软删除。于是「改一条 observation」= 追加一条更大 event_ts 的记录，无需读-改-写。

排序键 + 分区：为什么按 project_id 和时间

再看那两行：PARTITION BY toYYYYMM(start_time) 和 ORDER BY (project_id, type, toDate(start_time), id)。 这两句决定了数据在磁盘上怎么分块、怎么排列，也就决定了查询能不能少扫数据。在上亿行的规模下，能不能「少扫」往往比单点「扫得多快」更决定性能。

📅 分区 PARTITION BY 月

按 toYYYYMM(start_time) 把数据切成一月一块。查「最近 7 天」时，引擎直接跳过不相关月份的整块数据（partition pruning），连读都不读。

🔑 排序键 project_id 打头

ORDER BY 第一列是 project_id，于是同一项目的数据在磁盘上物理相邻。多租户查询天生只看自己项目，扫一段连续区间即可，无需全表过滤。

少扫才是关键：分区先按月裁掉无关数据，排序键（project_id 打头）再把目标项目的数据收拢成连续一段，最后列存只读你要的那几列。三者合力，让「按项目按时间」这类最高频查询在上亿行里也只碰一小片。

列存 + 压缩 + 索引：让「宽」也能快

宽事件行很宽，怎么保证不慢？靠三招，都写在建表 SQL 里：

手段建表里怎么写解决什么
列式存储MergeTree 家族天生列存查询只读用到的列，不为不相关的大字段买单

三张表的排序键略有不同，但都以 project_id 打头，体现「多租户隔离 = 排序键前缀」这一贯思路：

表ORDER BY
traces(project_id, toDate(timestamp), id)
observations(project_id, type, toDate(start_time), id)
scores(project_id, ...)

这种「三张表排序键高度一致」并非巧合，而是刻意为之：因为最常见的查询模式就是「某项目、某时间段」，把 project_id 和时间放在排序键最前， 等于把这类查询的成本压到最低。这也呼应第 3 课「三支柱 1:1 映射三张表」——三张表不仅结构对应，连物理排布的逻辑都一致，于是无论查 trace、查 observation 还是查 score， 引擎都能用同一套「裁分区 + 定区间」的本事。理解了 observations 这张，另外两张你就触类旁通了。

🎯 设计取舍
ReplacingMergeTree 的覆盖是「最终」的，不是「立刻」的——后台合并需要时间，所以理论上你可能在两次写入都还在、还没合并的瞬间读到旧行。 这是它换来「只管追加、写入飞快」的代价。怎么补？查询侧用 FINAL 关键字强制合并后再读，或用聚合（按 id 取 event_ts 最大的那条）来保证读到最终态—— 这就把「去重」的成本从每次写入挪到了查询时（且可控）。第 2 课说「避免读时去重的隐藏开销」，Langfuse 的做法是：尽量让常规查询走聚合/最终态视图， 把代价显式化、可优化，而不是让每次写入都付出读-改-写的代价。追加换吞吐，去重挪到读侧并优化——这是高写入遥测系统的经典取舍。
🎯 本课要点
  • 引擎 = ReplacingMergeTree(event_ts, is_deleted)：相同排序键，后写（event_ts 更大）覆盖先写，is_deleted 软删除——完美承接第 5、6 课 create+update 的合并模型。
  • 分区 = 按月（toYYYYMM），查时间窗口可整块裁剪；排序键 = project_id 打头，多租户查询只扫连续一段。
  • 「少扫」靠分区裁剪 + 排序键定位 + 列存只读所需列这三者合力，是海量数据下保持高效的关键。
  • 列存 + ZSTD 压缩 + Map 列 + bloom_filter 索引，让又宽又大的行也能查得快、存得省，不为不相关的大字段买单。
  • 取舍：覆盖是「最终」的，去重成本从写入挪到查询（FINAL/聚合），换来写入飞快——避免读时去重的隐藏开销。

元数据 schema（Postgres/Prisma）

前两课的主角是 ClickHouse（海量遥测）。这一课换到另一半：Postgres。如果说 ClickHouse 存的是「发生了什么」（trace/observation/score）， 那 Postgres 存的就是「这套平台是怎么配置的」——谁是你的组织和项目、谁有权限、有哪些 API key、定义了哪些 prompt、配了哪些评估规则、各模型怎么定价。 这一面叫控制面（control plane），数据量不大却事关全局。它的形状全写在 packages/shared/prisma/schema.prisma 里。

🔌 生活类比
把两套存储想成一家公司的两类系统：Postgres 是行政/人事系统——记录谁是员工、分在哪个部门、有什么权限、签了哪些规章。 数据不多，但要求准确、可改、立刻生效（新员工入职，权限马上得能用）。ClickHouse 则是业务流水账——海量交易记录， 平时只往里追加，事后拿来按月按品类统计。 你不会把规章制度记到流水账里，也不会把每笔交易塞进人事档案——两类数据，两套系统，各得其所。这一课就专门看「人事系统」这一半：它存了什么、为什么这么存。
两面分工：Postgres 存「平台怎么配置」——组织/项目/用户、API key、prompt、eval/数据集配置、定价、集成、审计；ClickHouse 存「实际发生了什么」——三张宽事件表。这正是第 3 课「领域形状 vs 物理存储」的宏观版：一个管控制、一个管数据。

Postgres 里装的是「控制面」

打开 schema.prisma，你会看到几十个模型。它们可以按用途归成几组（都用 @@map 映射成 snake_case 表名、用 cuid 做主键）：

分组主要模型管什么

看出规律了吗？这些全是「配置」和「定义」，而不是「海量发生的事」。它们数量级在千到百万之间，要被频繁精确地读写单条 （创建一个项目、改一个 prompt、查一把 key 是否有效），还要强一致——这正是关系型数据库 Postgres 的拿手好戏。

✅ 配置放 Postgres

要外键关系（project → 它的 prompt/key/配置）、要事务（一次改多张表要么全成要么全败）、要「改完立刻可读」。这些都是关系型数据库的强项。

❌ 配置不放 ClickHouse

ClickHouse 弱事务、最终一致，且不擅长频繁改单条。把「创建项目立刻可用」交给它，会出现刚建好却查不到的尴尬。配置类数据放它纯属错配。

第一层身份与租户
Organization / Project / User / Membership / Role / ApiKey——决定「谁、在哪个项目、有什么权限」。是其它一切的前提。
第二层功能配置
Prompt、Dataset、ScoreConfig、EvalTemplate/JobConfiguration、Model/Price——各功能的「定义与编排」，都挂在 project 下。
第三层集成与运维
各种 *Integration、BatchExport/Action、Media、AuditLog、BackgroundMigration——对外连接与平台自身的运维记录。

Project 是枢纽

在这几十个模型里，Project 是当之无愧的中心。打开它的定义你会发现，它字段不多（id、orgId、name、deletedAt 软删、retentionDays…）， 但挂着几十条关系——几乎所有其它东西都从属于某个 project：

Project 是控制面的中心：Organization 之下，几乎一切（API key、成员、prompt、数据集/eval、模型/定价、仪表盘、自动化/监控、集成、媒体/导出）都1:N 地挂在某个 project 上。所以 project 是天然的「数据边界」——一把 API key 属于某个 project，你查的所有数据也都被限定在它所属的 project 里（多租户，下一课展开）。

「project 是枢纽」还带来两个很实在的后果。其一是级联：删掉一个 project，它名下的 key、prompt、数据集、配置会跟着被清理（schema 里很多关系标了 onDelete: Cascade）， 这让「删一个项目」变成一个干净的操作，而不是到处留下孤儿数据。其二是边界即权限：因为一切都挂在 project 下，鉴权只要回答「你属不属于这个 project」就够了—— 这也是下一课多租户、以及第 21 课 tRPC 中间件 protectedProjectProcedure 的工作基础。记住这张「以 project 为中心」的图，后面很多看似分散的功能，其实都挂在它上面。

为什么 Postgres 里还有 trace 表？

翻 schema.prisma 时你可能会困惑：怎么还有 LegacyPrismaTrace、LegacyPrismaObservation、LegacyPrismaScore？ 不是说遥测都在 ClickHouse 吗？答案在名字里——Legacy（遗留）。这是 Langfuse 早期把 trace/observation/score 存在 Postgres 的历史残留； 后来为了应对规模，分析数据整体迁到了 ClickHouse（这正是第 7、8 课的存在理由）。这些 legacy 模型留着主要是兼容与迁移用，不再是热路径。 看到带 Legacy 前缀的，就知道「这是老路，别照着学」。

🎯 设计取舍
把「控制面」和「数据面」分到两个数据库，最大的好处是什么？ 是让两者能各自按自己的规律演进与伸缩。控制面数据量小、改动要强一致， 适合一台（或主从）Postgres 稳稳扛住；数据面是洪流，要水平扩展的 ClickHouse 集群。如果硬塞进一个库，要么委屈了配置的强一致，要么拖垮了遥测的吞吐。 代价是：一次操作有时要跨两个库（比如摄取时要回 Postgres 查模型定价、查 prompt 关联）——但这种「读配置、写数据」的跨库协作是可控且高频缓存的， 远比「让一个库同时做好 OLTP 和 OLAP」现实。按职责分库，让每个库只做自己最擅长的事。
🎯 本课要点
  • Postgres = 控制面：存「平台怎么配置」——org/project/user、API key、prompt、eval/数据集配置、模型定价、集成、审计。定义在 schema.prisma。
  • Project 是枢纽：字段不多但挂着几十条关系，几乎一切都 1:N 属于某个 project——它是天然的数据边界（多租户基础）。
  • LegacyPrismaTrace/Observation/Score 是历史残留：分析数据早已迁到 ClickHouse，这些只为兼容/迁移而留。
  • API key 存哈希、不存明文；权限用 Role（OWNER/ADMIN/MEMBER/VIEWER/NONE）+ 组织/项目 membership（第 49、53 课）。
  • 控制面 vs 数据面，是第 3 课「领域形状 vs 物理存储」的宏观版：按职责分库，各做擅长的事。

多租户：org → project → environment

上一课说 project 是枢纽、是「数据边界」。这一课把这件事讲透：Langfuse 是个多租户平台——同一套部署要同时服务成千上万个互不相干的团队， 它靠三层嵌套把大家的数据干净地隔开：organization（组织）→ project（项目）→ environment（环境）。 理解这三层，你就明白了「为什么你只看得到自己的数据」「为什么 ClickHouse 排序键非要以 project_id 打头」。

🔌 生活类比
把多租户想成一栋写字楼：organization 是租下楼层的公司——签合同、付账单、管自己的员工（成员与计费在这一层）； project 是公司里的一个部门，有独立的办公区和门禁，部门的东西不会跑到别的部门去；environment 则是部门内部再隔出的 「正式区 / 测试区」——同一个部门的人，把线上数据和测试数据分开摆，互不干扰。大家可能同在一栋楼（同一套部署、同一个数据库）， 但靠层层门禁谁也串不了门。
三层嵌套：一个 organization 下有多个 project，每个 project 内可再分多个 environment（如 production / staging / development，默认 default）。组织管计费与成员，project 是硬数据边界，environment 是项目内的软切片。

三层各管什么

层级管什么隔离强度
Organization 组织计费、成员、SSO、组织级 API key最外层的「账户」边界
Project 项目实际数据归属：trace/observation/score、prompt、配置全挂在某 project 下硬隔离：跨 project 互不可见
Environment 环境同一 project 内区分 production / staging / development 等软切片：同 project 内可一起查、也可按环境过滤

关键区别在最后两层：project 是「硬边界」——你拿着某 project 的 key，根本看不到别的 project 的数据； 而 environment 是「软切片」——它在同一个 project 内部，把线上和测试的数据贴上标签分开，你既能只看 production，也能把几个环境放一起比。 为什么这么设计？因为「不同团队的数据绝不能串」是安全红线（硬隔离），而「线上 vs 测试」只是同一团队自己的视图偏好（软切片），强度不同，待遇也不同。

举个具体场景体会这三层。一家公司（organization）在 Langfuse 上签约、付费、邀请了 5 名工程师做成员。他们开了两个项目（project）： 「客服机器人」和「文档问答」。这两个项目的数据互不可见——查客服机器人时绝不会混进文档问答的 trace。在「客服机器人」项目内，他们又用 environment 把线上流量（production）和压测/调试流量（staging）分开，于是看线上成本时不会被压测数据污染， 但需要时也能把两个环境放一起对比。最外层的「公司」只关心账单和谁是成员，不直接持有数据；数据全都落在某个 project + environment 上。 这套「组织管钱和人、项目管数据、环境管视图」的分工，就是多租户的全部骨架。

project_id 是硬隔离键，贯穿始终

多租户隔离不是事后加一个 WHERE project_id = ? 过滤这么简单。在 Langfuse 里，project_id 是一条贯穿全链路的主线：

project_id 贯穿全链路：鉴权时从 API key 解析出 project_id（第 9 课 key 属于某 project）；写入时每行都带上它；ClickHouse 的排序键以 project_id 打头（第 8 课）；查询时据它定位到连续区间。所以隔离不是事后过滤，而是写进了物理排布——隔离和「查得快」是同一件事。

这条「project_id 贯穿全链路」的设计，最大的价值是把安全从「靠程序员记得写过滤」变成「靠结构保证」。想想看：如果隔离只靠每个查询都记得加 WHERE project_id = ?， 那只要有一处查询忘了写、或写错了，就可能让 A 公司看到 B 公司的数据——这种 bug 在多租户系统里是灾难级的。Langfuse 的做法是把 project_id 焊进存储与鉴权的底层： 仓储层的读取 API 统一要求传入 project_id，tRPC 的 protectedProjectProcedure 在中间件里先校验你属不属于这个 project（第 21 课）。 于是「越权看别人数据」不是「容易写错」，而是压根没有那条路。这就是「安全做进结构」与「安全靠自觉」的本质区别。

environment：项目内的软切片

environment 是后来加上的一层。看它在 ClickHouse 里的样子——三张宽事件表都加了一个 environment 列：

clickhouse/migrations/unclustered/0008_add_environments_column.up.sqlALTER
ALTER TABLE traces       ADD COLUMN environment LowCardinality(String) DEFAULT 'default' AFTER project_id;
ALTER TABLE observations ADD COLUMN environment LowCardinality(String) DEFAULT 'default' AFTER project_id;
ALTER TABLE scores       ADD COLUMN environment LowCardinality(String) DEFAULT 'default' AFTER project_id;

三个细节值得注意：① 类型是 LowCardinality(String)——环境名翻来覆去就那么几个（prod/staging/dev），低基数编码又省空间又查得快； ② 默认值 'default'——你不显式指定环境时，数据自动落到 default，向后兼容（老数据天然属于 default）；③ 位置 AFTER project_id—— 这只是让它在表结构里紧挨 project_id 排列（纯粹是列的书写顺序，便于阅读），注意它并没有进入排序键。所以和 project_id 不同，environment 本质是行上的一个 普通过滤列，也就是软切片：想按环境过滤就在查询里加个条件，不想分就忽略它。

这正好对照出 project 与 environment 隔离机制的不同：project_id 焊进了排序键（隔离即定位，结构性安全），而 environment 只是一个普通列（查询时过滤）。 这也合理——硬隔离要靠结构保证，软切片只是个方便的视图维度，不必、也不该动用排序键这种重武器。

🔒 project = 硬隔离

安全红线：不同 project 的数据绝不可见。焊进排序键前缀 + API key 归属，从存储到鉴权层层保证。

🏷️ environment = 软切片

视图偏好：同一 project 内给数据贴 prod/staging/dev 标签，可分可合，只是查询时的一个过滤维度。

2

3

🎯 设计取舍
为什么把 project_id 焊进排序键，而不是当成一个普通的过滤列？ 因为这样「隔离」和「高效」就成了同一件事。如果 project_id 只是个普通列， 那「只看我的项目」就是一次全表过滤，海量数据下既慢又危险（万一过滤写漏了，就串了租户）。把它放进排序键最前面，同一 project 的数据物理相邻， 「只看我的项目」退化成「扫一段连续区间」——又快又结构性地不可能看到别人。代价是排序键一旦定了就难改、且要求几乎所有查询都带上 project_id（这正是仓储层 API 的统一约定）。 把多租户隔离做成存储结构的一部分，而不是一道可能写漏的过滤。
🎯 本课要点
  • 三层嵌套：organization（计费/成员）⊃ project（数据边界）⊃ environment（项目内 prod/staging/dev）。
  • project = 硬隔离（跨项目绝不可见），environment = 软切片（同项目内可分可合的标签）。
  • project_id 贯穿全链路：从 API key 解析 → 写入每行 → ClickHouse 排序键打头 → 查询定位。隔离不是事后过滤，而是焊进排序键。
  • environment 列：LowCardinality(String) DEFAULT 'default' AFTER project_id（第 8 课迁移 0008），低基数、向后兼容、紧贴 project_id。
  • 取舍：把隔离做进存储结构 → 隔离与高效合一、且结构性安全；代价是排序键难改、查询须带 project_id。

部署拓扑与依赖

第二部分讲了数据模型与存储「是什么」。这一课收尾，讲「跑起来要什么」——自托管一套 Langfuse，你实际要启动哪些容器、它们怎么连。 答案很干净：2 个应用容器 + 4 个基础设施，全写在根目录的 docker-compose.yml 里。把前面几课抽象的架构图，落到这张具体的「装箱清单」上， 你对 Langfuse 的「形」就完整了。

🔌 生活类比
把一套 Langfuse 部署想成开一家餐厅需要的几个「房间」：前厅（web）接待客人、上菜单、端菜；后厨（worker）在后面默默备菜、洗碗、处理大单； 冷库（ClickHouse）存海量食材（遥测）；文件柜（Postgres）放菜单和员工档案（配置）；传菜口（Redis）排订单（队列）； 大仓库（S3/MinIO）堆又大又不常用的存货（原件、媒体）。 少了哪个房间，餐厅都开不利索——但凑齐这 6 个，店就能营业了。这一课就带你认全这六个「房间」，看清它们各自是谁、为什么少不了、又怎么连成一家能营业的店。
自托管装箱清单：两个应用容器 langfuse-web（langfuse/langfuse:3）与 langfuse-worker（langfuse/langfuse-worker:3），加四个基础设施 postgres:17 · clickhouse-server · redis:7 · minio（S3 兼容）。每个基础设施配一个 named volume 持久化。这正是第 5、7 课架构图的「实体版」。

注意 minio：它是一个兼容 S3 协议的开源对象存储，在本地/自托管时充当 AWS S3 的替身。这意味着同一套代码， 在云上用真 S3、在自托管用 MinIO，应用层完全无感——这正是「面向接口（S3 协议）而非实现」带来的可移植性。四个基础设施容器各自挂 named volume 做持久化 （postgres / clickhouse data+logs / redis / minio），而 web 与 worker 无状态、不挂卷；所以你 docker compose down 再 up，数据还在。

应用层web · worker
无状态的业务容器，可随时重启、按需扩容。它们启动后连上下面的存储干活。
数据层postgres · clickhouse
两类数据库：控制面（配置）与数据面（遥测）。有状态，靠 named volume 持久化，是整套系统的根基。
支撑层redis · minio
队列/缓存与对象存储。让摄取异步、让大字段/媒体有处可放。同样有状态、需持久化。

这三层有个朴素的启动次序：先把数据层与支撑层拉起、等它们健康，再启动应用层——所以 compose 里应用容器会 depends_on 这些基础设施并等待其 healthcheck。 理解这个分层，你 debug 部署问题时就有了方向：连不上库先看数据层、摄取不动先看 Redis、媒体存不进先看 MinIO。

web 与 worker：分工与协作

两个应用容器是 Langfuse 的「左右手」。它们形态相似（都连同一套存储），但职责相反：

左右手分工：web 面向用户、追求低延迟、可随时重启；worker 啃重活、可重试、按队列积压伸缩。二者通过 Redis 队列解耦（第 5 课），所以你能分别给它们扩容——用户多了加 web，队列堵了加 worker。这正是第 4 课「窄腰」让两个容器独立演进的部署体现。

web 怎么扩

无状态、面向请求 → 看并发用户 / QPS，水平多开几份、前面挂负载均衡即可。

worker 怎么扩

面向队列 → 看队列积压，积压上来就多开 worker 实例消费（还能按队列分片，第 14 课）。

配置全靠环境变量

这六个容器怎么知道彼此地址、密钥、各种开关？答案是环境变量。仓库里有一组 .env.*.example 模板（如 .env.dev.example、.env.prod.example）， 列出了所有需要配置的变量（数据库连接、S3 端点、加密密钥…）。更重要的是，这些 env 不是随便读的——它们经过 Zod 校验 （packages/shared/src/env.ts、worker/src/env.ts 等），启动时就检查类型与必填项。配错了、漏填了，进程会直接启动失败并报清楚哪个变量有问题， 而不是带病运行到半路才崩。这条「启动即校验配置」的纪律，是把「配置错误」从难查的运行时 bug 变成一眼可见的启动错误（第 51 课展开）。

把这些拼起来，第一次自托管的体验其实很简单：克隆仓库 → 复制一份 .env.dev.example 填好关键变量 → docker compose up， 六个容器就按依赖次序拉起、连通，几分钟后你就能打开浏览器看到 Langfuse 界面。仓库里还备了多套 compose 文件应对不同场景—— docker-compose.dev.yml（本地开发，常只拉基础设施、应用跑在本机便于热重载）、docker-compose.yml（完整一套）、 以及针对特定云的变体（如 Azure）。这种「一份编排管全栈、按场景给变体」的安排，正是把前面说的「四依赖的复杂度」收进了几个 yml 文件里—— 你不需要手动一个个装数据库、配网络，compose 替你把这些都串好了。理解了这张拓扑，你不仅会「跑起来」，遇到问题也知道该去哪个容器看日志。

🎯 设计取舍
四个基础设施依赖，对自托管者是不是负担？ 坦白说，是——相比「一个二进制 + 一个 SQLite」，Langfuse 的自托管门槛确实更高。但这正是第 7 课那条取舍的部署侧后果： 为了能扛生产级规模（海量遥测、异步摄取、对象存储），这四个组件每一个都不可省。Langfuse 的缓解办法是用 docker-compose 一键拉起全部六个容器、用 MinIO 替身免去对云 S3 的依赖， 让「在自己机器上跑起完整一套」变得简单。复杂度无法消除，但可以被打包好、一键交付。这也是为什么官方强调「self-host in minutes」——不是没有依赖，而是把依赖编排好了。对自托管者来说，真正要权衡的不是「依赖多不多」，而是「这套规模化能力你用不用得上」。
🎯 本课要点
  • 最小自托管栈 = 2 应用 + 4 基础设施：langfuse-web + langfuse-worker，加 postgres · clickhouse · redis · minio(S3)，全在 docker-compose.yml。
  • web vs worker：同连一套存储但职责相反——web 面向用户低延迟、按 QPS 扩；worker 啃重活、按队列积压扩；靠 Redis 队列解耦。
  • MinIO 是 S3 协议替身：同一套代码云上用真 S3、本地用 MinIO，应用无感。
  • 配置全靠环境变量，并经 Zod 启动即校验（env.ts）——配错就启动失败、报清楚，而非带病运行。
  • 四依赖是规模化的必要代价（第 7 课），用 docker-compose 一键编排来缓解——「依赖不少，但都打包好了」。

摄取 API

第二部分把「数据长什么样、存在哪」讲完了。从这一课起，我们正式走摄取链路——也就是第 5 课那条「写入主干」的放大版。 第一站是公共摄取 API：你的应用发来一批事件，服务端这一头是怎么收下的。剧透一下它的性格——极快、极轻： 它只做「鉴权 + 校验 + 入队」三件事，然后立刻返回，绝不在这里合并、算成本、写库（那些是 worker 的活）。这就是第 5 课说的「快路」。

🔌 生活类比
把摄取 API 想成快递的收件口：快递员（你的 SDK）抱来一批包裹（events），收件员（API）做三件事就放行—— 核对寄件人身份（鉴权：你这把 key 是哪个项目的？）、扫码登记面单（校验：每个包裹的格式对不对？）、丢上传送带（入队）。 然后当场给你一张回执（HTTP 200），就让你走了——不会当着你的面去分拣、称重、送到目的地。那些慢活，传送带后面的人（worker）慢慢做。 收件口的唯一目标，就是收得飞快、永不堵门。
极简的收件流程：鉴权（API key 解析出 project，第 9·10 课）→ Zod 校验事件格式 → processEventBatch 把事件入队（同时把原件落 S3）→ 立即 200。整条链路上「重活」一件都不在这里做，所以 API 永远毫秒级返回。

一个摄取请求的一生

入口在 web/src/pages/api/public/ingestion.ts 的 handler。它收下请求体后，真正干活的是一个叫 processEventBatch 的共享函数：

web/src/pages/api/public/ingestion.tshandler
import { processEventBatch } from "@langfuse/shared/src/server";

export default async function handler(req, res) {
  // …鉴权、解析 body…
  const result = await processEventBatch(batchForProcessing, authCheck);
  // 立即把结果返回（成功/部分失败），不等 worker 处理
}

注意一个设计细节：processEventBatch 是共享的——除了 /ingestion，连「单独提交 score」的接口也复用它。 为什么？因为「收一批事件、校验、入队」这套逻辑到处都用得上，抽成一个函数就不必每个端点各写一遍、各错一遍（又是第 6 课「复杂度收敛到一处」的体现）。 它支持的事件类型涵盖了 TRACE_CREATE / OBSERVATION_CREATE / SPAN_CREATE / GENERATION_CREATE / SCORE_CREATE 等 （还包括 SDK_LOG、DATASET_RUN_ITEM_CREATE）——正是第 6 课说的那套事件信封。

为什么是「一批」，不是「一条」

SDK 不是每产生一个事件就发一次 HTTP，而是攒成一批再发。这个「批」的设计省了大事：

😖 一条一发

每个 observation 都来一次 HTTP 往返，高频应用下网络开销巨大、连接数爆炸，对你的应用是实打实的拖累。

😀 攒批一发

几十上百个事件打成一个请求，一次往返搞定。网络开销摊薄，API 也能一次性校验、一次性入队，吞吐高得多。

批处理还带来一个语义上的好处：同一批里可能既有某 observation 的 CREATE 又有它的 UPDATE，服务端可以整批一起处理， 减少来回。当然，批也不能无限大——请求体过大时 API 会拒绝（返回「请求体过大」），这也是一道保护：别让单个请求把服务端撑爆。

1

SDK 攒批 → POST /ingestion

应用侧把一段时间内的事件打成一批，一次发出。

2

handler 鉴权

从 API key 解析 project、校验写入权限（失败整批拒）。

3

Zod 校验每个事件

结构不对的事件被拒；其余继续（可部分成功）。

4

processEventBatch 入队 + 落 S3

合法事件进 Redis 队列、原件存 S3（第 14 课）。

5

立即返回结果

告诉调用方哪些收下、哪些被拒——全程毫秒级，不等 worker。

守门的两道关：鉴权 + 校验

API 在「入队」之前，必须先过两道关。它们看似例行公事，却是整个系统安全与健壮的第一道防线：

两道关：鉴权从 API key 解析出 project（第 9 课 key 属于某 project）并确认有写入权限——这一步也把租户钉死（第 10 课 project_id 贯穿）；Zod 校验逐个事件检查结构，坏数据在门口就被拒，不会污染下游。两关都过，才允许入队。

第二道关「Zod 校验」尤其重要：摄取系统每天吞下海量来自各种 SDK、各种版本的数据，难免有格式不对、字段缺失的「脏事件」。 在入口就用 Zod schema 把它们挡下、给出清晰错误，比让它们流进队列、到 worker 处理时才崩要好得多——越早拒绝坏数据，整个系统越省心。 这一批里如果有几个坏事件，API 还能返回「部分成功」，告诉你哪几条没收下，而不是整批一刀切。

这一关把守什么失败会怎样
鉴权API key 有效吗？属于哪个 project？能写吗？401/403，整批拒绝
Zod 校验每个事件的结构/类型是否合法坏事件被拒，可返回部分成功
体积保护请求体是否过大返回「请求体过大」，防撑爆

「部分成功」不是说说而已，它写进了响应契约：ingestion.ts 的 handler 最后一行是 res.status(207).json(result)， 而 result 形如 { successes, errors }——两个数组，分别列出收下的事件与被拒的事件（每条带错误原因）。207 Multi-Status 这个 HTTP 状态码 天生就是为「一个请求里多个子操作、结果各异」设计的。于是即便你这批 100 个事件里混进 3 个脏的，服务端也会收下 97 个、退回 3 个并告诉你为什么， 而不是因为 3 颗老鼠屎把整锅粥倒掉。这对 SDK 极其友好：客户端可以只重传失败的那几条，无需整批重来。

🎯 设计取舍
为什么 API 只做「鉴权+校验+入队」，把一切重活都推给 worker？ 因为摄取 API 是直面你应用的那一层，它的首要 KPI 是 「永远快、永远不拖累调用方」。任何放在这里的额外工作——合并、查定价、写库——都会变成你每次埋点调用的延迟，还会因为下游抖动而失败。 所以 Langfuse 把这条线切得极干净：API 端只保留「无论如何都必须当场做」的最小集合（鉴权防止越权写入、校验防止脏数据入库、入队完成交接）， 剩下能异步的一律异步。代价是「最终一致」（第 5 课），但收益是一个几乎不可能慢、不可能拖垮你业务的摄取入口。入口越薄，越扛得住洪峰。
🎯 本课要点
  • 摄取 API 的全部职责：鉴权 + Zod 校验 + 入队，然后立即返回——合并/算成本/写库都不在这里（是 worker 的活）。
  • 入口 ingestion.ts 的 handler 调用共享的 processEventBatch（连单独提交 score 的接口也复用它）。
  • 攒批上报：几十上百个事件一次 HTTP，摊薄网络开销、一次校验一次入队；批过大会被拒（体积保护）。
  • 两道关：鉴权（key→project、权限，把租户钉死）+ Zod 校验（坏数据门口就拒，可部分成功）。
  • 取舍：入口只留「必须当场做」的最小集合，能异步的全异步——入口越薄越扛洪峰，代价是最终一致。

事件类型与合并语义

上一课，API 把一批事件收下并入队。这一课我们把镜头怼到单个事件上：它长什么样？为什么有 create 和 update 两副面孔？ 最关键的是——你的 SDK 先后发来的好几个事件，最后怎么会变成数据库里同一行？这就是 Langfuse 摄取的灵魂：事件流 → 合并成记录。 看懂这一课，第 8 课那张 ReplacingMergeTree(event_ts, is_deleted) 的设计才算真正落地。

🩺 生活类比
把一个观测想成医院里的一份病历。病人挂号那一刻，医生写下第一条就诊记录（generation-create：开始时间、输入的主诉）； 检查做完、开完药，医生补一条记录（generation-update：结束时间、诊断结论、用了多少药）。这两条记录都贴着同一个病人号（实体 id）—— 护士归档时，不会建两份病历，而是把它们合并进同一份，冲突字段以最新一条为准。而每条记录本身还有自己的流水号（事件 id）， 保证同一条记录归档两次也不会重复。「病人号合并、流水号去重」——正是这一课要拆解的两套 id。

事件的信封：id · timestamp · type · body

每个摄取事件，无论它描述的是 trace、observation 还是 score，都套着同一个信封。信封里有一个公共的「外壳」，再加一个随类型而变的「内容物」body。 Langfuse 用 Zod 把它定义得一清二楚：先有一个 base（外壳），各类型在它上面 extend 出自己的 type 与 body， 最后用 判别联合（z.discriminatedUnion("type", […])）把十几种事件拢成一个 schema。校验时，Zod 看 type 这个字段， 就知道该用哪一套 body 规则去验——这就是「一个字段定身份，一套规则配一种身」。

信封 = 固定外壳 + 可变 body：base 持有 id（事件 id）、timestamp、metadata 和判别字段 type；type 决定 body 套用哪一套 Zod 规则。这正是 z.discriminatedUnion("type", […]) 的工作方式。源码：packages/shared/src/server/ingestion/types.ts:597。
packages/shared/src/server/ingestion/types.tsL597
const base = z.object({
  id: idSchema,                        // 事件 id（流水号）
  timestamp: z.iso.datetime({ offset: true }),
  metadata: jsonSchema.nullish(),
});

const generationCreateEvent = base.extend({
  type: z.literal(eventTypes.GENERATION_CREATE),
  body: CreateGenerationBody,          // 随 type 而变的内容物
});
// …span-create / span-update / score-create / 共 18 种…

const ingestionEvent = z.discriminatedUnion("type", [
  traceEvent, scoreEvent, spanCreateEvent, spanUpdateEvent,
  generationCreateEvent, generationUpdateEvent, /* … */
]);

判别联合的妙处在于：它不是「把所有字段都设成可选、随便你填」，而是每种 type 配一套严格的 body 规则。 发 generation-create 就必须符合 CreateGenerationBody，发 score-create 就必须符合 ScoreBody——填错了类型、缺了字段，第 12 课那道 Zod 关当场就拦下。 而且这套 schema 还分公共版与内部版两个实例：像 dataset-run-item-create 这种就只允许内部使用，公共 API 发它会被拒。

body 的「俄罗斯套娃」：Event ⊂ Span ⊂ Generation

那些 body 不是各写各的，而是层层继承出来的。最里层是 OptionalObservationBody（traceId、name、startTime、input/output…）， 往外 extend 一层加 id 成 CreateEventEvent，再加 endTime 成 CreateSpanBody，再加 model / usage / modelParameters 成 CreateGenerationBody。 这条链精确对应第 3 课的观测三型：EVENT 是时间点（无时长）、SPAN 有起止（多了 endTime）、GENERATION 是带 LLM 字段的 SPAN（多了 model/usage）。 有趣的是，第 8 课提到的那些新观测类型——AGENT / TOOL / CHAIN / RETRIEVER / EVALUATOR / EMBEDDING / GUARDRAIL——它们的 body 全都复用 CreateGenerationBody： 本质上都是「generation 那么丰富的一段观测」，只是语义标签不同。

越往外，能力越强：EVENT 是无时长的时间点 → SPAN 加 endTime 有了起止 → GENERATION 再加 model/usage 成为可记 token 与成本的 LLM 调用。AGENT/TOOL/CHAIN 等 7 种新类型直接复用 CreateGenerationBody。源码：types.ts:421-470。

这套继承不仅省代码，更让类型语义层层递进：你不可能给一个 EVENT 填 endTime（它的 body 里压根没这字段），也不可能给一个 SPAN 填 usage。 schema 即文档——读完这条链，三型观测「能装什么、不能装什么」一目了然。

灵魂所在：好几个事件，怎么变成一行

现在来回答开篇那个问题。一次 LLM 调用，SDK 通常发两个事件：开始时 generation-create（带 startTime、input），结束时 generation-update（带 endTime、output、usage）。 它们最终却落成 ClickHouse 里同一行 observation。秘密在于两套 id 各司其职：

实体 idbody.id（合并键）
观测/trace 本身的 id。同一个 body.id 的所有事件，注定合并成一行。create 和 update 共享它，所以它们指向同一条记录。
事件 id信封 base.id（去重键）
每个事件自己的流水号。它在 S3 里成为文件名 <eventId>.json——同一事件重发，覆盖同一个文件，天然幂等，不会算两次。
版本timestamp（谁更新）
合并冲突字段时，以时间更晚的事件为准（last-write-wins）。这正是第 8 课 ReplacingMergeTree(event_ts) 在写入侧的镜像。
create + update → 一行：两个事件共享 body.id=obs-42，被分到一组合并；各字段取有值的一方，冲突时以 timestamp 更晚者为准。worker 端的合并逻辑（第 15 课）正是第 8 课 ReplacingMergeTree 行为的「写入侧预演」。

把这条「事件流 → 一行」的路径拆成五步，合并的来龙去脉就完全清楚了——前两步在你的应用侧发生，后三步在 Langfuse 服务端发生：

1

开始：发 create

LLM 调用一开始，SDK 发 generation-create，带上 body.id、startTime、input。发完即忘，不在内存里攥着。

2

结束：发 update

调用结束，SDK 再发 generation-update，沿用同一个 body.id，补上 endTime、output、usage。

3

按 body.id 分组

服务端把 dedupKey = 实体类型-body.id 相同的事件归到一组——这两条就被认成「同一条记录的两次上报」。

4

合并字段（最新者胜）

逐字段取有值的一方；create 与 update 都给了的字段，以 timestamp 更晚的为准。

5

落成一行

合并结果写进 observations 表的同一行（id=body.id）；ReplacingMergeTree 在后台再做一次同样的去重兜底（第 8 课）。

🎯 设计取舍
为什么不直接做一个 upsert，非要拆成 create / update 两个事件？ 因为SDK 端不该背状态。一次 LLM 调用，开始和结束之间可能隔着几秒、几十秒， 期间应用甚至可能崩、可能换机器。如果要求「等结束后一次性 upsert 全量」，SDK 就得在内存里攥着这条记录直到结束——既占内存又怕丢。 拆成 create + update，SDK 可以开始就发一半、结束再补一半，发完即忘；合并的脏活交给服务端按 body.id 去拼。把状态从客户端推给服务端， 正是分布式系统里「让边缘无状态、让中心兜底」的经典思路——也呼应第 5 课那条「快路 / 慢路」分工。
🎯 本课要点
  • 每个事件 = 信封（base: id · timestamp · metadata + 判别字段 type）+ 随 type 而变的 body，由 z.discriminatedUnion("type") 拢成一个 schema。
  • body 是继承链：OptionalObservationBody → +id CreateEventEvent → +endTime CreateSpanBody → +model/usage CreateGenerationBody，对应 EVENT⊂SPAN⊂GENERATION。
  • 两套 id：实体 id（body.id）是合并键——同 id 的 create/update 汇成一行；事件 id（信封 id）是去重键——成为 S3 文件名，保证幂等。
  • 冲突字段以 timestamp 更晚者为准（last-write-wins），是第 8 课 ReplacingMergeTree(event_ts) 在写入侧的镜像。
  • getClickhouseEntityType 把 18 种事件 type 坍缩成 3 个实体类型（trace / observation / score）→ 第 8 课的三张宽表。

摄取队列

第 12 课 API「入队」后就闪人了，第 13 课讲清了入的是什么「事件」。这一课的主角，就是那条队列本身——Langfuse 用 Redis + BullMQ 搭起来的传送带。它是第 5 课「快路 / 慢路」之间的交接点：web 把活儿往上一放就走人，worker 在另一头不紧不慢地取走慢慢做。 这一课要回答三个问题：队列里到底放了什么（剧透：不是事件本体，而是一张「取件票」）、为什么要分主队列 / 次队列、以及高并发下怎么分片扛住洪峰。

🛄 生活类比
把队列想成机场的行李传送带。值机柜台（API）不会把你的行李箱直接堆在柜台上——它把箱子送进后仓（S3），只在传送带上放一张行李条（指针：箱子在后仓的编号）。 分拣员（worker）从传送带上取下行李条，照着编号去后仓取箱子，再处理。这样设计的妙处：传送带（Redis 内存）只跑轻飘飘的纸条，又快又省； 真正笨重的箱子（事件本体，可能很大）安安静静躺在后仓。万一分拣员忙不过来，纸条就在带上排队等着，箱子一件都不会丢——因为后仓才是货物的「真账本」。

队列里放的是「取件票」，不是事件本体

这是整条摄取链路最容易误解的一点：Redis 队列里并不存事件的完整内容。第 12 课 processEventBatch 在入队前，先把每个事件按 <eventId>.json 的名字写进 S3（第 13 课的事件 id 在这里当文件名）；队列里只放一个轻量任务，载荷里关键的就一个 fileKey—— 指向 S3 里那个文件。worker 取到任务，再照着 fileKey 去 S3 把事件读回来。

指针与本体分离：事件本体写入 S3（<eventId>.json），队列里只放含 fileKey 的轻量任务。worker 取任务后照 fileKey 回 S3 读本体。源码：processEventBatch.ts:340-398（queue.add(IngestionJob, {payload:{data:{type, eventBodyId, fileKey,…}, authCheck}}})）。

const shardingKey = `${projectId}-${eventData.eventBodyId}`;   // 分片键（见下）

📨 队列只放指针

载荷是 { type, eventBodyId, fileKey, authCheck } 这种几十字节的小任务。Redis 内存金贵，任务越小，入队越快、积压时也撑得住。

📦 S3 存本体

事件原文（input/output 可能很大）躺在 S3。它还是第 15 课「合并」要回读的历史事件源，更是任务万一丢失时的可重建依据。

入队时还藏着一个细节：queue.add(…, { delay }) 给任务挂了一小段延迟（默认约 5 秒）。为什么要故意「慢一拍」？因为同一实体的 create 和 update 可能先后脚到达，若 worker 抢在 update 落 S3 之前就处理 create，容易产生重复或乱序写入。延迟让一批相关事件先「落定」，worker 再统一取走，天然减少重复处理。 更妙的是 getDelay 在每天 UTC 的 23:45–00:15 这段「日界线」会把延迟拉长——因为第 8 课的排序键里有 toDate(start_time)，跨午夜的事件若处理太急， 可能被分到相邻两天的分区里造成重复。这个不起眼的延迟，正是为第 8 课那套按天组织的存储「保驾护航」，是工程上「用一点延迟换一致性」的典型权衡。

分片：把一条队列摊成 N 条，扛住洪峰

单条 Redis 队列总有上限。当部署开启 Redis 集群（REDIS_CLUSTER_ENABLED=true）时，摄取队列会分片成多条： ingestion-queue、ingestion-queue-1、ingestion-queue-2……数量由 LANGFUSE_INGESTION_QUEUE_SHARD_COUNT 决定。 分到哪条，取决于分片键 projectId-eventBodyId 的哈希：用 SHA-256 算出一个数，对分片数取模，得到 shard 下标。

分片但不打乱实体：哈希的是 projectId-eventBodyId，所以同一实体的所有事件（create/update）必然落到同一条 shard，既把负载摊到多个 Redis 节点，又保证一条记录的多次上报被同一个 worker 有序处理。源码：redis/ingestionQueue.ts:38-52、redis/sharding.ts:9。

注意分片键选 projectId-eventBodyId 而不是随机数，是有讲究的：随机分会把同一条记录的 create 和 update 打散到不同 shard， 合并时就得跨队列协调，既复杂又可能乱序。用实体 id 当分片键，「同一实体同一 shard」天然成立——这和第 8 课用 project_id 领头排序键、 第 13 课用 body.id 当合并键，是同一种「让相关数据物理上聚在一起」的思路。

所有项目共用一条主队列，会有个隐患：某个超高吞吐的项目瞬间灌进百万事件，把队列塞满，其他项目的事件只能干等。Langfuse 的解法是次队列隔离： worker 在处理每个任务前先判断——这个 project 该不该改道去次队列？判断有两条：

🎯 设计取舍
为什么非要队列 + S3 这一套，不能 API 直接写库？ 因为这套组合同时买到了解耦和持久两样东西。解耦：web 和 worker 各按自己的节奏跑， worker 挂了、ClickHouse 抖了，事件就在队列里排队、在 S3 里躺着，一条不丢，等下游恢复再慢慢消化（第 5 课最终一致）。持久：S3 是事件的「真账本」， 即便某个队列任务彻底失败，数据也能从 S3 重放重建。代价是多了一跳（写 S3 + 入队 + 读 S3）和最终一致的延迟，但换来的是一个削峰、容错、可重放的摄取管道—— 对一个每天吞百亿事件的系统，这笔买卖太划算了。
🎯 本课要点
  • 队列用 Redis + BullMQ，是 web（快路）与 worker（慢路）的交接点；任务载荷只放指针（fileKey 等），事件本体在 S3。
  • S3 是真账本：本体落 <eventId>.json，既供第 15 课合并回读，也让任务丢失时能重建——队列轻、内存省、可重放。
  • 分片：集群下队列拆成 ingestion-queue-N，按 projectId-eventBodyId 的 SHA-256 取模定 shard——同一实体永远同一 shard，有序又均衡。
  • 主 / 次队列：按 env 白名单（静态）或 S3 限流标志（动态）把高吞吐项目改道次队列，物理隔离，避免一个吵闹租户堵住所有人。
  • 韧性：任务 attempts: 6 + 指数退避；成功即删、失败保留。队列 + S3 = 削峰 + 容错 + 可重放。

IngestionService：合并的心脏

第 14 课 worker 取走任务、照 fileKey 把事件从 S3 读了回来。这一课是整条写入链路的心脏：那一堆事件，外加数据库里这条记录的当前状态， 究竟怎么被揉成一条最终记录？答案藏在 IngestionService 里——mergeAndWrite 按实体类型分流， mergeRecords 用一招「左折叠 + 后者覆盖」把多份输入合一，再盖上 event_ts=now 的版本戳。读懂它，第 8 课 ReplacingMergeTree 的「写入侧另一半」就补齐了。

📝 生活类比
把合并想成协作编辑一篇维基词条。数据库里当前发布的版本是底稿；这次收到的每个事件，就像一条按时间排好的修订， 依次盖在底稿上。后改的覆盖先改的（同一字段，新值赢）；但某条修订没碰的字段，保持原样（你只改了「结论」，「正文」不会被清空）。 还有几样东西被锁死，谁都不能改：词条的创建时间、唯一 id。所有修订叠完，得到一份合并稿，标上「本次合并时刻」当版本号—— 这正是 IngestionService 干的事：底稿来自 ClickHouse，修订来自 S3，叠出来的合并稿写回库。

mergeAndWrite 本身只是个调度员。它拿到第 13 课算出的实体类型（trace / observation / score / dataset_run_item）， 用一个 switch 把活儿分给对应的 process{Type}EventList。每条线路的骨架一样——读旧、转换、合并、写回——只是字段细节随类型而异。

📜 S3：本实体的全部事件

🗄️ ClickHouse：当前这条记录

同时去 observations 表 getClickhouseRecord 把已存在的那一行读出来当底稿。它排在合并数组的最前面，作为 immutable 字段的基线。

为什么两个来源都要？看似冗余，其实各有不可替代的作用。S3 那份是事件的真账本（第 14 课），重读全部事件能保证合并出的记录完整无遗漏， 即便这次任务是个迟到的补传；ClickHouse 那份则提供 immutable 字段的权威基线（created_at、start_time 这些「第一次写定就锁死」的值）， 还兜住那些可能已经从 S3 滚动清理掉的早期事件。两者叠在一起，既不丢新、又不改旧——这就是「读旧底稿 + 叠新修订」的精髓。

mergeRecords：左折叠，后者覆盖

核心算法只有几行：把「ClickHouse 底稿 + 按时间排序的事件记录」拼成一个数组，从左往右一个个 overwriteObject 叠上去，最后盖一个 event_ts = now。 关键全在 overwriteObject 的三条「不覆盖」规则里。

叠的是「有值的新字段」：后到的事件覆盖先到的，但三种情况例外——immutable 键、值为 undefined、空对象。所以 update 只带 endTime/output/usage，create 的 startTime/input/model 安然无恙。源码：IngestionService/index.ts:984-1005、utils.ts:56-81。

哪些字段被锁成 immutable？看 observations 这张表：[id, project_id, trace_id, start_time, created_at, environment]。 它们是记录的身份与锚点——一旦第一条事件写定，后续 update 再怎么改都动不了；其余 payload 字段（output、usage、end_time、level、metadata……）才走 last-write-wins。

合并过程中还藏着一个体贴的向后兼容细节：如果一个 observation 合并完发现没有 trace_id（很老的 SDK，< 2.0 时代埋点不带 trace）， IngestionService 会顺手造一条「包装 trace」——用 observation 自己的 id 当 trace id，补写进 traces 表，再把它挂上去。这样无论数据多老， 每个 observation 都不会变成「孤儿」，第 5 课那棵 trace 树永远完整。一个小小的补偿逻辑，体现的是这套系统对历史数据与多版本 SDK 的尊重与照顾。

第 7 步要特别留意：IngestionService 自己从不直接写 ClickHouse，它只把合并好的记录塞进 ClickhouseWriter 的队列。 为什么？因为逐条写 ClickHouse 是性能灾难（第 8 课说过它爱大批量）。把「合并」和「落盘」再切开，正是第 6 课「每层只干一件事」的又一次体现——合并归 IngestionService，攒批归 ClickhouseWriter。这种「先在内存里把记录算到位、再交给专门的批写器统一落盘」的两段式，让高频写入既保持正确，又不至于把列式存储压垮；第 17 课会看到批写器如何按表攒批、定时刷新。

🎯 设计取舍
既然第 8 课的 ReplacingMergeTree 自己就会按 event_ts 去重，为什么写入侧还要再合并一遍？ 因为两者解决的是不同时刻的问题。 写入侧的预合并保证：每次写出的那一行，当下就是正确的完整状态——查询立刻就能读到对的数据，不必等后台合并。而 ReplacingMergeTree 的后台去重是最终发生的， 用来收拾「同一条记录被并发处理、产生多行」这类竞态的尾巴。两层各管一段：写入侧求「即时正确」，存储侧求「最终唯一」。 再加上 event_ts=now 这个版本戳把两层咬合起来——预合并产出的新行永远有更大的 event_ts，于是它一定能在存储层的去重里胜出。这是典型的纵深防御。
🎯 本课要点
  • mergeAndWrite 是调度员：按实体类型 switch 到 process{Trace,Observation,Score,DatasetRunItem}EventList，骨架都是「读旧→转换→合并→写回」。
  • 合并有两个输入：S3 里本实体的全部事件（listFiles，按时间排序）+ ClickHouse 里当前那一行（当 immutable 基线）。
  • mergeRecords 把它们左折叠（overwriteObject 后者覆盖前者），并打 event_ts=now 版本戳。
  • 三条不覆盖规则：immutable 键（id/start_time/created_at…first-wins）、undefined、空对象——所以 update 只带变化字段、不抹掉 create 的值。
  • 合并完只 addToQueue，不直接写库（落盘归第 17 课）。写入侧预合并求「即时正确」，存储侧 ReplacingMergeTree 求「最终唯一」，靠 event_ts 咬合——纵深防御。
