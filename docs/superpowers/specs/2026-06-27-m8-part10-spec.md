## Part 10 · 平台与运维 / Platform & Operations (L48–L53)

- **L48 鉴权与会话 / Auth & sessions.** `web/src/server/auth.ts`, `web/src/features/auth*`,
  `web/src/features/auth-credentials/*`. Key theme: NextAuth with a credentials provider + ~12 SSO
  providers (Google/GitHub/Okta/AzureAD/Keycloak/WorkOS/Auth0…) + email; `session.strategy="jwt"`,
  `maxAge=AUTH_SESSION_MAX_AGE*60`; an extended PrismaAdapter; the **session callback** loads the
  user's `organizationMemberships → projects` + `ProjectMemberships` and injects them into the
  session — this is the bridge that feeds L49's RBAC. SSO-only enforcement (sign-in with
  email/password can be disabled per-instance and per-email-domain). Dynamic (DB-configured) SSO
  providers merged with static env-configured ones.

- **L49 RBAC · API key · SCIM / RBAC, API keys & SCIM.** `web/src/features/rbac/constants/
  {projectAccessRights,organizationAccessRights}.ts`, `web/src/features/rbac/utils/
  {checkProjectAccess,checkOrganizationAccess}.ts`, `packages/shared/src/server/auth/apiKeys.ts`,
  `web/src/pages/api/public/scim/*`. Key theme: **two auth axes** — humans authorize via
  **roles→scopes** (fine-grained `projectScopes` like `traces:delete`/`prompts:CUD`;
  `roleAccessRights` maps OWNER/ADMIN/MEMBER/VIEWER/NONE → scope lists; `throwIfNoProjectAccess`
  on the server + `useHasProjectAccess` hook on the client), machines authorize via **API keys**
  (`generateKeySet` → `pk-lf-…`/`sk-lf-…`; a **two-tier hash**: `fastHashedSecretKey` = sha256 for
  O(1) DB lookup + `hashedSecretKey` = bcrypt for secure verification + `displaySecretKey` for UI;
  `salt`=env.SALT). SCIM provisions enterprise users.

- **L50 开源核与权益 / Open-core & entitlements.** `web/src/features/entitlements/constants/
  entitlements.ts` (`entitlements` list, `entitlementLimits`, `entitlementAccess` Record keyed by
  plan), `web/src/features/entitlements/server/{getPlan,hasEntitlement,hasEntitlementLimit}.ts`,
  `packages/shared/src/server/ee/licenseCheck/index.ts` (`LANGFUSE_EE_LICENSE_KEY` must start with
  `langfuse_ee_`; keygen verification → org plan), `ee/src/*`. Key theme: the **open-core** model —
  a free OSS core plus commercial **entitlements** gated by an org's **plan** (cloud tiers or a
  self-hosted EE license). `hasEntitlement(org)` is how feature code asks "is this allowed here?";
  `hasEntitlementLimit` caps quantities. Dependency rule: `web → @langfuse/ee → @langfuse/shared`.

- **L51 自我可观测 · 配置 / Self-observability & config.** `packages/shared/src/server/
  instrumentation/index.ts` (`instrumentAsync`:54 — the OTel span wrapper already used by L30 eval,
  L33 monitors, L46 integrations; `getCurrentSpan`, `traceException`, `recordGauge/Increment/
  Histogram`, `getTracer`), `packages/shared/src/server/otel/*` (OtelIngestionProcessor — Langfuse
  also *ingests* OTel), `packages/shared/src/server/logger.ts` (winston structured logging,
  text/json format by env), `packages/shared/src/env.ts` + `web|worker env` (Zod-validated,
  type-safe config). Key theme: **Langfuse observes itself** — structured logs + OTel traces/metrics
  for its own services — and validates all config through Zod so a missing/bad env fails fast at
  boot, not at runtime.

- **L52 数据生命周期与删除 / Data lifecycle & deletion.** `worker/src/queues/dataRetentionQueue.ts`
  (`dataRetentionProcessor` scheduler + `dataRetentionProcessingProcessor` — the same **two-level
  fan-out** as L46), `worker/src/queues/{traceDelete,projectDelete,datasetDelete,scoreDelete}.ts`
  (`traceDeleteProcessor`; `projectDeleteProcessor` → `deleteTracesByProjectId` across ClickHouse),
  `packages/shared/.../backgroundMigrations/*`. Key theme: deletion/retention must span **Postgres +
  ClickHouse + blob storage** consistently; a retention policy fans out per-project; cascading
  project/trace deletion; background migrations evolve schema without downtime. Reuses fan-out (L46),
  streaming, and idempotency (L47).

- **L53 构建 · 测试 · 开发流 / Build, test & dev workflow.** `turbo.json` (`build` dependsOn
  `["db:generate","^build"]`, `outputs` dist/.next, `cache:true`; non-cached `dev`/`start`),
  root `package.json` (pnpm workspaces + scripts: dev/lint/typecheck/build/test/seed),
  `packages/shared/scripts/seeder/*` (seed CLI for local test data), vitest. Key theme: the
  **monorepo developer workflow** — turbo task pipeline + caching gives fast incremental
  builds across `web`/`worker`/`packages/shared`/`ee`; pnpm workspaces enforce the dependency
  direction; the seed CLI fills realistic data for local dev. A fitting close: how the platform
  you've studied is itself built and iterated.
