## Part 11 · 设计专题与终章 / Design Themes & Capstone (L54–L55)

- **L54 设计专题综合 / Design themes synthesis.** Cross-cutting; primary anchor
  `.agents/ARCHITECTURE_PRINCIPLES.md` (10 principles + practical defaults). Key theme: pull the
  recurring architectural decisions seen across all 53 lessons into **six named themes** and show
  each is one coherent idea applied repeatedly:
  1. **宽事件 / Wide events** (observability 2.0): observation as the primary analytical unit, a
     trace is a correlation handle; wide richly-attributed events over fragmented metrics/logs/traces;
     preserve high-cardinality context for unknown-unknowns (principles 1-3; L05/L06/L13/L41).
  2. **不可变 / Immutability**: favor append-oriented event records; updates force read-time dedup =
     hidden query cost; ClickHouse AggregatingMergeTree + final aggregation (principle 4; L13/L18).
  3. **异步 / Async**: the request path stays thin; heavy work goes to BullMQ queues + workers,
     fan-out per object, idempotency, watermarks, locks (L12 ingestion → S3+Redis+worker; L30 eval,
     L43 metering exactly-once, L46 fan-out, L52 background migrations).
  4. **双存储 / Dual storage** (really tri-store): Postgres (OLTP config/metadata) + ClickHouse
     (OLAP events) + S3/blob (bulky payloads); denormalize to kill hot-path joins; columnar access;
     compact representations for lists, raw payloads only for detail (principles 5-7; L02/L22/L24/L52).
  5. **多租户 / Multi-tenancy**: projectId scoping everywhere; auth (L48) → RBAC scopes + API-key
     scopes (L49) → entitlements by plan (L50); per-project fan-out and deletion (L46/L52).
  6. **成本 / Cost**: cost & operational simplicity as architectural constraints; extra
     DBs/queues/MVs must earn their burden; scale-aware API contracts (time windows, field selection,
     token pagination); model/usage cost tracking (principle 8-9; L27/L42/L43).
  Spark: why these are not six separate tricks but facets of one stance — "high-scale exploratory
  observability on wide structured events"; the discipline of letting each extra moving part earn
  its keep.

- **L55 终章 · 一条 trace 的完整一生 / Capstone: the full life of a trace.** End-to-end synthesis of
  L05–L53. Follow ONE trace through its entire lifecycle as a single narrative, naming the lesson at
  each stage:
  1. **Born** — SDK creates trace+observations, batches, POSTs to the ingestion API (L12).
  2. **Ingested** — API validates/auth (L49 API key), writes raw event to S3, enqueues to Redis;
     worker consumes, processes, upserts into ClickHouse (L13–L19); big payloads to S3.
  3. **Stored** — lands as events/observations in ClickHouse, metadata in Postgres, payloads in S3
     (dual/tri storage); immutable + merge-dedup (L13/L18).
  4. **Read** — appears in the trace list via compact CH queries (L20–L24), the detail view fetches
     the full tree + raw payloads (L25); sessions group it (L26); public REST exposes it (L27).
  5. **Evaluated** — an LLM-as-judge eval scores it (L28–L31), a monitor may alert (L33), it's added
     to a dataset / experiment (L34–L36).
  6. **Acted on** — an automation fires a webhook/Slack (L44–L45), it streams to analytics/blob
     (L46), or is batch-exported (L47).
  7. **Retired** — retention or manual/cascade deletion clears it across all three stores (L52).
  Throughout, it's observed by the platform's own OTel/logs (L51), gated by the org's plan (L50),
  scoped to its project (multi-tenancy). Spark + send-off: the whole system is one coherent answer to
  "how do you record, find, evaluate, and act on what your AI app did, at scale, affordably."
