## Verified source anchors (spot-check while authoring each lesson)

- **L28 评分模型**: `packages/shared/src/domain/score-configs.ts` — 3 `dataType`s: `NUMERIC`
  (maxValue/minValue, no categories, :19-22), `BOOLEAN` (exactly 2 categories, :27-33),
  `CATEGORICAL` (:78). `packages/shared/src/domain/scores.ts`: `ScoreSourceArray =
  ["API","EVAL","ANNOTATION"]` (:4), `ScoreDataType` incl. NUMERIC/CATEGORICAL/BOOLEAN (+ legacy
  CORRECTION/TEXT seen in L25). A score = (name, value/stringValue, dataType, source, comment) on a
  trace/observation; config validates value against its dataType.
- **L29 LLM-as-a-judge**: `web/src/features/evals/server/*` (evalTemplateCreation.ts,
  evaluatorRepository.ts, defaultEvalModelRouter.ts, evaluator-preflight.ts);
  `packages/shared/src/server/evals/extractObservationVariables.ts` (variable extraction — map trace
  fields into prompt variables). An **eval template** = prompt + output schema (score name/reasoning);
  an **evaluator** (JobConfiguration) binds a template to a target (which traces, sampling, variable
  mapping). The judge LLM reads the mapped variables, returns a score + reasoning.
- **L30 eval 执行流水线**: `worker/src/features/evaluation/evalService.ts` — `createEvalJobs` (:180)
  matches incoming traces against evaluator configs (filter + sampling via
  `InMemoryFilterService.evaluateFilter` :422), creates `JobExecution` rows (status lifecycle
  `JobExecutionStatus`); `worker/src/queues/evalQueue.ts` runs them; `executeLLMAsJudgeEvaluation`.
  Result → a score (source=EVAL) written back through the ingestion path.
- **L31 代码 eval**: `packages/shared/src/server/evals/codeEvalExecution.ts`;
  `web/src/features/evals/server/codeEvalJobConfigValidation.ts`, `codeEvalTestRun.ts`;
  `worker/src/queues/codeEvalQueue.ts`. Code evals run user JS either on **AWS Lambda** or a **local
  dispatcher** (sandboxed); deterministic checks (regex, JSON match, etc.) → score.
- **L32 人工标注**: `web/src/features/annotation-queues/*` (queues to route items to human
  reviewers), `web/src/features/comments/*` (threaded comments on traces/observations),
  corrections. Annotation scores have source=ANNOTATION; an annotation queue assigns items to
  reviewers who score against score-configs.
- **L33 监控器与告警**: `web/src/features/monitors/*`, `worker/src/queues/monitorQueue.ts`. A
  monitor watches a metric/score over a window and triggers an alert/automation when a threshold is
  crossed.

## Recurring themes to thread
- (a) one `score` unit, three sources (API/EVAL/ANNOTATION) — same storage (L08), different origin.
- (b) eval = match (filter+sample) → execute (LLM/code/human) → write back a score via ingestion.
- (c) reuse: eval results re-enter through the SAME ingestion path (M3) and become queryable scores
  (M4) — evaluation is not a separate pipeline but a producer of scores.
