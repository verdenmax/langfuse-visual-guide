## L02 · 可观测性 2.0 与宽事件 / Observability 2.0 & wide events
**Goal:** Establish the design philosophy that the whole platform is built on, so later lessons can
reference it.

**Verified source anchors:**
- `.agents/ARCHITECTURE_PRINCIPLES.md` — the canonical principles. Key, quotable principles:
  - "Model observations as the primary analytical unit. A trace is a correlation handle…"
  - "Prefer wide, richly attributed events over fragmented metrics, logs, and trace records…"
  - "Preserve high-cardinality context so users can slice, group, filter, and debug unknown unknowns…"
  - "Favor immutable or append-oriented event records for high-volume telemetry."
  - "Denormalize carefully when it removes hot-path joins…"
  - "Design storage and query paths around columnar access patterns…"
  - "Keep list/dashboard/aggregate views on compact query-optimized representations. Fetch large raw
    payloads only for focused detail views."
  - "Make API contracts scale-aware: require time windows…token pagination…"
  - "Treat cost and operational simplicity as architectural constraints."
  - Practical defaults section ("Before adding a metric, ask whether…wide event data").
- Reference posts (external, cite as further reading, do NOT invent their content):
  "Simplifying Langfuse for Scale" (langfuse.com/blog), Charity Majors "Observability 2.0",
  "All you need is Wide Events" (isburmistrov substack) — all listed in ARCHITECTURE_PRINCIPLES.md.

**Key claims to make (all grounded):** metrics/logs/traces ("three pillars 1.0") vs wide events
(2.0); the cost is bigger rows, paid back by columnar+compression+deferred large-field reads; this
principle explains *why* ClickHouse, *why* immutable events, *why* compact list models — forward-ref
to L08/L09/L24.

**Planned figures (≥2 per language):**
1. metrics+logs+traces (fragmented, reassemble later) → vs → one wide event row (slice any way).
2. the "unknown unknowns" idea: predefined dashboards answer only known questions; a wide table lets

Plus HTML blocks: a `.cols` (1.0 vs 2.0), a `table.t` (principle → what it buys), a `.card.spark`
(the central tradeoff), a `.vflow` of the 10 principles grouped.

## L03 · 三大支柱深入：trace / observation / score / The three pillars in depth
**Goal:** Go deeper than L01 into the actual domain shapes and how they map to ClickHouse + sessions.

**Verified source anchors:**
- `packages/shared/src/domain/traces.ts` — `TraceDomain` fields: id, name, timestamp, environment,
  tags[], bookmarked, public, release, version, input, output, metadata, createdAt, updatedAt,
  sessionId, userId, projectId. `MetadataDomain` = record(string, json|undefined).
- `packages/shared/src/domain/observations.ts` — `ObservationType` (10: SPAN, EVENT, GENERATION,
  AGENT, TOOL, CHAIN, RETRIEVER, EVALUATOR, EMBEDDING, GUARDRAIL); `ObservationLevel`
  (DEBUG/DEFAULT/WARNING/ERROR); `ObservationSchema` rich fields: parentObservationId, model,
  internalModelId, modelParameters, input, output, completionStartTime, promptId/promptName/
  promptVersion, latency, timeToFirstToken, providedUsageDetails/usageDetails/costDetails/
  providedCostDetails, inputCost/outputCost/totalCost, inputUsage/outputUsage/totalUsage,
  usagePricingTierId/Name, toolDefinitions/toolCalls/toolCallNames.
- `packages/shared/src/domain/scores.ts` — `ScoreSourceArray` = [API, EVAL, ANNOTATION];
  `ScoreDataTypeArray` = [NUMERIC, CATEGORICAL, BOOLEAN, CORRECTION, TEXT]; annotation scores need a
  matching `configId` (except CORRECTION); `TEXT_SCORE_MAX_LENGTH=500`.
- ClickHouse tables: `clickhouse/migrations/unclustered/0001_traces.up.sql`,
  `0002_observations.up.sql`, `0003_scores.up.sql` (cite the column shapes; parentObservationId,
  Map columns, ReplacingMergeTree).
- Sessions: `TraceSession` model in `prisma/schema.prisma`; `sessionId` on the trace links traces
  into a session (forward-ref L26).

**Key claims:** trace = thin correlation handle (few fields, mostly identity+tags); observation =
the rich unit (parentObservationId builds the tree; usage/cost/prompt inlined); score = the verdict
(source × dataType matrix). The three map 1:1 to three ClickHouse tables.
