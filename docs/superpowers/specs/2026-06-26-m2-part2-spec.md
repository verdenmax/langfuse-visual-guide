## L08 ┬и ClickHouse СИјт«йС║ІС╗ХУАе / ClickHouse & the wide-event tables
**Goal:** The heart of storage Рђћ how the three wide-event tables are physically built and why.

**Verified anchors:**
- `packages/shared/clickhouse/migrations/unclustered/0001_traces.up.sql`,
  `0002_observations.up.sql`, `0003_scores.up.sql`: `ENGINE = ReplacingMergeTree(event_ts, is_deleted)`,
  `PARTITION BY toYYYYMM(timestamp/start_time)`, `ORDER BY (project_id, [type], toDate(time), id)`,
  bloom_filter indexes (idx_id, idx_trace_id, Рђд), `CODEC(ZSTD(3))` on input/output, `Map(...)` columns
  for metadata/usage_details/cost_details.
- `environment` column added later: `0008_add_environments_column.up.sql`.
- ClickHouse query/mapping: `src/server/clickhouse/schema.ts`, `schemaUtils.ts` (event Рєњ table mapping).
- Why columnar / MergeTree: ties to ARCHITECTURE_PRINCIPLES (columnar access, immutable/append).

**Key claims:** ReplacingMergeTree = "last write per (sort key) wins" via `event_ts`, enabling
idempotent upserts + soft delete via `is_deleted` (no read-time dedup needed if you query the final
state); monthly partitions enable pruning; ordering key (project_id first) makes per-project +
time-window scans cheap; ZSTD shrinks big text; bloom filters speed id/trace lookups.

**Figures:** (1) one observations row anatomy (columns + Map fields + codecs); (2) ReplacingMergeTree
behavior (two writes same id Рєњ later event_ts wins); (3) ordering-key/partition Рєњ pruned scan. Blocks:
`table.t` (column Рєњ type Рєњ why), `.cellgroup` for the row, `.timeline` for replacement.
