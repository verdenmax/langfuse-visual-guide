## Part 6 · 数据集与实验 / Datasets & Experiments (L34–L36)

- **L34 数据集与数据项 / Datasets & items.** A dataset = a curated collection of test cases.
  `schema.prisma:661` Dataset (name, description, metadata, inputSchema/expectedOutputSchema Json,
  remoteExperiment*); `:685` DatasetItem (input, expectedOutput, metadata, **sourceTraceId /
  sourceObservationId** — promote a real production trace into a test case; DatasetStatus
  ACTIVE/ARCHIVED). Key theme: **items are versioned** — `@@id([id, projectId, validFrom])` with
  validFrom/validTo/isDeleted (temporal versioning); this is exactly the `datasetItemValidFrom` a
  dataset eval pins to (L30). `web/src/features/datasets/*`.

- **L35 dataset run 与 run item / Dataset runs & run items.** `schema.prisma:719` DatasetRuns
  (a run = one execution of the whole dataset, unique [datasetId, projectId, name]); `:739`
  DatasetRunItems links (datasetRunId + datasetItemId) → a **traceId** (+ observationId): each
  (run × item) produces one trace. Key theme: run items are **mirrored to ClickHouse**
  (`dataset_run_items` table, migration `clustered/0022`+`0024`+`0032`; repo
  `shared/src/server/repositories/dataset-run-items*`) for fast run-level aggregation (avg score
  per run). Ties back: running an item triggers the L30 eval pipeline via dataset-run-item-upsert.

- **L36 实验与对比 / Experiments & comparison.** An experiment runs a dataset against an
  app/prompt/model config, producing a run; you then **compare runs** side by side (score deltas).
  `web/src/features/experiments/*` (CreateExperimentsForm, ExperimentComparison*, baseline
  controls); `worker/src/queues/experimentQueue.ts` + `worker/src/features/experiments/*`
  (experimentServiceClickhouse.ts, scheduleExperimentEvals.ts). Key theme: experiments tie
  datasets (test cases) + runs (executions) + scores (L28–32) + eval scheduling into a
  regression-testing loop for prompt/model changes.

## Part 7 · Prompt 管理与 Playground / Prompts & Playground (L37–L39)

- **L37 Prompt 管理（版本·label·依赖）/ Prompt management.** `schema.prisma:835` Prompt
  ((name, version) unique; type text/chat; config Json; **labels** String[] like production;
  **commitMessage** — git-like); `:864` PromptDependency (parent Prompt depends on a child by
  childName + childLabel OR childVersion — **prompt composition / dependency graph**).
  Key theme: **versions are immutable commits, labels are movable pointers** (production →
  version N), commitMessage = commit message — prompts as version-controlled artifacts.
  `web/src/features/prompts/*`, `shared/src/domain/prompts.ts`.

- **L38 Prompt 服务与缓存 / Prompt serving & caching.** Production reads a prompt by name +
  label/version on (potentially) every LLM call, so serving must be fast.
  `web/src/pages/api/public/prompts.ts` → `web/src/features/prompts/server/prompt-api-service.ts`
  → `packages/shared/src/server/services/PromptService/*` (cache). Key theme: a read-through cache
  (resolve by label → version), cache invalidation on new version/label move, and the
  composition resolution (a prompt's dependencies get inlined).

- **L39 Playground 与 LLM 连接（加密·schema·tool）/ Playground & LLM connections.** The
  playground is an interactive prompt tester; it needs provider credentials, structured-output
  schemas, and tool defs. `web/src/features/playground/*`; `web/src/features/llm-api-key/*`
  (**encrypted** key storage — `packages/shared/src/encryption/encryption.ts`, AES);
  `web/src/features/llm-schemas/*` + `web/src/features/llm-tools/*` (structured output + tools);
  `shared/src/server/llm/*` (fetchLLMCompletion). Key theme: **secrets are encrypted at rest**,
  and the playground reuses the same LLM-call machinery as evals (L29) and production.
