What is Langfuse

The hardest thing about LLM apps is non-determinism: the same input can give different answers today and tomorrow; and inside one multi-step agent run, figuring out which step went wrong, how many tokens it burned, how much it cost, or where the latency went is nearly impossible from a few log lines. Langfuse is an open-source LLM engineering platform (langfuse/langfuse, MIT-licensed). In one line: it turns every invisible step inside your LLM app into observable, evaluable, reproducible data — around one loop: develop → monitor → evaluate → debug. Put plainly, it answers the three questions engineers ask daily but struggle to answer: what actually happened this time? was it any good? how do I make it better?

🔌 Analogy
Think of an LLM app as an airplane: it flies fine until something goes wrong — and then you're blind. Langfuse bolts on two things: a black box (flight recorder) and a lab notebook. The black box records every model call, every retrieval step, the prompt used, the tokens spent, the money burned and the latency of each leg; the lab notebook lets you line up every “try another prompt / model / parameter” experiment side by side. When things break you can replay; when you want to improve you have evidence — that's what Langfuse gives an LLM app. In short, the black box is for "seeing" and the lab notebook is for "comparing"; together they turn "tuning by vibes" into "improving with data".
One loop: your app ships run-time events through the SDK into Langfuse; the platform distills them into four capabilities — observe / evaluate / prompts / datasets; engineers find problems, improve the app, and push improvements back to production. Langfuse always observes from beside the app, never intruding on business logic.

What it actually is

A one-line definition: a platform that turns an LLM app's run-time behavior into data. It does not train models for you, and it is not yet another framework for calling an LLM; its work happens outside your app — faithfully recording and measuring every run, and supporting you to evaluate and improve it. The official capabilities map to one folder each under web/src/features/:

It helps to say what it is not: it is not a framework like LangChain that helps you write an LLM app — use whatever framework you like; Langfuse only records what happens at run time. It is also not a model server or gateway that calls models for you (though the Playground lets you experiment). Its place is the side-channel observer: your app runs as usual and simply "reports" each step to Langfuse.

🗺️ Big picture
Langfuse is both a SaaS cloud (cloud.langfuse.com) and fully self-hostable — because it's open source and depends only on commodity infra: Postgres, ClickHouse, Redis, S3. This guide explains the self-host codebase: what you see in langfuse/langfuse is the same stack that runs in the cloud. Understand this code and you can both use it well and change it / send a PR.

The three pillars: trace / observation / score

Every piece of "observable" data in Langfuse boils down to three things. Their domain models live under packages/shared/src/domain/ (traces.ts / observations.ts / scores.ts); their analytical storage is three same-named ClickHouse tables (packages/shared/clickhouse/migrations/unclustered/0001_traces, 0002_observations, 0003_scores):

Containment: a trace wraps all steps of one interaction; each step is an observation (which can nest); a score rates the trace or one observation. Note the architectural stance — the observation is the primary analytical unit; the trace is more of a "correlation handle" stringing them together.

🧵 trace · correlation handle

Represents one complete request/session. It carries little itself (name, user, session, tags, time); its main job is to correlate all the observations of one interaction so you can follow the thread. See domain/traces.ts.

⭐ observation · the star

One concrete step. The three core types are GENERATION (an LLM call, with model/token/cost), SPAN (a timed unit of logic, e.g. retrieval) and EVENT (an instantaneous event); the current domain model further specializes these into AGENT / TOOL / CHAIN / RETRIEVER and more. It carries rich attributes — input/output, timing, usage, cost — and is the subject of querying and aggregation.

And the third pillar, score? It is a rating attached to a trace or an observation. Its value is mainly one of three types — NUMERIC (e.g. 0.9), CATEGORICAL (e.g. "correct / wrong"), BOOLEAN (the domain model also has CORRECTION and TEXT); from three sources — EVAL (auto-judge), ANNOTATION (human), API (submitted via SDK), see packages/shared/src/domain/scores.ts. Score is what upgrades "observable" into "evaluable": with comparable numbers you can quantify "did this change actually make the app better" instead of guessing. A score's config (numeric range, categories) lives in Postgres as ScoreConfig, while the scores themselves go into ClickHouse like traces/observations.

🔬 Source detail
The observation type is an enum in the domain layer: the original LegacyPrismaObservationType had only SPAN / EVENT / GENERATION (see packages/shared/prisma/schema.prisma), while the current domain model packages/shared/src/domain/observations.ts defines an ObservationType with 10 values (adding AGENT / TOOL / CHAIN / RETRIEVER / EMBEDDING / EVALUATOR / GUARDRAIL). The ClickHouse observations table inlines model, provided_usage_details, cost_details, prompt_id and more as columns and Maps on a single row — exactly what a "wide event" looks like (next section).

You can only fix what you can see: a debugging scene

An abstract definition is weaker than a concrete scene. Say you built a customer-support bot and users report "it sometimes answers off-topic". Without Langfuse you only have a few app-log lines like prompt=... response=... — you can't tell which step failed, and you can't reproduce it. With Langfuse, triage looks like this:

① In the trace list, filter by "user thumbs-down" or "low score" and surface the offending traces in seconds; ② open one, and the observation tree makes it obvious — it was the retrieve-docs step that fetched irrelevant documents, not the model itself; ③ inspect that generation's input and confirm the polluted context is exactly those wrong docs; ④ add this trace to a dataset, fix the retrieval logic, run an experiment comparing old vs new, and use a score to confirm it's truly fixed. The whole flow rests on "every step was recorded as-is".

⚠️ The trap without observability
Staring only at "the model output is bad" makes it easy to blame the model wrongly. In the real world a huge share of "the LLM answered poorly" is actually bad retrieval, a mis-assembled prompt, or an upstream step that timed out and returned empty. You only see these by laying out the whole chain — which is why Langfuse centers on the observation (the "step"), not just the final answer.

What it's made of

Why pull the core into packages/shared and have both web and worker depend on it one-way only? Because web and worker are two very different workloads: web faces users, wants low latency and must restart anytime; the worker chews through heavy jobs, must be retryable and scales by queue. Putting the shared domain models, storage access and queue contracts into a narrow waist that nothing depends on backwards lets the two sides evolve, deploy and scale independently while sharing one contract that can't drift. That one-way dependency discipline (web/worker/ee → shared, and shared depends on none) is key to keeping the whole monorepo maintainable.

Why this design: wide events / Observability 2.0

Langfuse's storage and query paths are all threaded by one design principle — treat the observation as the primary analytical unit and record everything as "wide events" (stated explicitly in .agents/ARCHITECTURE_PRINCIPLES.md). A wide event means you do not shatter information into separate metrics, logs and traces to be reassembled later; instead you cram all the high-cardinality context of one step (model, usage, cost, prompt, user, tags, environment…) onto a single row.

Contrast the classic "three pillars of observability": metrics are pre-aggregated numbers (QPS, average latency) — cheap but they lose detail, so drilling into "this user + this model + this time window" later is hopeless; logs are scattered text lines — detail is there but they're hard to aggregate and correlate; traces show call structure but usually carry no business attributes. The wide-event idea fuses all three: each observation is at once a structured "trace node", a bag of aggregatable numbers (token/cost/latency), and a carrier of arbitrary business fields (user/tag/metadata). So "drill down" and "aggregate" stop being two systems and become different queries over one wide table. That's also the root reason Langfuse picked ClickHouse — columnar storage is built for "time-bounded scans + arbitrary-dimension aggregation over a wide table".

🎯 Design tradeoff
Why keep each row "wide" instead of splitting it into tidy normalized tables? Because the essence of observability is answering questions you didn't think of at the time (unknown unknowns). If you split fields into fixed metrics before shipping, you can only query the dimensions you predefined; keep the high-cardinality context on the row and you can slice, group and filter arbitrarily after the fact. The cost is bigger rows, more storage and smarter writes — Langfuse balances it with columnar storage + ZSTD compression + reading big fields only on the detail page (later lessons). Trade a little storage for unlimited freedom to ask questions.

Observability 2.0 & wide events

Last lesson called Langfuse's "design soul" wide events / Observability 2.0. This lesson makes the slogan concrete — it is not marketing; it's a first principle written into .agents/ARCHITECTURE_PRINCIPLES.md that constrains every storage and query decision in Langfuse. Grasp it and the later questions — "why ClickHouse", "why immutable events", "why lists read only narrow fields" — answer themselves. This lesson is the guide's foundation lesson: not about one module but about a way of seeing that the fifty-odd later lessons build on.

🔌 Analogy
Classic "Observability 1.0" is like a hospital splitting your check-up across three disconnected departments: one records only numbers (blood pressure, heart rate — metrics), one writes only text (the doctor's notes — logs), one draws only the route (register, then blood draw — traces). To understand "why I feel unwell" you must reconcile across all three yourself. "Observability 2.0" flips it: each visit writes one complete medical record — numbers, description and process on a single sheet. That single sheet is the wide event. The benefit is obvious: a different doctor, or you revisiting later, can reconstruct the whole picture from that one sheet — no re-digging three departments to line up the timeline.
1.0 vs 2.0: the classic three pillars fragment information, forcing manual reassembly before any cross-dimension question; a wide event puts the numbers, structure and business fields of one step on a single row, so "drill down" and "aggregate" are just different queries over one wide table. This is exactly what ARCHITECTURE_PRINCIPLES.md means by "prefer wide, richly attributed events over fragmented metrics, logs, and trace records that require later reconstruction".

The trouble with Observability 1.0

The classic "metrics + logs + traces" combo (often called Observability 1.0) is fine piece by piece; the problem is that the three are separate:

  • Metrics are pre-aggregated numbers — "average latency", "requests per minute". Cheap, but you must decide before shipping which dimensions to aggregate; ask later for "the latency of this user on this model in this window" and without that pre-aggregation you're stuck.
  • Logs are text lines — all the detail is there but they're hard to aggregate and correlate: computing "median tokens grouped by model" over free text is painful.
  • Traces show call structure (who called whom, how long) but usually carry no business attributes — they know "an LLM was called" but not "who the user was or which prompt version".

So to triage a cross-dimension problem you end up reconciling across three systems. Langfuse's very first architecture principle targets exactly this — treat the observation as the primary analytical unit, with the trace as a correlation handle (verbatim: "Model observations as the primary analytical unit. A trace is a correlation handle that links related observations, not the only useful entry point.").

A concrete contrast. Someone complains that "answers get slow at peak hours" and you want to know "is a particular model slow for particular users". In 1.0: metrics give you one "global average latency" curve that can't tell which model or which users; logs have the individual records but tens of thousands of text lines won't quickly group-by-model and compute percentiles; traces show per-call duration but never recorded the user and model business fields. All three are "related", yet none can answer directly — you end up exporting data and scripting the join yourself. In 2.0: every observation already carries model, userId and latency, so the question collapses to one ordinary query — "group by model + userId, look at p95 of latency". Same question: 1.0 stitches across three systems for ages; 2.0 just changes the query.

Wide events: put everything on one row

The "Observability 2.0" fix is direct: don't reassemble later — don't split it when you write. Cram all the high-cardinality context of one step (which model, how many tokens, how much money, how long, which user, which tags, what environment…) into one record — that record is a Langfuse observation. It's both a "trace node" (with parentObservationId to rebuild the call tree), a bag of aggregatable numbers (usageDetails / costDetails / latency), and a carrier of arbitrary business fields (metadata / tags / userId). Everything the three pillars do separately, one wide event does together.

You might ask: why not split these fields into several normalized tables with foreign keys — isn't that "cleaner"? Because JOINs at query time are a hidden hot-path cost. Observability data is enormous (an active project can write hundreds of millions of observations a day); if every "cost by model" had to JOIN the observation table with a "models" table and a "usage" table, the scan-and-join cost explodes with volume. Langfuse's principle is to denormalize carefully: inline the fields commonly used to filter and aggregate directly onto the observation row, so "by model / by user / by time" become plain column predicates instead of JOINs. That's the other meaning of "wide" — not just many fields, but deliberately keeping together what belongs together, trading a little storage redundancy for not joining at query time.

😕 1.0: split then stitch

Metrics, logs and traces live apart; a cross-dimension question means three queries plus manual reconciliation. What you can ask is locked by field design before you ship.

😀 2.0: one row holds it

One observation carries numbers, structure and business fields at once. Slice, group and filter are all queries over one wide table; ask anything later. Cost: wider rows, more storage.

Why it's worth it: unknown unknowns

The biggest payoff of wide events is answering questions you didn't think to ask up front. Pre-aggregated metrics answer only known questions (you see only the dimensions of dashboards you built); keep the high-cardinality context on the row and a new question is just a new query — no re-instrumenting, no waiting for the next deploy.

This matters especially for a platform. Langfuse serves thousands of projects and cannot know in advance which dimensions each team cares about — one wants "cost by prompt version", another "total latency of a user's multi-turn session", another to slice by custom metadata tags. The pre-aggregation route would force the platform to predefine a metric per need — never-ending and rigid. Wide events let Langfuse demote "define a new view" from "change instrumentation + redeploy" to "write a new query": dashboards, custom metrics and filters (lessons 23, 40) are all just different queries over the same wide table. That's why "preserve high-cardinality context" isn't a nicety — it's the foundation of the platform's extensibility.

Known vs unknown: pre-aggregation fixes "what you can ask" at ship time; wide events keep the high-cardinality context on the row, so facing unknown unknowns — the questions you never thought to ask — you just write a new query. That's the root reason Langfuse keeps rows wide.

The three pillars in depth

Lesson 1 introduced the names trace / observation / score; this lesson opens them up: exactly which fields each has, why it's designed that way, and how each lands in one of three ClickHouse tables. Their "true form" is three Zod schemas under packages/shared/src/domain/ — read these three schemas and you hold the key to every later Langfuse feature, because evaluation, datasets, dashboards… all ultimately operate on these three objects.

🔌 Analogy
Picture one full user interaction as a hospital chart folder (trace): the cover has just a few identity lines (who, when, which tags). The pages inside are the point (observations): this page is "ran a CT scan", that one "prescribed a drug", another "the doctor reached a diagnosis" — each page detailing what was used, how long it took, how much it cost. Finally, red-pen margin notes (score) rate a page or the whole folder: "is this diagnosis accurate? 9/10". Thin cover, thick pages, pinpoint notes — exactly the division of the three pillars. Hold this "folder — pages — notes" image and you won't get lost in the fields below: whatever is "whole-interaction level" goes on the cover (trace), whatever is "single-step level" goes on a page (observation), and whatever is "judgment level" is a margin note (score).
The tree is "pointed" into existence: the observations table has no "children" field; each observation stores one parentObservationId pointing at its parent, and the UI reconstructs the call tree from that (L25). The root's parent is null. Storing only a parent pointer lets observations be independent and arrive out of order at write time — ideal for async ingestion.

In short, the trace is thin, but every field is hand-picked for "easy to find, share and categorize later".

observation: the real star

The observation is the "heavy" one. Types first — many assume there are only 3, but the current ObservationType has 10:

Before the fields, grasp one key design behind that tree figure: the parent-child relationship among observations is expressed only by a parentObservationId pointer on the child — the parent has no idea which children it has. Simple as it looks, this is tailor-made for async ingestion: because each observation carries "who my parent is", they can arrive independently and out of order at the server — a child arriving before its parent is fine, since the UI rebuilds the tree from the pointers at query time (L25). Flip it — make the parent maintain a children list — and you'd have to "wait for all children before writing the parent", a disaster in a high-concurrency async pipeline. One small choice of pointer direction underwrites the whole ingestion path's feasibility.

One easily-confused detail that shows real design care: usage and cost each have two sets of fields — providedUsageDetails / providedCostDetails are what the SDK "brought" at report time (if your app knows the token count and price, it can send them directly), while usageDetails / costDetails are what Langfuse "computed" itself (at ingestion, from the model pricing table, L16). This "provided wins, computed backfills" pattern recurs in the cost lesson.

Two more humble-but-invaluable fields for debugging: level (DEBUG / DEFAULT / WARNING / ERROR) and statusMessage. The former tags each observation with a severity; the latter records a status line (e.g. an error message). With them you can filter tens of thousands of observations down to those with level = ERROR in one click — jumping straight to "which step failed and what it said" instead of scanning the whole tree. Again the value of "keep high-cardinality context on each step": even "did this step error" is a queryable, aggregatable column.

score: the verdict

The third pillar, score, upgrades "observable" into "evaluable". Its two core dimensions are source and dataType, both defined in packages/shared/src/domain/scores.ts:

Score's two ends: on the left, "who scored" — EVAL (auto-judge, L29–31), ANNOTATION (human, L32), API (SDK); on the right, "scored what" — attaches to a trace (whole interaction) or an observation (one step). The score itself carries a value and a dataType.

Why do the three pillars land in three separate ClickHouse tables rather than one big table? Because their access patterns differ: observations are the largest, most-queried, widest; scores are few and usually queried by trace/observation association; traces sit in the middle as the list-page entry point. Separate tables let each use the ordering key and partitioning best for itself (L08) without dragging on the others. This is again "choose storage by access pattern" — the same idea as "why Postgres + ClickHouse coexist" (L07), just at finer granularity.

There's an easily-missed "fourth" aggregation: the session. It isn't a wide-event table; it's the trace's sessionId stringing multiple traces into one continuous conversation (e.g. ten back-and-forth turns with your bot). Session metadata lives in Postgres' TraceSession model; the aggregate view is computed at query time (L26). So strictly, Langfuse's core data is "three pillars + session aggregation".

A session example: a user asks a support bot three turns in a row — "where's my order?" → "can I change the address?" → "change it to my office address". Those three turns are each an independent trace (one full Q&A each), but they carry the same sessionId. So in the "session" view you can see the three traces as one continuous conversation: how long the whole chat took, how much it cost in total, at which turn the user got impatient. This "single trace vs continuous session" dual view works precisely because the one sessionId field plants the relationship at write time, so queries need no complex session segmentation.

Put the pillars together and a clean division of labor appears: trace correlates, observation records, score judges, session links. Each does its job, no field is piled up redundantly, yet all can be joined in one query. Whether later lessons cover ingestion, querying, evaluation or dashboards, they all operate on these four objects — carve their shapes into your mind now and the rest gets much easier.

🎯 Design tradeoff
Why pile the "heavy" fields on the observation and keep the trace so thin? Because the vast majority of queries happen at the "step" granularity: you aggregate and filter by model, by prompt version, by tool call — dimensions that naturally belong to one step, not the whole interaction. Put them on the observation and common queries become direct column predicates over the observations wide table; the trace just keeps correlation and a few search markers. The cost is that some info (e.g. userId) may be redundant across trace and observation — exactly Lesson 2's "denormalize carefully". Read the trade the other way: Langfuse chooses to pay a little duplication so that the queries users run all day stay JOIN-free and fast. At observability scale, that trade almost always wins.
🎯 Key points
  • trace is thin: mostly identity and markers (userId/sessionId/tags/bookmarked/public); its job is to correlate.
  • observation is heavy: ObservationType has 10 values (core three SPAN/EVENT/GENERATION), builds a tree via parentObservationId, inlines model/usage/cost/prompt/tool rich attributes.
  • score has two dimensions: source (API/EVAL/ANNOTATION) × dataType (NUMERIC/CATEGORICAL/BOOLEAN/CORRECTION/TEXT); attaches to a trace or observation.
  • The three map 1:1 to three ClickHouse wide-event tables; a session strings traces via sessionId.
  • Read these three domain/*.ts schemas and you hold the key to every later feature.

Project map (monorepo & narrow waist)

The first three lessons covered "what the data looks like"; this one switches axes: where the code lives. Langfuse is a monorepo (one repo, many projects) — everything sits in the single langfuse/langfuse repo, split internally into a few purpose-built "workspaces". Hold this map in mind and whenever a later lesson says "open web/src/..." or "this logic is in worker", you'll instantly know where it is and why.

🔌 Analogy
Think of the repo as one big shopping mall: the whole building is one monorepo housing a few independent shops — web is the customer-facing storefront (busy, fast, always open), worker is the back-of-house workshop (heads-down heavy lifting, can queue and take its time). And packages/shared is the central warehouse: every shop draws its goods (domain models, DB access, queue contracts) from here, but the warehouse never depends back on any shop. All under one roof, sharing utilities (build tools), yet cleanly bounded and independently run. Grab this "floor directory" before you wander the mall and you won't get lost no matter which shop you visit.
Just a few top-level blocks: web (storefront) and worker (workshop) are two runtime containers, both depending up on packages/shared (central warehouse / narrow waist); ee holds enterprise features used by web; fern defines the public API contract and generates the generated clients. Remember this and every "where is file X" question has a coordinate system.

Unpacking what each runtime actually does makes it tangible. web is the Next.js app you open in your browser; it does three things: render the UI, serve that UI a type-safe internal API (tRPC, L21), and serve external SDKs a public REST API (web/src/pages/api/public, L27) — the ingestion entry point lives here. worker is a headless background process that subscribes to dozens of BullMQ queues and takes on all the heavy, slow, retryable work: merge ingested events into ClickHouse, run LLM evals, export data, delete projects… (covered from Part 3 on). In a line: web does "fast in, fast out", worker does "heads-down grind", decoupled by a Redis queue in the middle — exactly the through-line of the next lesson ("dual store") and Part 3 ("the ingestion path").

One repo, four workspaces

Langfuse uses pnpm workspaces to slice one repo into independently buildable packages. The root pnpm-workspace.yaml lists them line by line:

  - "worker"       # queue consumer, background processing
  - "packages/**"  # shared packages, mainly packages/shared

Under packages/** there's more than just shared — also "tooling config" packages like config-eslint, config-typescript, eslint-plugin; but the real business shared core is packages/shared — the central warehouse in the figure. Why split into workspaces instead of one big folder? Because each package can then declare deps, build and test independently, and pnpm symlinks them so web can simply import ... from "@langfuse/shared" against the latest code with no publishing step. This hugely helps the dev experience: change a type in packages/shared and the editors in web and worker light up red immediately showing what must follow — surfacing the impact of a "contract change" at coding time rather than at runtime.

Dependency direction: why a "narrow waist"

The four workspaces don't reference each other freely — there's a strictly one-way dependency rule (written in .agents/AGENTS.md):

.agents/AGENTS.mdDependency direction
- web    -> @langfuse/shared, @langfuse/ee
- worker -> @langfuse/shared
- ee     -> @langfuse/shared
- @langfuse/shared -> no imports from web, worker, or ee
The "narrow waist": collect what everyone needs into a thin layer that nothing depends on backwards. web/worker/ee are the top of an hourglass, shared is the thin "waist", and below is the database. The payoff: the two containers evolve and deploy independently while sharing one set of domain models and queue contracts, so the contract can't drift.

Why does this discipline matter? Because web and worker are two very different workloads: web faces users, wants low latency, must restart anytime; worker chews heavy jobs, must be retryable, scales by queue. Let them import each other directly and changing one can entangle the other, binding their deployments. Collecting shared things into shared and allowing only downward dependencies erects a clean contract wall between the two containers: what goes through queues, what the DB tables look like — all defined in one place (queue payloads, for instance, belong to packages/shared/src/server/queues.ts). Both sides work to the same contract without stepping on each other.

A concrete bug this rule blocks: imagine no narrow waist, and worker lazily imports a utility function from web. One day web refactors that function to change the UI and tweaks its parameters — and worker breaks without anyone knowing, maybe surfacing only as a production ingestion error. Under the narrow-waist rule, worker simply cannot depend on web; their only "shared language" is the explicit contract in shared. Want to share logic? Sink it down into shared so it becomes a contract-bound first-class citizen, not something borrowed quietly from next door. "Dependencies only go down" is really using the compiler to block implicit cross-container coupling.

Look at the build task in turbo.json: "dependsOn": ["db:generate", "^build"] — meaning "before building any package, generate the DB client and first build the packages it depends on (^ = upstream deps)". That's why, in a big repo of dozens of packages, changing one line doesn't force a full rebuild.

Put the narrow waist and Turbo together and watch a concrete chain reaction: say you add a field to the trace. You edit one spot in packages/shared/prisma/schema.prisma and one in domain/traces.ts — one db:generate regenerates the Prisma client and both web and worker immediately see the new field, because both pull types from the same @langfuse/shared. The whole thing lands in one PR, verified together by CI. Split into three repos and this same change needs three releases, three dependency bumps, and a prayer that versions line up — that's the practical meaning of "change atomically".

🧩 polyrepo (split)

web/worker/shared each a repo. Change a shared structure → cross-repo releases, version juggling, easy drift. Upside: smaller, more independent repos.

📦 monorepo (Langfuse's choice)

one repo holds it all. A shared change is one atomic PR, verified together by CI. Cost: bigger repo, Turbo to manage build caching.

Caching is especially valuable for CI. In a dozens-of-packages repo, rebuilding + linting + testing everything on every PR would make CI unusably slow. Turbo computes a fingerprint of each task's inputs (source + deps + config) and, if the fingerprint is unchanged, reuses last time's output and skips the whole step. So editing one file in worker lets the unaffected web builds/tests hit cache and pass instantly. With a remote cache, teammates even share build results. That's the secret to keeping CI fast in a big repo — and the engineering precondition that makes "use a monorepo" viable: without good caching, monorepo build cost scares everyone off.

Life of a trace (bird's-eye)

Part 1 is nearly done. You now know what the data looks like (trace/observation/score), where the code lives (monorepo & narrow waist), and the design philosophy behind it (wide events). This lesson strings them into one line: follow one concrete trace from the moment your app calls langfuse.trace(...) all the way to it appearing in the UI list, being opened, even being auto-scored. It's a trailer — each leg is tagged with the later lesson that covers it, so by the end you have the whole map's route plan. Put differently, the first four lessons took the parts out one by one; this one puts them back and fires the engine once, so you see the data actually flowing — with that dynamic panorama, digging into each part lesson by lesson won't lose the forest for the trees.

🔌 Analogy
Think of a trace's life as shipping a parcel: at home you place the order and seal the box (create trace/observation in the SDK); a courier picks it up, scans it into a bag (the public API accepts the events and quickly enqueues); the parcel queues in a sorting center and is processed in the back (the worker pulls from the queue, merges, computes "shipping cost" = cost); then it's shelved into the warehouse (written to ClickHouse); finally you open your phone to track it (see the trace in the UI). The key: placing the order is not the same as being shelved — pickup, sorting and shelving sit in between, so there's a brief processing delay. That's the essence of Langfuse's "async" ingestion.
A trace's full journey: the top half is the write path (SDK→API→Redis→worker→ClickHouse, with S3 alongside), each leg tagged with its lesson (L12–19); the bottom half is the read path (ClickHouse→repository→tRPC→UI, L20–27). The two paths "meet" at the ClickHouse wide table: writes put data in, reads take it out.

Top half · write: SDK to ClickHouse

First the parcel's "shipped to shelved" journey. It's deliberately async — the API only accepts and enqueues fast, leaving the heavy lifting to the worker:

1

SDK creates events (L06)

Your app calls the SDK or emits an OTel span, producing events like trace-create / observation-create / score-create.

2

Public API accepts & enqueues (L12·14)

web/src/pages/api/public/ingestion.ts authenticates, Zod-validates a batch, then quickly drops it into the Redis queue and returns — without waiting for processing.

3

worker merges & aggregates (L15)

IngestionService pulls events from the queue, fetches this record's prior events from S3 and merges (multiple updates to one observation fold into one), deciding the final shape.

4

compute token & cost (L16)

Match the model name to the pricing table and backfill any unreported usage/cost (remember provided vs computed from L03?).

5

ClickhouseWriter batches the write (L17)

Batch processed records and write them to ClickHouse traces/observations/scores in one go (ReplacingMergeTree handles "later write of same id wins").

In this path, S3 plays "the raw event manuscript": each incoming event lands in S3 first, and the worker re-reads S3 when merging. That allows replay on failure and lets "multiple updates to one observation" merge correctly — details unpacked in L15–17. For now, remember the spine: accept → enqueue → merge → cost → store.

It's worth calling out why step 3, "merge", can't be skipped. The reason: an SDK often reports one observation not all at once. For an LLM call, the SDK might first send an "observation-create" saying "I started, model is gpt-4o", then after the call returns send an "observation-update" adding "output is X, used 1240 tokens". Those two events arrive separately, and the worker must merge them into one final record. That's why it re-reads S3 for "this id's prior events" — stitching the fragments that arrived at different times into one complete row. Grasp this and you see how Lesson 3's "observations can arrive out of order via parentObservationId" is actually realized in ingestion: events may come in multiple pieces, out of order, and the worker converges them into a consistent final state.

Once stored, seeing it in your browser takes a different, more real-time path — you click, it queries:

Note that write and read converge on this one ClickHouse table: writes put the wide event in, reads pull it out every which way. Lesson 2's "lists read narrow fields, detail reads big fields" is realized on exactly this read path.

Why separate "accept" from "process"

The cleverest step in this path is the API "accept and return". It splits the whole thing into a "fast lane" and a "slow lane":

One clean cut: between app and API is the fast lane (returns in ms, almost never slow); after the API is the slow lane (enqueue → worker batches in the background). The queue is a reservoir: spikes queue up, and you add workers when overwhelmed.

Consider: a high-traffic LLM app may produce thousands of observations per second. If the API had to synchronously merge, compute cost and write ClickHouse before returning, two bad things happen: one, every app call waits those extra hundreds of ms, needlessly slowed by your monitoring tool; two, if ClickHouse hiccups or a spike maxes it out, the API times out and even bounces errors back into your business. Cutting "accept" from "process" reduces the API to a featherweight "validate + enqueue" endpoint — almost impossible to slow, almost impossible to drag your app down via a downstream issue.

The queue is a reservoir: spikes queue up, the worker consumes in batches at its own pace, and you add workers (scale out) when it can't keep up. Langfuse even splits the ingestion queue into primary/secondary lanes (L14), isolating high-throughput big projects from ordinary ones so one project's spike can't drown everyone. The cost is the bit of lag mentioned next — but keep Δ to seconds and users barely notice. This is why nearly every high-throughput telemetry system (not just Langfuse) looks like this: lightweight accept + async process + batched store. Grasp this spine and you can predict what each Part 3 lesson is solving.

The cost of async: eventual consistency

Stitch the halves together and one fact must be clear in your mind: from SDK send to UI visibility there's queue processing in between, a few seconds of lag. This is eventual consistency.

What does "eventual consistency" mean in practice? It means that right after a call runs in your app, if you immediately switch to the Langfuse UI and refresh, you might not see it yet — wait a second or two, refresh, and it appears. For almost all observability use this is perfectly fine: you come after the fact to triage, view dashboards, evaluate — a few seconds don't matter. But it flags two things: first, don't treat Langfuse as a strongly-consistent business database — it's a telemetry system, not a "write-then-read-immediately" store; second, if you write automated tests to verify "is instrumentation correct", remember to allow ingestion time to process, or you'll falsely conclude "the data never arrived". Understand this boundary and the lag won't surprise you — you'll appreciate the "never drag the business" trade behind it.

Instrumenting an LLM app

In Lesson 5's pipeline, the first leg was "SDK creates events". This lesson focuses on that first leg: how, before data ever leaves your app, it gets instrumented into existence. You'll see Langfuse's two entry points — sending "native events" via the Langfuse SDK, or sending standard OpenTelemetry (OTLP). Understanding "what the app side emits" makes Part 3 ("how the server receives and processes it") land.

🔌 Analogy
Instrumentation is like fitting a building with sensors: you don't change the building's function, you just place recorders at key spots (door, elevator, server room) noting "who passed when, stayed how long". The Langfuse SDK is its own custom sensor — the best fit, richest signal; OpenTelemetry is like a standard sensor interface — if your building is already wired to that standard, Langfuse just plugs into the existing wiring. Two paths, same end: ship the app's "traces" out. This lesson's goal is to make clear "what the sensor end actually emits" — only by understanding the raw material the app sends can you grasp how the server later refines it into trace trees and wide-event tables.
Two entries: send native events with the Langfuse SDK (web/src/pages/api/public/ingestion.ts), or standard spans via OpenTelemetry OTLP (web/src/pages/api/public/otel/v1/traces, spans mapped to observations). Both converge on the same server-side ingestion path (Part 3).

The event envelope: what your app emits

Whichever entry, the app essentially emits a batch of events. Langfuse defines a set of event types (eventTypes in packages/shared/src/server/ingestion/types.ts), each describing "what was done to which object":

Event typeMeaning
TRACE_CREATEcreate/update a trace (the interaction's shell)
SPAN_CREATE / SPAN_UPDATEstart / end a span-type observation (a timed unit)
GENERATION_CREATErecord an LLM call (generation-type observation)
OBSERVATION_CREATE / OBSERVATION_UPDATEgeneric observation create / update
EVENT_CREATErecord an instantaneous event
SCORE_CREATEsubmit a score

Note there's both CREATE and UPDATE — this is the source of Lesson 5's "merge": a span usually starts with SPAN_CREATE and ends with SPAN_UPDATE (adding duration, output). The app batches these events into one HTTP request to the ingestion API, and the server merges them. So your langfuse.trace(...) and span.end() calls are essentially generating these events.

Conceptually, your instrumentation code looks roughly like this (SDK shapes differ by language; this just shows the "call → event" mapping):

# create a trace (emits TRACE_CREATE)
trace = langfuse.trace(name="chat", user_id="u_42")

# record an LLM call under the trace (emits GENERATION_CREATE)
gen = trace.generation(name="answer", model="gpt-4o", input=prompt)
result = call_llm(prompt)
# call ends; add output and usage (emits OBSERVATION_UPDATE)
gen.end(output=result, usage={"input": 800, "output": 440})

# score this answer (emits SCORE_CREATE)
trace.score(name="helpfulness", value=0.9)

Key observation: you merely "jot a note alongside" your existing business code — you didn't change call_llm's logic. Good instrumentation is as non-intrusive as possible — it shouldn't dictate how your app runs, only observe and record. This echoes Lesson 1: Langfuse sits beside your app, not inside it. Because instrumentation is "side-channel", its failure (e.g. a network hiccup) shouldn't take down your main flow — SDKs typically send asynchronously, batched, drop-on-failure, minimizing impact on the app.

Two entries: native SDK vs OpenTelemetry

Why support both? Because they serve two kinds of users:

🎯 Langfuse SDK (native)

The official Python/JS SDKs emit the native events above. Richest, best-fit signal (prompt linkage, usage, cost reported precisely), the default for most. Entry: ingestion.ts.

🔌 OpenTelemetry (OTLP)

If your system already uses OTel for observability and you'd rather not add another SDK, send OTel spans to Langfuse's OTLP endpoint (otel/v1/traces). Langfuse maps standard spans to observations. The cost: some Langfuse-specific semantics ride on convention-based attribute names.

This is a classic "embrace the standard vs play to your strengths" tradeoff. OTel is the de-facto standard for cloud-native observability, so supporting it lets Langfuse plug into existing OTel ecosystems — you needn't rewrite all your instrumentation to adopt Langfuse. But OTel's generic model doesn't natively understand LLM-specific concepts like "prompt version" or "token usage", so the native SDK is still more expressive. Langfuse chooses both: OTLP lowers the barrier, the native SDK gives the best experience. The server-side handling of the OTel path is its own Lesson 18.

create then update: events are "incremental"

Once more, a point crucial to understanding the whole path: an observation's info often isn't sent all at once. Look at a typical LLM call timeline:

t0 startGENERATION_CREATEmodel=gpt-4o, input=… (no output yet)
t1 endOBSERVATION_UPDATEoutput=…, usage=1240 tok, latency=1.8s
serverworker merges create + update into one final observation (L15)
Two events, one record: the CREATE and UPDATE arriving at different times are merged by the worker into one final observation. That's why the SDK can "send in pieces" while you see one complete row in the UI.

Why report "incrementally" instead of waiting and sending one complete event at the end? So the app can record "it started" earlier — even if the call later crashes or times out, you at least know "it did start and got stuck here". Wait until the end and a mid-flight crash makes that step vanish entirely, losing exactly the failure cases most worth observing. Incremental reporting + server merge is the protocol-level expression of "record as it happens" observability.

🎯 Design tradeoff
Where does the "incremental events + server merge" protocol put the complexity? On the server. The SDK is deliberately "dumb" — it just splits what happens into small events and ships them, not assembling the final state; the truly tricky "converge possibly-out-of-order, possibly-duplicate events into one consistent row" goes to the worker (L15). The payoff: the SDK is light, low-impact, easy to implement in every language; and the most complex, careful merge logic lives in one place (the server), not re-implemented (and re-bugged) in each language SDK. Converge complexity to one strictly testable place. This "thin client, thick server" trade shows up in many large systems; Langfuse just applies it to a telemetry protocol.
🎯 Key points
  • Instrumentation happens in your app: split the run-time trace into a batch of events sent to Langfuse, without changing business logic.
  • Two entries: Langfuse native SDK (ingestion.ts, most expressive) or OpenTelemetry OTLP (otel/v1/traces, plugs into existing OTel).
  • Event types (eventTypes): TRACE_CREATE / SPAN_CREATE / GENERATION_CREATE / OBSERVATION_UPDATE / SCORE_CREATE …, describing "what was done to which object".
  • Events are incremental (create starts, update completes), so "record as it happens" and failures aren't lost — the server merges (L15).
  • The protocol puts complexity on the server: SDKs stay light and portable; merge logic lives in one place.

The dual-store architecture

Last lesson, data left your app as "events". This lesson is about where it lands. Many assume one database suffices, but Langfuse uses four stores — Postgres, ClickHouse, Redis, S3 — each for its own job. Not showing off: different data has wildly different access patterns, and the wrong store will crush the hot path. Understand the division of labor and you understand the foundation of Langfuse's performance and scalability.

🔌 Analogy
Picture the four stores as a company's four "ways to keep things": Postgres is the filing cabinet — important records (who's an employee, project config), needing to be accurate, editable, quick to look up but not huge; ClickHouse is the giant warehouse — mountains of goods (hundreds of millions of events a day), needing bulk stock-taking by category; Redis is the conveyor belt + staging table — goods queue on the belt (queues), and frequently-used small items stay at hand (cache); S3 is off-site storage — big, rarely-flipped originals (event manuscripts, uploaded images, large exports). Each excels at its own; cram them into one and you lose.
The four stores' division of labor: Postgres=config/metadata (txn, consistent, small); ClickHouse=wide events (columnar, huge, fast aggregation); Redis=queue+cache (in-memory, fast, temporary); S3=event manuscript/media/exports (object store, big, cheap). web leans read, worker leans write, but both pick the store that fits the data.

Why not just one

The core reason in one line: transactional (OLTP) and analytical (OLAP) workloads are opposite optimization targets, and no single store nails both extremes.

  • Config/metadata (projects, users, API keys, prompt definitions) is classic OLTP: small, frequent precise single-row read/write, strong consistency (create a project, use it instantly), with foreign keys. Postgres's home turf.
  • Telemetry (trace/observation/score) is classic OLAP: enormous (hundreds of millions/day), write-mostly, queries are mostly time-window + dimensional aggregation ("cost by model over 7 days"). Columnar ClickHouse's home turf.

🗄️ Postgres is good at (OLTP)

precise single-row read/write, strong consistency, foreign keys, transactions. Fits small, important, must-be-instant config data. Bad at: aggregating scans over huge rows.

📊 ClickHouse is good at (OLAP)

time-window scans + aggregation over a few columns of huge rows, columnar + compressed. Fits massive, append-write, dimensional telemetry. Bad at: frequent single-row edits + strong consistency.

Cram telemetry into Postgres and huge rows + frequent aggregation overwhelm it; cram config into ClickHouse and its weak transactions, eventual consistency fail "instantly usable". So Langfuse isn't being greedy — it's letting each kind of data live in the store that fits it best, the storage-layer landing point of Lesson 2's "design around columnar access" and "denormalize carefully".

Two opposite access patterns: OLTP uses an index to hit exactly one row (read config, change settings); OLAP must scan a few columns of a big slice of rows and aggregate (cost by time/model). Row stores excel at the former, columnar at the latter — which is why config goes to Postgres and telemetry to ClickHouse.

In other words, these four aren't "add because we can" — each earned its existence. That's also the test for "should we add another store": does the problem it solves justify its long-term operational cost?

ClickHouse & the wide-event tables

Last lesson said telemetry goes to ClickHouse. This lesson opens the hood on how the three wide-event tables (traces / observations / scores) are actually built and why. It's the must-understand storage lesson — once you grasp the table engine, ordering key and partitioning, later questions ("why are lists fast", "why is by-project + by-time so efficient", "why can data be queried while being written") all fall into place. We use the observations table as the example (CREATE TABLE in packages/shared/clickhouse/migrations/unclustered/0002_observations.up.sql).

🔌 Analogy
Picture a ClickHouse table as a giant library. Partitioning is like big shelves by publish month — for "books from June 2024" you go straight to that shelf and never touch the others; the ordering key is the arrangement within a shelf (by subject, then date), so "a subject over a time range" is just a short contiguous scan; the ReplacingMergeTree engine is a shelving rule: a new edition of the same call number replaces the old (decided by version number). Together, even a vast collection is located and counted in milliseconds.
Five kinds of columns in one wide-event row: identity/linkage (tree, lookup), inlined rich wide columns (model, cost, prompt…), semi-structured Map columns (metadata, usage/cost details), ZSTD-compressed big fields (input/output), and version columns (event_ts, is_deleted). One row holds all of Lesson 3's rich attributes.

ReplacingMergeTree: later write of the same id wins

All three tables use the ReplacingMergeTree engine. The declaration looks like:

clickhouse/migrations/unclustered/0002_observations.up.sqlENGINE
ENGINE = ReplacingMergeTree(event_ts, is_deleted)
PARTITION BY toYYYYMM(start_time)
ORDER BY (project_id, type, toDate(start_time), id)

The key is ReplacingMergeTree(event_ts, is_deleted). Its semantics: multiple rows with the identical ordering key are "replaced" down to the latest one — whoever has the larger event_ts (version timestamp) wins; is_deleted=1 marks a soft delete. This perfectly fits Lessons 5 and 6's merge model: an observation that's created then updated produces two writes (same id, different event_ts), and ClickHouse automatically lets the update overwrite the create during background merge. You needn't read the old row, edit, and write back — just append; overwrite is the engine's job. This is the storage basis that makes Lesson 2's "immutable / append-oriented events" viable.

Later write wins: for two writes with the same ordering key, the engine keeps the newer event_ts; is_deleted implements soft delete. So "edit an observation" = append a record with a larger event_ts, no read-modify-write.

Ordering key + partition: why by project_id and time

Now the other two lines: PARTITION BY toYYYYMM(start_time) and ORDER BY (project_id, type, toDate(start_time), id). They decide how data is chunked and arranged on disk — and therefore whether a query can scan less.

📅 PARTITION BY month

toYYYYMM(start_time) cuts data into one chunk per month. For "last 7 days", the engine simply skips whole irrelevant months (partition pruning) — never even reads them.

🔑 ORDER BY project_id first

The first ORDER BY column is project_id, so one project's data is physically adjacent on disk. Multi-tenant queries naturally see only their project — a contiguous range, no full-table filtering.

Scanning less is the point: partitions prune irrelevant data by month, the ordering key (project_id first) clusters the target project into a contiguous range, and columnar storage reads only the columns you want. Together, "by project, by time" — the hottest query — touches only a small slice even across hundreds of millions of rows.

🎯 Design tradeoff
ReplacingMergeTree's overwrite is "eventual", not "immediate" — background merge takes time, so in principle you could read the old row in the instant both writes exist but aren't merged yet. That's the price of "just append, write fast". The fix: on the query side use the FINAL keyword to force-merge before reading, or use aggregation (take the max-event_ts row per id) to guarantee the final state — moving the "dedup" cost from every write to query time (and controllable). Lesson 2 says "avoid read-time dedup's hidden cost"; Langfuse's approach is to route regular queries through aggregation/final-state views, making the cost explicit and optimizable rather than paying read-modify-write on every write. Append for throughput, move dedup to the read side and optimize it — a classic write-heavy-telemetry trade.
🎯 Key points
  • Engine = ReplacingMergeTree(event_ts, is_deleted): same ordering key, later write (larger event_ts) overwrites earlier, is_deleted soft-deletes — perfectly absorbing the create+update merge model.
  • Partition = by month (toYYYYMM), so time-window queries prune whole chunks; ordering key = project_id first, so multi-tenant queries scan one contiguous range.
  • "Scan less" via partition pruning + ordering-key locating + columnar column-selection is the key to scale efficiency.
  • Columnar + ZSTD + Map columns + bloom_filter indexes keep wide, big rows fast to query and cheap to store.
  • Tradeoff: overwrite is "eventual"; dedup cost moves from writes to queries (FINAL/aggregation), buying fast writes — avoiding read-time dedup's hidden cost.

The metadata schema (Postgres)

The last two lessons starred ClickHouse (massive telemetry). This one switches to the other half: Postgres. If ClickHouse stores "what happened" (trace/observation/score), Postgres stores "how the platform is configured" — who your orgs and projects are, who has access, which API keys exist, which prompts are defined, which eval rules are set, how each model is priced. This side is the control plane — small in volume yet global in importance. Its shape is all in packages/shared/prisma/schema.prisma.

🔌 Analogy
Think of the two stores as a company's two systems: Postgres is the HR/admin system — who's an employee, in which department, with what permissions, under which policies. Not much data, but it must be accurate, editable, instantly effective (a new hire's permissions must work right away). ClickHouse is the transaction ledger — mountains of records, append-only, used later for monthly/category stats. You wouldn't write policies into the ledger, nor stuff every transaction into HR files — two kinds of data, two systems, each in its place.
Two halves: Postgres stores "how the platform is configured" — org/project/user, API keys, prompts, eval/dataset config, pricing, integrations, audit; ClickHouse stores "what actually happened" — three wide-event tables. This is the macro version of Lesson 3's "domain shape vs physical storage": one governs control, one governs data.

🎯 Design tradeoff
What's the biggest payoff of splitting "control plane" and "data plane" into two databases? Letting each evolve and scale by its own rules. The control plane is small and needs strong consistency — fine for a single (or primary/replica) Postgres; the data plane is a flood needing a horizontally-scalable ClickHouse cluster. Force them into one store and you either shortchange config consistency or crush telemetry throughput. The cost: some operations span two stores (e.g. ingestion reads model pricing and prompt links back from Postgres) — but this "read config, write data" cross-store collaboration is controllable and heavily cached, far more realistic than "making one store great at both OLTP and OLAP". Split by responsibility; let each store do only what it's best at.

Multi-tenancy

Last lesson called project the hub and the "data boundary". This lesson makes that concrete: Langfuse is a multi-tenant platform — one deployment serves thousands of unrelated teams at once, isolating their data cleanly via three nesting levels: organization → project → environment. Grasp these three and you understand "why you only see your own data" and "why the ClickHouse ordering key must start with project_id".

🔌 Analogy
Picture multi-tenancy as an office building: the organization is the company renting a floor — signs the lease, pays the bills, manages its staff (members and billing live here); the project is a department with its own area and access control, whose stuff never wanders into another department; the environment is a further "production / test area" split inside the department — the same people keep live and test data apart. Everyone may be in one building (one deployment, one database), but layered access controls keep them from crossing over.
Three nesting levels: one organization holds multiple projects, each project may split into multiple environments (e.g. production / staging / development, default default). The org manages billing and members, the project is the hard data boundary, and the environment is a soft slice within a project.

What each level governs

LevelGovernsIsolation strength
Organizationbilling, members, SSO, org-level API keysthe outer "account" boundary

The key distinction is the last two: project is a "hard boundary" — hold one project's key and you simply cannot see another project's data; environment is a "soft slice" — within one project it tags live vs test data apart, so you can view only production, or compare several environments together. Why? Because "different teams' data must never mix" is a security red line (hard isolation), while "live vs test" is just one team's own view preference (soft slice) — different strengths, different treatment.

A concrete scenario for all three levels. A company (organization) signs up and pays on Langfuse, inviting 5 engineers as members. They open two projects (project): "support bot" and "doc Q&A". The two projects' data is mutually invisible — querying the support bot never mixes in doc-Q&A traces. Within "support bot" they use environment to separate live traffic (production) from load-test/debug traffic (staging), so live cost isn't polluted by load-test data — yet they can compare the two environments together when needed. The outer "company" cares only about the bill and who's a member, holding no data directly; all data lands on some project + environment. This "org manages money and people, project manages data, environment manages views" division is the entire skeleton of multi-tenancy.

project_id is the hard isolation key, threaded throughout

Multi-tenant isolation is not as simple as adding a WHERE project_id = ? filter afterward. In Langfuse, project_id is a thread running through the whole pipeline:

project_id threads the whole pipeline: auth resolves project_id from the API key (L09: a key belongs to a project); writes stamp it on every row; ClickHouse's ordering key starts with project_id (L08); queries locate the contiguous range by it. So isolation is not an afterthought filter — it's baked into the physical layout; isolation and "fast" are the same thing.

This "project_id threads the whole pipeline" design's biggest value is turning security from "the programmer remembers to filter" into "the structure guarantees it". Consider: if isolation relied on every query remembering to add WHERE project_id = ?, then one forgotten or wrong filter could let company A see company B's data — a catastrophic bug in a multi-tenant system. Langfuse instead welds project_id into the storage and auth foundations: the repository read APIs uniformly require a project_id, and tRPC's protectedProjectProcedure middleware checks first whether you belong to this project (L21). So "see another tenant's data by accident" isn't "easy to get wrong" — there simply is no such path. That's the essential difference between "security in the structure" and "security by discipline".

environment: a soft slice within a project

environment was added later. Here's how it looks in ClickHouse — all three wide-event tables gained an environment column:

clickhouse/migrations/unclustered/0008_add_environments_column.up.sqlALTER
ALTER TABLE traces       ADD COLUMN environment LowCardinality(String) DEFAULT 'default' AFTER project_id;
ALTER TABLE observations ADD COLUMN environment LowCardinality(String) DEFAULT 'default' AFTER project_id;
ALTER TABLE scores       ADD COLUMN environment LowCardinality(String) DEFAULT 'default' AFTER project_id;

Three details: (1) the type is LowCardinality(String) — env names are a small fixed set (prod/staging/dev), and low-cardinality encoding saves space and speeds queries; (2) the default 'default' — without an explicit environment, data lands in default, backward-compatible (old data naturally belongs to default); (3) the position AFTER project_id — this only places it right after project_id in the table's column layout (purely a cosmetic column order for readability); note it is NOT added to the ordering key. So unlike project_id, environment is essentially a plain filter column on the row — a soft slice: add a condition to filter by env, or ignore it to see them together.

This neatly contrasts the two isolation mechanisms: project_id is welded into the ordering key (isolate = locate, structurally safe), while environment is just an ordinary column (filtered at query time). Reasonably so — hard isolation must be guaranteed by structure, while a soft slice is just a convenient view dimension that needn't (and shouldn't) wield the heavy weapon of the ordering key.

🎯 Design tradeoff
Why weld project_id into the ordering key instead of treating it as an ordinary filter column? Because that makes "isolation" and "efficiency" the same thing. Configuration is all environment variables

How do the six containers know each other's addresses, secrets and toggles? Environment variables. The repo has a set of .env.*.example templates (e.g. .env.dev.example, .env.prod.example) listing every variable to set (DB connections, S3 endpoint, encryption keys…). More importantly, these envs aren't read loosely — they pass through Zod validation (packages/shared/src/env.ts, worker/src/env.ts, etc.), checking types and required fields at startup. Misconfigure or miss one and the process fails to start and tells you exactly which variable is wrong, rather than running sick and crashing midway. This "validate config at startup" discipline turns "config errors" from hard-to-find runtime bugs into obvious startup errors (expanded in L51).

Put it together and the first self-host experience is actually simple: clone the repo → copy an .env.dev.example and fill the key variables → docker compose up, and the six containers come up in dependency order, connect, and within minutes you open a browser to the Langfuse UI. The repo also ships several compose files for different scenarios — docker-compose.dev.yml (local dev, often just the infra while the apps run on your machine for hot reload), docker-compose.yml (the full stack), and cloud-specific variants (e.g. Azure). This "one orchestration for the whole stack, variants per scenario" is exactly how the earlier "four-dependency complexity" gets folded into a few yml files — you needn't hand-install each database or wire the network; compose strings it all together. Grasp this topology and you'll not only "get it running" but also know which container's logs to check when something breaks.

Are four infra dependencies a burden for self-hosters? Honestly, yes — versus "one binary + one SQLite", Langfuse's self-host bar is genuinely higher. But that's the deployment-side consequence of Lesson 7's tradeoff: to handle production scale (massive telemetry, async ingestion, object storage), none of the four is droppable. Langfuse's mitigation is bringing up all six containers with one docker-compose command and using a MinIO stand-in to remove the dependency on cloud S3, making "run the whole thing on your own machine" easy. Complexity can't be erased, but it can be packaged and delivered in one command. That's why the docs say "self-host in minutes" — not no dependencies, but dependencies orchestrated for you.
🎯 Key points

The ingestion API

Part 2 finished "what the data looks like and where it lives". From here we walk the ingestion path — the close-up of Lesson 5's "write trunk". First stop: the public ingestion API — how the server receives a batch of events your app sends. Spoiler on its character: extremely fast and light. It does just three things — auth + validate + enqueue — then returns immediately, never merging, computing cost, or writing the DB here (those are the worker's job). This is Lesson 5's "fast lane".

🔌 Analogy
Picture the ingestion API as a courier's drop-off counter: the courier (your SDK) brings a batch of parcels (events), and the clerk (the API) does three things before waving you through — check the sender's identity (auth: which project is this key?), scan and log the labels (validate: is each parcel well-formed?), drop them on the belt (enqueue). Then it hands you a receipt on the spot (HTTP 200) and lets you go — it won't sort, weigh and deliver in front of you. The slow work is done by people down the belt (the worker). The counter's only goal: accept fast, never block the door.
A minimal accept flow: auth (API key resolves a project, L09·10) → Zod-validate event shapes → processEventBatch enqueues (and lands originals in S3) → immediate 200. No "heavy work" happens here, so the API always returns in milliseconds.

The life of one ingestion request

The entry is the handler in web/src/pages/api/public/ingestion.ts. After taking the body, the real work is a shared function called processEventBatch:

web/src/pages/api/public/ingestion.tshandler
import { processEventBatch } from "@langfuse/shared/src/server";

export default async function handler(req, res) {
  // …auth, parse body…
  const result = await processEventBatch(batchForProcessing, authCheck);
  // return the result immediately (success / partial failure), not waiting for the worker
}

One design detail: processEventBatch is shared — besides /ingestion, even the standalone "submit a score" endpoint reuses it. Why? Because "accept a batch, validate, enqueue" is needed everywhere; factoring it into one function avoids re-implementing (and re-bugging) it per endpoint (again Lesson 6's "converge complexity to one place"). The event types it supports cover TRACE_CREATE / OBSERVATION_CREATE / SPAN_CREATE / GENERATION_CREATE / SCORE_CREATE and more (including SDK_LOG, DATASET_RUN_ITEM_CREATE) — exactly Lesson 6's event envelope.

Why a "batch", not "one"

The SDK doesn't fire one HTTP per event — it collects a batch and sends it. This "batch" design saves a lot:

😖 one-by-one

Every observation a full HTTP round trip; at high frequency the network overhead is huge and connections explode — a real drag on your app.

😀 batched

Dozens or hundreds of events in one request, one round trip. Network overhead amortizes, and the API validates and enqueues in one go — far higher throughput.

Batching also gives a semantic benefit: a batch may hold both a CREATE and an UPDATE for one observation, and the server can process them together, cutting round trips. Of course a batch can't be infinite — when the body is too large the API rejects it ("payload too large"), another guard: don't let one request blow up the server.

Two gates: auth + validation

Before "enqueue", the API must pass two gates. They look routine but are the system's first line of security and robustness:

Two gates: auth resolves the project from the API key (L09: a key belongs to a project) and confirms write permission — this also pins the tenant (L10's project_id threading); Zod validation checks each event's shape, so bad data is rejected at the door and never pollutes downstream. Pass both, then enqueue.

The second gate, Zod validation, especially matters: the ingestion system swallows a flood of data from many SDKs and versions daily, inevitably with malformed or missing-field "dirty events". Rejecting them at the entry with a Zod schema and a clear error beats letting them flow into the queue and crash the worker — the earlier you reject bad data, the calmer the whole system. If a few events in a batch are bad, the API can still return "partial success", telling you which weren't accepted, rather than failing the whole batch.

🎯 Design tradeoff
Why does the API only "auth + validate + enqueue", pushing all heavy work to the worker? Because the ingestion API is the layer facing your app directly, and its top KPI is "always fast, never drag the caller". Any extra work here — merging, looking up pricing, writing the DB — becomes latency on every instrumentation call and can fail on downstream hiccups. So Langfuse cuts this line very cleanly: the API keeps only the minimum that must happen on the spot (auth to prevent unauthorized writes, validation to keep dirty data out, enqueue to hand off), and everything else that can be async is async. The cost is "eventual consistency" (L05), but the gain is an ingestion entry that's almost impossible to slow or to drag your business down. The thinner the entry, the better it survives spikes.

Event types & merge semantics

Last lesson the API accepted a batch of events and enqueued them. Now we zoom into a single event: what does it look like? Why does it have two faces, create and update? And most crucially — how do the several events your SDK sends end up as one row in the database? This is the soul of Langfuse ingestion: an event stream merges into a record. Get this and Lesson 8's ReplacingMergeTree(event_ts, is_deleted) design finally clicks into place.

🩺 Analogy
Think of one observation as a hospital chart. When the patient checks in, the doctor writes the first visit note (generation-create: start time, the input complaint). After tests and a prescription, the doctor adds a note (generation-update: end time, diagnosis, dosage). Both notes carry the same patient number (entity id) — so when filing, the nurse doesn't create two charts, she merges them into one, latest note winning on conflicts. Each note also has its own serial number (event id), so filing the same note twice never duplicates it. "Merge by patient number, dedup by serial" — exactly the two ids this lesson dissects.

The envelope: id · timestamp · type · body

Every ingestion event — whether it describes a trace, an observation or a score — rides in the same envelope: a shared "shell" plus a type-dependent payload body. Langfuse pins this down with Zod: a base (the shell), each type extends it with its own type and body, and finally a discriminated union (z.discriminatedUnion("type", […])) gathers the dozen-plus events into one schema. To validate, Zod reads the type field and knows which body rules to apply — "one field sets the identity, one ruleset per identity".

Envelope = fixed shell + variable body: base holds id (event id), timestamp, metadata and the discriminator type; type decides which Zod ruleset the body follows. That's exactly how z.discriminatedUnion("type", […]) works. Source: packages/shared/src/server/ingestion/types.ts:597.

const generationCreateEvent = base.extend({

The beauty of the discriminated union: it isn't "make every field optional, fill in whatever" — it's a strict body ruleset per type. Send generation-create and you must match CreateGenerationBody; send score-create and you must match ScoreBody — wrong type or missing field, and Lesson 12's Zod gate rejects it on the spot. The schema even ships as two instances, public and internal: things like dataset-run-item-create are internal-only and rejected on the public API.

The body "matryoshka": Event ⊂ Span ⊂ Generation

Those body schemas aren't written separately — they inherit layer by layer. The innermost is OptionalObservationBody (traceId, name, startTime, input/output…); extend it with id to get CreateEventEvent, add endTime for CreateSpanBody, then add model / usage / modelParameters for CreateGenerationBody. This chain maps precisely onto Lesson 3's three observation kinds: EVENT is a point in time (no duration), SPAN has start & end (adds endTime), GENERATION is a SPAN with LLM fields (adds model/usage). Tellingly, the newer observation types from Lesson 8 — AGENT / TOOL / CHAIN / RETRIEVER / EVALUATOR / EMBEDDING / GUARDRAIL — all reuse CreateGenerationBody: at heart they're each "a generation-rich span", differing only in semantic label.

The further out, the more capable: EVENT is a duration-less point → SPAN adds endTime and gains start/end → GENERATION adds model/usage, becoming an LLM call that records tokens and cost. The 7 newer types (AGENT/TOOL/CHAIN…) reuse CreateGenerationBody directly. Source: types.ts:421-470.

This inheritance doesn't just save code, it makes the type semantics escalate step by step: you can't put an endTime on an EVENT (its body simply has no such field), nor a usage on a SPAN. The schema is the documentation — read the chain and you see at a glance what each of the three observation kinds can and cannot carry.

The heart of it: several events, one row

Now to answer the opening question. For one LLM call, the SDK usually sends two events: generation-create at the start (with startTime, input) and generation-update at the end (with endTime, output, usage). Yet they land as the same observation row in ClickHouse. The secret is two ids, each with its own job:

entity idbody.id (merge key)
The id of the observation/trace itself. All events with the same body.id are destined to merge into one row. create and update share it, so they point at the same record.
event idenvelope base.id (dedup key)
Each event's own serial. In S3 it becomes the filename <eventId>.json — resend the same event, overwrite the same file, naturally idempotent, never counted twice.
versiontimestamp (who wins)
When merging conflicting fields, the later-timestamped event wins (last-write-wins). This is the write-side mirror of Lesson 8's ReplacingMergeTree(event_ts).
create + update → one row: the two events share body.id=obs-42, get grouped and merged; each field takes the side that has a value, conflicts resolved by later timestamp. The worker's merge logic (Lesson 15) is a "write-side rehearsal" of Lesson 8's ReplacingMergeTree behavior.

Break this "event stream → one row" path into five steps and the whole merge story is clear — the first two happen on your app side, the last three on the Langfuse server:

1

Start: send create

As the LLM call begins, the SDK sends generation-create with body.id, startTime, input. Fire-and-forget, nothing held in memory.

2

End: send update

When the call ends, the SDK sends generation-update, reusing the same body.id, adding endTime, output, usage.

3

Group by body.id

The server groups events with the same dedupKey = entityType-body.id — these two are recognized as "two reports of one record".

4

Merge fields (latest wins)

Take the set side of each field; where both create and update gave a value, the later timestamp wins.

5

Land as one row

The merged result writes into one row of observations (id=body.id); ReplacingMergeTree re-dedups in the background as a safety net (Lesson 8).

🎯 Design tradeoff
Why not just do an upsert — why split into two create / update events? Because the SDK shouldn't carry state. An LLM call can take seconds, even tens of seconds, between start and end; meanwhile the app might crash or move machines. Requiring "wait until the end, then upsert the whole thing" forces the SDK to hold that record in memory until completion — costing memory and risking loss. Splitting into create + update lets the SDK send half at the start, the rest at the end, forgetting each as it goes; the dirty work of merging by body.id falls to the server. Pushing state from the client to the center is the classic distributed-systems move — "keep the edge stateless, let the center backstop it" — echoing Lesson 5's fast-lane / slow-lane split.

The ingestion queue

Lesson 12 had the API "enqueue" and vanish; Lesson 13 explained what "event" it enqueues. The star of this lesson is the queue itself — a conveyor belt built on Redis + BullMQ. It's the handoff point between Lesson 5's fast lane and slow lane: web drops the work and leaves, while the worker pulls it off the other end and processes at its own pace. Three questions to answer: what's actually in the queue (spoiler: not the event itself, but a "claim ticket"), why split into a primary / secondary queue, and how sharding survives spikes under high concurrency.

🛄 Analogy
Think of the queue as an airport baggage belt. The check-in desk (API) doesn't pile your suitcase on the counter — it sends the case to the back room (S3) and puts only a baggage tag (a pointer: the case's id in the back room) on the belt. A handler (worker) takes the tag off the belt, fetches the case by its id, then processes it. The beauty: the belt (Redis memory) only carries featherweight tags, fast and cheap; the bulky cases (event bodies, possibly large) rest quietly in the back. If handlers fall behind, tags simply queue on the belt and not a single case is lost — because the back room is the goods' "real ledger".

The queue holds a "claim ticket", not the event body

Here's the most-misunderstood point of the whole ingestion path: the Redis queue does not store the event's full content. Before enqueuing, Lesson 12's processEventBatch first writes each event to S3 named <eventId>.json (Lesson 13's event id becomes the filename); the queue holds only a lightweight job whose key field is a fileKey — a pointer to that S3 file. The worker takes the job, then reads the event back from S3 by fileKey.

Pointer and body, separated: the event body is written to S3 (<eventId>.json); the queue holds only a lightweight job carrying fileKey. The worker reads the body back from S3 by fileKey. Source: processEventBatch.ts:340-398.

📨 the queue holds a pointer

The payload is a tiny { type, eventBodyId, fileKey, authCheck } job of a few dozen bytes. Redis memory is precious; the smaller the job, the faster the enqueue and the better it holds up under backlog.

📦 S3 holds the body

The raw event (input/output can be large) sits in S3. It's also the historical event source Lesson 15's merge reads back, and the basis for rebuilding should a job ever be lost.

One more detail hides in the enqueue: queue.add(…, { delay }) attaches a small delay (about 5s by default). Why be deliberately "a beat slow"? Because the create and update for one entity may arrive back to back; if a worker processes the create before the update lands in S3, duplicates or out-of-order writes can result. The delay lets a batch of related events "settle" first, so the worker takes them together, naturally reducing duplicate processing. Better still, getDelay lengthens the delay around the UTC 23:45–00:15 "date boundary" — because Lesson 8's ordering key contains toDate(start_time), events straddling midnight, if processed too eagerly, could split across two date partitions and duplicate. This unassuming delay is exactly what safeguards Lesson 8's day-organized storage — a classic "trade a little latency for consistency".

Sharding: split one queue into N, survive the spike

A single Redis queue always has a ceiling. When a deployment enables Redis cluster (REDIS_CLUSTER_ENABLED=true), the ingestion queue is sharded into several: ingestion-queue, ingestion-queue-1, ingestion-queue-2… the count set by LANGFUSE_INGESTION_QUEUE_SHARD_COUNT. Which shard depends on the hash of the shard key projectId-eventBodyId: SHA-256 produces a number, taken modulo the shard count to yield a shard index.

Shard without scattering an entity: the hash is over projectId-eventBodyId, so all events of one entity (create/update) necessarily land on the same shard — spreading load across Redis nodes while keeping a record's multiple reports ordered on one worker. Source: redis/ingestionQueue.ts:38-52, redis/sharding.ts:9.

Choosing projectId-eventBodyId as the shard key — rather than a random number — is deliberate: random sharding would scatter one record's create and update across different shards, forcing cross-queue coordination at merge time, complex and prone to disorder. Using the entity id as the shard key makes "same entity, same shard" hold naturally — the same idea as Lesson 8 leading the ordering key with project_id and Lesson 13 using body.id as the merge key: keep related data physically together.

🎯 Design tradeoff
Why insist on queue + S3 — why not have the API write the DB directly? Because the combo buys both decoupling and durability. IngestionService: the merge heart

Lesson 14's worker took the job and read the events back from S3 by fileKey. This lesson is the heart of the whole write path: how does that pile of events, plus the record's current state in the database, get kneaded into one final record? The answer lives in IngestionService — mergeAndWrite dispatches by entity type, mergeRecords folds many inputs into one with a single "left-fold + later-overwrites" move, then stamps a event_ts=now version. Get it, and Lesson 8's ReplacingMergeTree gains its "other (write-side) half".

📝 Analogy
Think of merging as collaboratively editing a wiki article. The database's currently published version is the base; each event received this time is like a chronologically ordered revision, applied onto the base in turn. Later edits override earlier ones (same field, newer value wins); but a field a revision didn't touch stays as is (you only changed the "conclusion", the "body" isn't wiped). A few things are locked, untouchable by anyone: the article's creation time, its unique id. Stack all revisions and you get a merged draft, tagged with "this merge's moment" as the version — exactly what IngestionService does: the base from ClickHouse, the revisions from S3, the merged draft written back.

mergeAndWrite: split by entity type first

mergeAndWrite itself is just a dispatcher. It takes Lesson 13's computed entity type (trace / observation / score / dataset_run_item) and a switch hands the work to the matching process{Type}EventList. Every line has the same skeleton — read old, transform, merge, write back — only the field details differ by type.

On each line, the merge actually has two sources, both indispensable:

📜 S3: all events of this entity

The worker uses listFiles to read back all events of this entity in S3 (create, update… none missed), then sorts them by timestamp. This is the "revision stream".

🗄️ ClickHouse: the current record

Meanwhile it getClickhouseRecords the existing row from observations as the base draft. It goes first in the merge array, the baseline for immutable fields.

Why need both? Seemingly redundant, but each is irreplaceable. The S3 one is the events' real ledger (Lesson 14); re-reading all events guarantees the merged record is complete, even if this job is a late backfill. The ClickHouse one provides the authoritative baseline for immutable fields (created_at, start_time — those "locked once first written" values) and backstops early events that may have aged out of S3. Stacked together, they lose nothing new while changing nothing old — the essence of "read the old base + apply new revisions".

mergeRecords: left-fold, later overwrites

The core algorithm is just a few lines: assemble "ClickHouse base + time-sorted event records" into an array, fold them left-to-right with overwriteObject, then stamp event_ts = now. The crux is entirely in overwriteObject's three "don't overwrite" rules.

It stacks "new fields that have a value": a later event overwrites an earlier one, with three exceptions — immutable keys, undefined values, empty objects. So update carrying only endTime/output/usage leaves create's startTime/input/model untouched. Source: IngestionService/index.ts:984-1005, utils.ts:56-81.

Token counting & cost

At step 6 of Lesson 15's merge we said in passing: "compute tokens & cost". This lesson unfolds that step. The core is a principle running through all of Langfuse — provided beats computed: the usage / cost you (or the LLM provider) report are always trusted first; only when you report nothing does Langfuse roll up its sleeves: count tokens with a tokenizer, price it from a rate card. This is the origin of Lesson 3's two field sets provided_* and usage_details, and the reason Lesson 8's provided_* Map columns exist.

📞 Analogy
Think of a phone bill. If the carrier already itemized it (how many minutes you used, the charge for each), the system just files it as is — that's provided. But if all you have is a pile of call logs with no amounts, the system must do it itself: first count the call durations (tokenize, count tokens), then pull up this plan's rate card (model prices) and compute what's owed — that's computed. Note: the moment the carrier gives even one amount, the system trusts its bill wholesale, never mixing "its own estimate" with "the carrier's figure" — mixing the two scrambles the books.

In real instrumentation, the two forks map to two very common integration styles:

🅰️ Official SDK: provided

The system's job is to record faithfully, not to "correct" an authoritative source with its own estimate. ClickhouseWriter: batched persistence

Lesson 15's merge ended with clickHouseWriter.addToQueue(...) — the record goes into a queue, not straight to the DB. This lesson covers what's behind that queue, the ClickhouseWriter: a singleton that gathers the flood of single records into a handful of big batches, then INSERTs them into ClickHouse in one shot. Why batch at all? Because, as Lesson 8 said, ClickHouse strongly prefers large batched writes and dreads row-by-row inserts. This layer is the buffer that translates "high-frequency small writes" into "low-frequency large writes".

🚌 Analogy
Think of ClickhouseWriter as a shuttle bus. Passengers (records) arrive one by one, but the driver doesn't depart for each one — the fuel cost (write overhead) would be absurd. He waits until the bus is full (hits batchSize), or the departure time arrives (the writeInterval timer fires), then takes the whole busload across in one trip. Whichever condition comes first wins: at rush hour the bus fills fast and leaves on "full"; in a lull, few passengers, so it leaves on "time" so the last few don't wait forever. And each route (each table) has its own bus: one for traces, one for observations, one for scores, each loading and departing independently, never holding up the others.

The buffer: one queue per table, two departure conditions

Why does ClickHouse so dread "row-by-row writes"? Back to Lesson 8's storage mechanics: a MergeTree drops a new data part on disk for every INSERT, then relies on background merges to combine small parts into large ones. INSERT each record separately and you instantly spawn a flood of tiny parts — exploding disk metadata, background merges that never catch up, slowed queries (and even ClickHouse's "too many parts" guard rejecting writes). Batch ten thousand records into one INSERT and you create just one big part — easy to write, merge, and query. So this layer isn't an optional optimization; it's a precondition for a columnar store like ClickHouse to withstand high-frequency ingestion — without this buffer, even the best table design gets crushed by a deluge of small writes.

"Full" and "on time" each serve a different load, and both are needed:

Complementary conditions + graceful shutdown: rush hour leans on batchSize for throughput, lulls on writeInterval to cap latency; shutdown first clearInterval then flushAll(true) writes out everything left in memory. flushAll fans out across tables with Promise.all.

Unpack a merged record's journey through the writer, and it's exactly five steps:

1

enqueue

2

3

This table hits batchSize (by size) or the timer fires (by time); splice takes up to a batch.

4

5

On success the batch is done; retryable errors back off and retry, oversized batches split and requeue — sparing the rest of the batch.

Why a singleton, and why dare to buffer in memory

ClickhouseWriter is a singleton (private constructor + getInstance): one writer, one set of queues per worker process. That's crucial — only by converging to one place can scattered records from countless jobs actually batch up large; if each job wrote on its own, batches would never grow. Writes carry backoff retry (maxAttempts): retryable errors back off and retry, "string too long" errors split the batch and retry — chasing big batches without letting one bad record drag down the whole batch. This "the batch rises and falls together, the bad ones get pulled out alone" strategy echoes Lesson 12's ingestion-API "partial success": at every layer the system tries to be fault-tolerant without collateral damage, so a problem record or two won't block the whole pipeline.

🧩 singleton + per-table queues

One writer per process gathers all jobs' records into big batches; per-table queues let traces/observations/scores each fill and flush in parallel, never blocking each other.

🛡️ bulk write + backoff retry

One INSERT per whole batch (JSONEachRow); failures retry with maxAttempts backoff; oversized batches auto-split. "Fast" and "stable" together.

🎯 Design tradeoff
Records sit in an in-memory queue — if the worker suddenly crashes, isn't the batch lost? It is — but that doesn't matter, because the real "ledger" isn't here. Recall Lesson 14: every event rests verbatim in S3, and the queue job isn't acked until the DB write succeeds. So a mid-flight worker crash, at worst, means this batch of merged results didn't land; the corresponding queue job is redelivered, re-reads events from S3, re-merges and re-writes — not a single record lost. Precisely because durability is already backstopped by upstream S3 + the queue, this layer can safely trade volatile in-memory buffering for write throughput. A classic layered-design insight: assign durability and efficiency to different layers, each taken to the extreme — S3 owns "never lose", ClickhouseWriter owns "write fast", neither compromising for the other's goal.
🎯 Key points
  • ClickhouseWriter is a singleton that gathers high-frequency single records into big batches for one INSERT — because Lesson 8's ClickHouse prefers large batches and dreads row-by-row.
  • One in-memory queue per table (traces/observations/scores…); addToQueue feeds it, flushAll fans out across tables with Promise.all.
  • Two flush triggers: ① queue ≥ batchSize (by size, for throughput) ② every writeInterval timer (by time, capping latency); whichever first.
  • shutdown clears the timer then flushAll(true) drains all queues so nothing is lost on exit; writes carry backoff retry, oversized batches auto-split.
  • It dares to trade volatile memory buffering for throughput because durability is backstopped by upstream S3 + the queue (Lesson 14): crash → redeliver and recompute, nothing lost — layers each doing their job.

OpenTelemetry ingestion

The last six lessons covered Langfuse's native ingestion — you use its own SDK, report in its own event schema. But in reality, many teams already instrument with OpenTelemetry (the industry observability standard). This lesson covers how Langfuse opens its arms to OTLP: mapping the motley assortment of OpenTelemetry spans onto its own observation model. The key insight — OTel ingestion is essentially a universal adapter: the front end translates the various conventions, and once translated, it flows into the exact same merge / cost / persist pipeline as native.

🔌 Analogy
Think of the OTel endpoint as a universal travel adapter. Your devices (apps) come from all over, with different plug standards — some OpenTelemetry's gen_ai.* convention, some Vercel AI SDK's ai.*, some OpenLLMetry's llm.*. The universal adapter recognizes them all: it tries, one by one, "what's this field called in your convention", and once it recognizes it, routes the current into the same socket (Langfuse's observation model). The circuitry behind the socket (merge, cost, write) is just one — the adapter only "turns various plugs into one interface"; after that, everyone flows the same way.

🎯 Design tradeoff
If the native channel is more precise, why bother supporting OpenTelemetry? Because you meet users where they are. OpenTelemetry is the de facto observability standard; countless teams' apps are already instrumented with it. If Langfuse only spoke its own SDK, it would demand users re-instrument just for it — a barrier that turns many away. Supporting OTLP means "you change not one line of instrumentation, just point your OTel data over and it works". The cost is real: an adapter that must keep chasing each convention, full of priority fallbacks, plus the semantic loss of inference. But the math works out — because the adapter lives only at the very front, and after translating it converges into the shared core (merge, cost, write is just one). Confine the grunt work and special cases to one adapter layer at the edge, keep the core pure and unified — the most classic use of the "adapter pattern".

Media & blob storage

In the multimodal era, an LLM call's input/output may carry an image, an audio clip, a PDF — binaries of several MB. Write those verbatim into Lesson 8's ClickHouse wide tables and the input/output columns balloon instantly, queries slow, storage is wasted. Langfuse's answer is clean: large binaries never enter ClickHouse at all; they're stored separately in an S3 media bucket, and the event keeps only a tiny reference string. This lesson covers that media-storage design — keeping bulky blobs off the hot path.

📚 Analogy
Think of a library. The catalog card (the ClickHouse event) never transcribes the whole book, just a call number (the reference string @@@langfuseMedia:…@@@); the heavy books themselves (the media binaries) sit on the stacks (the S3 media bucket). Two people upload the same book and the library doesn't shelve two copies — it recognizes them as one by the book's "fingerprint" (the sha256 hash), storing just one. When you want to read, you fetch the book by its call number. Cards are light and searchable; stacks are cheap and roomy — each in its right place.

The tRPC backbone

Last lesson named the three doors; this one pushes open the "staff entrance" — tRPC — and shows its skeleton. tRPC is the type-safe contract between UI and server: types the backend defines flow end to end to the frontend caller, so a change in one place errors the whole chain instantly. Its structure is surprisingly lean: one root router aggregates ~64 feature routers, plus a handful of "procedure builders" that guard auth and RBAC uniformly via layered middleware. Grasp this middleware stack and you grasp the security and the flow of Langfuse's entire UI backend.

🛂 Analogy
Think of each tRPC request as walking a security corridor. The checkpoints along it are the middleware: first a camera logs that you came (OTel tracing), then a responder stands ready to catch exceptions (error handling), then an ID check (are you logged in), and finally the guest list (are you on the access list for this room — i.e. "are you a member of this project"). Different doors (procedures) carry different numbers of checkpoints: a public door passes only the first two, a protected door walks them all. Define a checkpoint once, each door picks what it needs — that's the essence of procedure builders.

A procedure = a stackable middleware chain

The essence of tRPC: a procedure is built by stacking middleware. Each middleware does one thing, chained in .use() order; different procedure builders stack different checkpoints. Langfuse defines a set in web/src/server/api/trpc.ts: from the bare-bones publicProcedure to the layered protectedProjectProcedure.

Layered middleware, composed on demand: publicProcedure(OTel+error) → authenticatedProcedure(+login) → protectedProjectProcedure(+project-member RBAC). Plus variants protectedOrganizationProcedure, protectedGetTraceProcedure. Source: web/src/server/api/trpc.ts:238-583.

The repository layer

The last lesson ended at "the resolver calls a @langfuse/shared repository to read". This lesson opens that repository layer door and shows how it pulls data out of ClickHouse. The core is a discipline: every analytical query funnels through one executor — queryClickhouse(). It uniformly wraps each query with OTel tracing, query tags, backoff retry, and friendly resource-error wrapping. Plus a key trick that makes cross-table JOINs cheap — the look-back window. Get this layer and you get the "master valve" for all of Langfuse's reads.

🏬 Analogy
Think of ClickHouse as a vast warehouse. If every worker (each piece of query code) barges into the warehouse to rummage on their own, nobody records who took what, and when things go wrong there's no trail. Langfuse instead opens one single dispatch window (queryClickhouse): everyone hands in their slip here. The clerk logs who's collecting (query tags), stamps a tracking ticket (OTel span), and waits and retries if a shelf is briefly busy (backoff). If your slip tries to haul off the whole warehouse (out of memory), the clerk won't let the warehouse collapse — they politely hand back "narrow it down and come again" (resource error). One window handles observability, accountability, and fault tolerance all at once.

Look-back windows: cross-table JOINs without full scans

Remember Lesson 8 — trace, observation, score are three independent wide tables, with no foreign keys, correlated by time. The problem: to stitch a trace's observations and scores together, must we full-scan an observations table of months and billions of rows? Unacceptably slow. The repository layer's answer is the look-back window: exploiting the fact that "a trace's observations almost all cluster in a short window after it happened", it searches only within a bounded time range.

Bounded look-back, skip old partitions: anchored at the trace's timestamp, observations are found within start_time ≥ anchor − 2 days, scores within 1 hour. These constants are injected into WHERE clauses, with Lesson 8's monthly partitioning letting JOINs touch only a few partitions. Source: repositories/constants.ts:4,9.

😖 no window: full-table JOIN

To find a trace's observations, search the whole observations table (months, billions of rows). Every list query doing this would soon drag the cluster down — the classic crash of using an analytical store like a transactional one.

😀 look-back window: bounded scan

Inject start_time ≥ trace.timestamp − 2 days into WHERE, and with monthly partitioning the JOIN reads only those few relevant partitions. The scan shrinks from "months" to "two days" — queries fly.

The window has a cost, of course: a rare unusually long observation (ending more than 2 days late) could fall outside the window and be missed. This is a deliberate tradeoff — trading "letting go of a tiny long tail of correctness" for "JOINs no longer scanning the whole table". Based on measurement (~96% of observations start within 2 minutes of the trace), Langfuse sets the window to 2 days, leaving a huge safety margin, so it almost never truly misses. Between correctness and cost, engineers drew a pragmatic line by the real data distribution.

Filtering & the search bar

Lesson 22's executor "sends queries out", but who builds those queries' WHERE clauses? This lesson is about filtering. The key insight: users filter in many ways — clicking the sidebar facets, typing search-bar grammar, even describing it in natural language to AI — yet they all converge into one FilterState (a set of typed conditions), which then compiles to parameterized SQL (ClickHouse or Postgres). One intermediate contract that both blocks SQL injection and makes the search bar and AI filters safe by construction.

🌐 Analogy
Think of FilterState as the UN's interlingua. Attendees speak different mother tongues — some pick from a menu (sidebar facets), some write a telegram (search-bar key:value grammar), some dictate a request (AI natural language) — but every utterance is first translated into one standard interlingua (FilterState), then translated again into what the database speaks (ClickHouse SQL or Postgres SQL). This translator has two iron rules: never splice the original words straight into the translation (pass values via placeholders, no injection); and refuse on the spot what it can't understand or express, never paraphrasing nonsense (validation + round-trip dropping hallucinations). All inputs converge to one intermediate representation, then fan out to various backends — that's the filtering system's skeleton.

😖 each input to each backend: N×M

3 inputs × 2 backends = 6 translation logics, 6 places to each guard injection. Add a new input (say AI) and you must rewrite the wiring and security checks for every backend — a combinatorial explosion, and one spot always gets missed.

😀 converge to FilterState: N+M

Each input only "translates to FilterState", each backend only "translates FilterState to SQL". Validation and parameterization happen once in the middle. A new input is just +1 translator-in, freeloading all backends and all the safety.

Lists & tables

The last two lessons covered how queries are built and run. This one looks at Langfuse's flagship list — the trace table — and how it opens instantly. Two secrets: one, a single generic function serves four "shapes" (count / rows / metrics / identifiers); two, it splits the cheap rows from the expensive metrics — the table paints rows first, while cost/latency/score metrics needing JOINs stream in later and merge client-side. Plus a conditional-FINAL query discipline that keeps the default view reading only the latest partition. Get this lesson and you've got the whole art of a big-data table being "complete yet fast".

🍽️ Analogy
Think of list loading as a restaurant serving in courses. Insisting the whole elaborate meal (rows + all metrics) arrive at once means you sit waiting a long time. Your experience is "something to eat the moment you sit down", not "staring at an empty table for twenty minutes". Langfuse's table does exactly this: give what you can instantly, fill in what you must compute as it comes — that's the "compact list model".

Sessions

Lessons 24 and 25 covered the list and detail of a single trace. But a real conversation is often multi-turn — the user asks, the AI answers, back and forth a dozen rounds, each round a trace. What threads them into one whole conversation is the session. This lesson's key insight is strikingly lean: a session's metrics and listing aren't stored separately; they're computed on the fly by grouping traces on session_id (only a tiny metadata stub is persisted, more below). Better still, the sessions table reuses Lesson 24's table machinery nearly verbatim — bluntly, it's the trace table, grouped.

💬 Analogy
Think of a session as a chat thread and a trace as a message in it. Each message is stored individually, but you read them as one whole conversation. That "thread" isn't a separate thing stored somewhere — it's just "all messages with the same thread-id, shown together, with a summary line": who took part, how many rounds, how long, total cost. You needn't build a separate table or maintain a "thread" — it grows naturally out of the messages. That's exactly where Langfuse's sessions come from.

🔴 pre-stored aggregates (not done)

Pre-compute each session's count, cost, duration into a table and every incoming trace must update its session's aggregates — an extra write path, an extra piece of state that could drift from the underlying traces. Compute rather than pre-store.

One detail to set straight: a session isn't entirely absent from storage. On ingestion, if a trace carries a session_id, the worker upserts a tiny "stub" row into Postgres's trace_sessions table (just id, project_id, environment, timestamps, with ON CONFLICT DO NOTHING — registered once per session). Its job is to "register that this session exists" and to hang some session-level metadata (like bookmarked, public). But note: the truly expensive parts — user_ids, round count, duration, total cost, the trace list — are none of them in this table; they're all computed by ClickHouse grouping traces on the fly. So the precise statement is: a lightweight session metadata entity is persisted, but all of its metrics and listing are derived. That's exactly the tradeoff — store what should be stored (identity, metadata), don't store what can be computed (aggregate metrics).

The public REST API

Lesson 21 opened the "staff entrance" for the in-house UI (tRPC). This lesson opens the third and last door — the public REST API for external SDKs. Its priorities differ sharply from tRPC's: internally seek type-safety, fast to change; externally seek contract stability, holding up at scale. Four things to see: how a route factory unifies auth/rate-limit/validation across all endpoints, how auth uses a Redis cache to withstand massive traffic, how versions are managed by folder, and how a Fern contract generates multi-language SDKs. Finish this and Lesson 20's "three doors" are all open.

🏛️ Analogy
Think of the public REST API as a government building's public service hall. Whatever business you come for (query a trace, submit a score, pull a dataset), you first go through the same security and queue process (the route factory: check ID, rate-limit, verify your form is filled right) — no one skips it. Frequent visitors get their ID info cached at the front desk, no trip to the archives each time (Redis-cached auth). The hall's procedures are printed into a manual and translated into your language (Fern generates multi-language SDKs); when procedures update, the old manual still works, the new procedure ships as "v2", existing users unaffected (versioning by folder). Unified, stable, cacheable, versioned — that's what an external API should look like.

An external API verifies a key on every request — and an SDK may send thousands per second. Hit Postgres for every key check and the database gets crushed by auth itself. So ApiAuthService wraps key verification in a Redis cache:

The SDK sends tens of thousands of requests a second, each verifying a key against Postgres — a big chunk of the DB's load is auth itself, slowing things before business even starts; attackers flooding with fake keys hit the DB directly.

😀 cache + negative cache: the DB is barely touched

Common keys hit Redis, key checks near-zero-cost; even "nonexistent keys" are stopped at the Redis layer by negative caching. Whether normal traffic or malicious flooding, Postgres is shielded behind, disturbed only on a cache miss.

The scoring model

The first four parts let you see what happened on every LLM call. But "what happened" is not the same as "how good was it". This part is about evaluation—how to put a number on a trace. And the atomic unit of all of it is the score. This lesson nails down the score's data model: what a score looks like, its three data types (numeric / categorical / boolean), what the score config ("the schema of a score") governs, and which three sources a score can come from. Once you have this model, the next five lessons (LLM-as-a-judge, code eval, human annotation, monitors) are all just "different ways of producing a score".

📋 Analogy
Think of a score as an evaluation sticker slapped on one answer. The sticker says what you're rating (name, e.g. "helpfulness"), the rating (value), and which kind of scale (dataType: a 0–1 number? a "good/ok/bad" tier? a "right/wrong" tick?). The score config is the printing standard for that kind of sticker: it mandates that "helpfulness" must be a 0–1 number, that "sentiment" must be one of {pos/neutral/neg}—so same-named stickers always use the same ruler, which is what makes them comparable and averageable. As for who applied the sticker (source): maybe you did (API), maybe an AI judge did (EVAL), maybe a human reviewer did (ANNOTATION)—but as long as the scale matches, all three live in the same comparable space.

Three scales: numeric, categorical, boolean

The soul of a score is its dataType—it decides how the value is read and validated. Langfuse pins down three types with Zod (plus a legacy TEXT):

Three scales, each with a strength: NUMERIC (a number with optional min/max, averageable/sortable), CATEGORICAL (a fixed set of label↔value, both unique, countable/groupable), BOOLEAN (a special category locked to True=1 / False=0, pass-rate computable). Source: packages/shared/src/domain/score-configs.ts:18-80.

Note one elegant move: boolean is not a separate fourth type—it is implemented as a "exactly two categories, locked to True=1/False=0" category. So the core only handles two logics, "numeric" and "categorical", and boolean reuses categorical storage and aggregation— one fewer special case, one more bit of consistency. This echoes Lesson 8's "wide event" idea: if a general mechanism can express it, don't invent a new one.

One more design that runs throughout: the value is always a number, while stringValue only holds the "label". A NUMERIC value is just that number with an empty stringValue; CATEGORICAL/BOOLEAN put the label in stringValue and the mapped number in value (e.g. "positive"→a number, "True"→1). Precisely because every score has a numeric value, averaging, trending, and pass-rate work uniformly across all types—the physical basis of "comparable".

score config: a "schema for scoring"

A dataType alone isn't enough. If the same "helpfulness" score is rated 0–1 by one person today and 1–100 by another tomorrow, the data turns to mush. score config fixes exactly this: it is the schema of a score name— declaring what dataType that name uses, and what bounds / which categories. So no matter where a score comes from, it is validated against the same ruler, which is the precondition for comparison and aggregation.

One config guards one name: a score config declares a name's dataType and constraints (numeric bounds / category set), validating scores from API / EVAL / ANNOTATION into one scores table, one ruler. Source: scores.ts:4 (ScoreSourceArray=["API","EVAL","ANNOTATION"]).

What fields a score object actually has

Unpack ScoreSchema and a score splits into three groups of fields—"what & how much", "who & with what", and "attached to what". The most worth remembering is the last: a score must attach to exactly one of trace / observation / session / dataset-run (four reference fields, exactly one non-null).

what · how much

name which dimension (e.g. "helpfulness"); value always a number; stringValue the label (categorical/boolean/text only); dataType which scale.

who · with what

source API / EVAL / ANNOTATION; authorUserId the human annotator (nullable); comment a note (nullable); configId which config validates it; queueId which annotation queue it came from.

attached to (one of)

traceId the whole trace; observationId one step within; sessionId the whole session; datasetRunId one dataset run—exactly one must be non-null.

This "one-of-four attachment point" matters: it lets a score sit on the whole trace (was this answer good overall), on one observation (was this retrieval step accurate), or on a whole session or a dataset run (the experiment scenarios of Lessons 34–35). One score model covers every evaluation granularity, from a single step to a whole batch experiment. Source: packages/shared/src/domain/scores.ts:89-133.

🎯 Design trade-off
Why bother with a dedicated score config instead of letting everyone score freely? Because "being comparable" is the entire value of evaluation. If under the name "helpfulness" team A rates 0–1, team B uses a percentage, and C uses "good/bad", then those scores are meaningless together—averaging is wrong, trending is wrong, comparing is more wrong still. A score config fixes and enforces "a given name = a given scale", so same-named scores are forever comparable, and you can confidently ask "did average helpfulness rise this week" or "which model has the higher toxicity pass-rate". This is the same engineering belief as Lesson 8's "provided vs computed" and Lesson 16's "pricing schema": model "what the data should look like" explicitly and validate up front, and every downstream analysis stands on solid ground. Unifying three sources into one scores table and one config set lets "AI-judged", "human-reviewed", and "API-reported" scores live in one space—which is the shared destination of every remaining lesson in Part 5: producing comparable scores.
🎯 Key points
  • A score is evaluation's atomic unit: one (name, value, dataType, source) attached to a trace or one observation within it (or a session / dataset run), turning "what happened" into "how well".
  • Three dataTypes: NUMERIC (a number with optional min/max), CATEGORICAL (a fixed set of label↔value, both unique), BOOLEAN (a special category locked to True=1/False=0).
  • Boolean reuses categorical: not a separate fourth type but an "exactly two" category—one fewer special case, echoing Lesson 8's "don't invent when a general mechanism works".
  • score config = a name's schema: declares dataType and constraints (bounds/categories), enforces validation, so same-named scores always share one ruler—comparable and aggregatable.
  • Three sources: API (you submit), EVAL (AI/code judge), ANNOTATION (human)—all converge, all written to Lesson 8's scores table. The next five lessons are just "producing scores in different ways".

LLM-as-a-judge

Last lesson said a score is evaluation's atomic unit, and that it has three sources. This lesson covers the most interesting source: EVAL—using one LLM to score another LLM's output, known industry-wide as LLM-as-a-judge. Human labeling is accurate but slow and costly; rule matching is fast but only checks the surface. Letting a "judge LLM" read your trace and score it against your criteria gives you automatic, scalable assessment of "was this answer actually good". This lesson takes apart Langfuse's judge pipeline: the template (the rubric), the variable mapping (which parts of the trace to hand the judge), the structured output (the judge must return a tidy score + reasoning), and most crucially—how the verdict turns back into a score.

📋 Analogy
Picture a judge at an essay contest. The scoring rubric (template prompt) says "rate 'helpfulness' 0–1 and explain why"; but the judge doesn't read the whole student file—staff hand over only the essay body and the prompt (variable mapping: fill the trace's input/output into the template's {{question}}/{{answer}}). After reading, the judge must fill in a fixed form with "score + comment" (structured output—not a prose blob no one can record). Finally that scoring sheet is filed into the gradebook (the verdict becomes a source=EVAL score, flowing back through Lesson 12's ingestion path into the scores table). The neat part: the judge is itself an LLM call, so it too is recorded as a Langfuse trace—the judge is also observed.

The judge pipeline: from one trace to one score

Langfuse splits LLM-as-a-judge into three reusable things: a template (what to rate, how, with which model), an evaluator (which traces to bind the template to, how variables are fetched, what the score is named), and repeated executions (actually fetching data, calling the judge, producing scores). ①Eval Template
A reusable "rubric": a prompt with {{variables}} (e.g. "rate helpfulness 0–1 with reasoning"), an output definition (which score it produces—NUMERIC/CATEGORICAL/BOOLEAN, what range), plus which provider/model judges. A template only describes "how to rate", binding no specific data.
②Evaluator (JobConfiguration)
"Grounds" the template: a filter picking which traces to evaluate (e.g. "only production, name=chat"), a variable mapping (which column the template's {{answer}} reads), and what the final score is named (scoreName).
③–⑥Execution (JobExecution)
Each matching trace triggers one execution: extract variables → compile prompt → call judge LLM for a structured result → normalize into a score → write back via the ingestion path. This "execution" has its own status lifecycle (detailed in Lesson 30).

Variable mapping: which part of the trace to hand the judge

The template holds placeholders like {{question}} {{answer}}—how do they become real content? Via the variable mapping: each template variable declares which langfuseObject (trace / observation / dataset_item) and which column (selectedColumnId, e.g. input/output/metadata) it comes from. At execution, extractVariablesFromTracingData pulls each value from the real trace per the mapping.

Extraction (done automatically at run)

For each variable find its mapping, fetch that column's value from the real trace/observation/dataset_item, yielding {var, value}. If missing, degrade to an empty string with a warning—never crash the whole eval.

Structured output and flow-back: how a verdict becomes a score

The judge LLM can't just return prose—that can't be recorded. Langfuse uses structured output to force tidy JSON: per the template's output definition, either {score: number, reasoning} (NUMERIC/BOOLEAN) or {matches: [...], reasoning} (CATEGORICAL). Then toNormalizedScores normalizes it into a score: value is the score, comment is the judge's reasoning.

🎯 Design trade-off
Why must the judge use "structured output" instead of free-forming? Because the product of evaluation must be recordable, aggregatable, comparable. If the judge returns prose like "I think this answer is decent but the opening is wordy…", you can neither reliably extract a score nor average across a thousand traces. Forcing a {score, reasoning} schema lets the score enter the scores table (aggregatable) while keeping "why this score" in comment (traceable)—both at once.

Deeper: why do eval results flow back through the ingestion path rather than write the DB directly? This is a recurring Langfuse belief—"one entry, many producers". Whether a score comes from API, human, or a judge LLM, it's wrapped as the same SCORE_CREATE event through Lesson 12's path, so dedup, merge, and ClickHouse persistence are written once and consistent for every source. Evaluation is therefore not "another system" but another upstream of the ingestion pipeline. And an elegant recursion: the judge's own LLM call is recorded as a Langfuse trace (environment tagged LLMJudge)—the observability platform observes even its own judge.
🎯 Key points
  • LLM-as-a-judge = use one LLM to score another's output: between human (accurate but slow) and rules (fast but shallow), it offers automatic, scalable quality assessment.
  • Three pieces: template (prompt+output def+model, "how to rate"), evaluator (target filter+variable mapping+scoreName, "whom to rate"), execution (actually runs once per trace).
  • Variable mapping: each template {{placeholder}} declares which column of trace/observation/dataset_item it comes from; extractVariablesFromTracingData fetches real values at run, and compileEvalPrompt flattens to a string only at the "feed-to-LLM" boundary.
  • Structured output: the judge is forced to return {score, reasoning} (or {matches, reasoning} for categorical); toNormalizedScores normalizes to value=score, comment=reasoning—aggregatable and traceable.
  • Flow back (source=EVAL): the result is wrapped as the same SCORE_CREATE event through Lesson 12's path. Evaluation is another producer for the ingestion pipeline, not a separate system; the judge's own call is also recorded as a trace.
